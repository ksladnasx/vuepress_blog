# 项目逻辑

## 人脸识别接口中加密算法

注：

- SHA1 / SHA256 ： 都是哈希算法，通过把任意长度的数据转换成一个固定长度的字符串（摘要），转换后不可逆且结果完全由输入决定，结果固定长度（SHA1 输出 160 位，SHA256 输出 256 位）。SHA1已经是老算法，存在安全性不足的问题，现在已经不推荐了。
- HMAC ：即Hash-based Message Authentication Code，为基于哈希的消息认证码。他本质上是 哈希算法 + 密钥(key)。他加入了secret key，使得只有知道密钥的人才能通过哈希算法算出结果，而不是像简单的哈希算法那样只能验证内容的完整性。HMAC_SHA256(data, secret)



嗯，这个加密对接的逻辑是这样的。接口要求我们每次请求的时候都要带一个签名，签名的作用是验证请求的合法性和完整性，防止被篡改或者重放攻击。

签名的生成规则其实不复杂。公司那边给了 accessKeyId 和 accessKeySecret 两个凭证，我这边需要先把四个东西拼接起来：accessKeyId、accessKeySecret、当前的时间戳 timestamp、还有请求参数的前 512 个字符。拼接成一个字符串作为待签名的原始数据。

然后根据文档要求的签名类型，比如 SHA256 或者 SHA1，用 HMAC 算法对这个拼接好的字符串进行签名。这里注意签名用的密钥是 accessKeySecret，不是凭证明文。签名出来是一个二进制数据，再转成 Base64 字符串，这就是最终的 sign 值。

我代码里用的是浏览器的 Web Crypto API 来做 HMAC 签名，因为原生支持 SHA 系列算法，性能也还可以。但 MD5 不行，Web Crypto API 不支持，如果要 MD5 就得换 crypto-js 这种库。

拿到 sign 之后，每次调用人脸识别的接口，我都会把 accessKeyId、timestamp、sign 这三个东西带上，后端收到后会拿同样的逻辑重新算一遍签名，比对是否一致。只要签名对得上，请求才会被认为是合法的。

然后插件这边我用的是浏览器打包的方式，把写好的代码打成插件文件，放到 126 浏览器的扩展程序里加载测试。整个流程跑通之后，活体识别接口就能正常调用了。



## 农担封装了什么组件

根据你给的截图，我梳理一下，在这个 AI 对话相关的模块里，我其实封装了蛮多可复用的通用组件。这个模块的目录结构已经能看出一些拆分的思路了。

首先最核心的是 **ChatInput.tsx**，这个是输入框组件。它不光是个输入框，还处理了多行文本的自动撑高、Enter 发送和 Shift+Enter 换行的区分，还有上传图片或者文件的触发入口。它和 **FileBs.tsx** 配合，文件那块逻辑单独抽出来，不至于让输入框组件太臃肿。

然后是消息展示这一块，我没有只写一个大组件把所有消息类型都塞进去，而是拆成了 **MessageUser.tsx** 和 **MessageSystem.tsx**，分别渲染用户发的消息和系统返回的消息。它们两个都依赖底层的 **MessageBs.tsx** 来承载公共样式或者气泡结构。另外还单独抽了 **MessageButtons.tsx**，用来渲染消息下面的快捷回复按钮，这个按钮组在不同场景会复用。还有一个 **MessagePanne.tsx**，是用做消息面板的容器，管理滚动到底部、加载历史消息这些逻辑。

我还封装了 **Separator.tsx** 配合 **SeparatorFactory.tsx** 来做分隔线。比如显示“昨天”、“今天”这种时间分段，或者用来插一些提示文案。用工厂模式是为了根据不同的分隔类型动态生成不同的样式和行为，方便扩展。

**RunLog.tsx** 是用于展示 AI 执行步骤或者调用链信息的，比如“思考中”、“正在检索”这种过程性的日志，方便用户理解 AI 的进度。**GuideQuestions.tsx** 是进入对话前的引导问题列表，用户点一下就能直接发送，减少输入成本。

再说另外两个通用组件。**多选下拉栏组件**，我封装的时候支持了搜索过滤、批量删除、全选、最大选择数限制这些功能。对外暴露的接口比较简洁，传 options 数组和一个 v-model 绑定的选中值就行。**主页的卡片组件**，我做了比较通用的设计，支持传入 ICON 的组件或者图片，卡片标题、描述内容、底部的操作按钮都可以通过插槽或者 props 来定制。这样在不同的列表页或者仪表盘都能复用，不需要重复写卡片的布局和样式。

总的来说，我的习惯是尽量让组件保持单一职责，组合起来用。像 AI 对话这种复杂模块，拆得细一点，后期维护和替换具体实现都会轻松很多。
