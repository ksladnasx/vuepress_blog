## 1. SSE + BroadcastChannel（非常容易被问）

**面试官：** 你简历里提到用 BroadcastChannel 解决 SSE 连接数限制，详细说一下。

**面试者：**
当时我们的数据上云平台用了 SSE 做实时消息推送。用户如果打开多个标签页连接会迅速增加，而在http1.1的情况下，浏览器只支持最大六个sse长连接，如果多开页面，会导致新的浏览器页面echart的数据展示阻塞，从而影响用户体验。

由于从业务逻辑上讲，用户客观上存在打开多个页面监控多个不同服务器数据的需求，因此不能简单从逻辑上限制用户多开页面，并且升级http协议或者sharedworkker的方式都会导致代码大幅度的修改，浪费大量时间且会影响后续的代码维护，因此我在搜寻多方解法后选择了BroadcastChannel 的方式来解决这个问题。这个解决方式的核心就是，通过只允许一个主标签页建立 SSE，其他标签页通过 BroadcastChannel 与主页面通信来获取各自的数据更新消息，在主标签页被关闭后重新选举来实现多页面只占一个sse连接从而大幅减少连接资源消耗，提高页面稳定性并优化用户体验。并且只需修改一个sse的总的配置文件，而不需要对所有建立sse连接的地方进行代码修改。

## 2. React Error Boundary

**面试官：** Error Boundary 你是怎么封装的？

**面试者：**
因为智能体平台模块很多，有构建，模型，知识库，系统等模块页面，其中部分模块渲染异常后会直接导致白屏。

我在未封装全局报错边界的之前，用户有反馈点击某些页面会白屏系统崩溃的情况，但是在有些pc上却无法复现，我在排查了之后发现是用户的谷歌浏览器自带的翻译插件存在修改源代码的dom的情况导致dom报错

因此我封装了全局 Error Boundary，核心目的是给用户更完备的系统体验，给用户重新加载或者返回上一次运行正常模块的页面，不会导致用户认为整个系统的崩溃而手足无措。

实现方式是：

```jsx
class ErrorBoundary extends React.Component {
  state = { hasError: false }

  static getDerivedStateFromError() {
    return { hasError: true }
  }

  componentDidCatch(error, info) {
    console.log(error, info)
  }

  render() {
    if (this.state.hasError) {
      return <Fallback />
    }

    return this.props.children
  }
}
```

然后在路由文件的关键业务模块外层包裹。这样即使某个节点异常：

- 页面不会整体崩溃
- 用户还能继续操作其他功能

## 3. 动态渲染工作流

**面试官：** 你说“配置驱动动态渲染”，什么意思？

**面试者：**

因为智能体开发平台中，用户创建的工作流会包含很多节点，比如LLM节点，知识库节点等，每次用户编辑工作流的时候，会有很多节点要进行渲染和关系对接，如果纯用后端返回用户所配置节点的html，会增加后端的代码量和服务器负担，因此这里的逻辑是由后端只返回相关节点的配置json文件，然后前端根据配置，在每个节点模板库中根据配置动态对节点进行渲染，每次用户修改节点的并保持更新后，后端只需修改配置文件的相关配置，至于节点怎么去渲染只由前端决定，大大降低了后端的压力。

## 4. React Hooks 为什么比 class 更好？

**面试者：**

class组件存在this指向辅助，生命周期逻辑分散且逻辑复用困难的问题，而Hooks的这种函数式开发的方式更加简洁且逻辑的复用更方便，更适合TS语言以及项目的维护。因此我个人封装组件的时候都优先用函数式组件的方式。

------

## 5. useEffect 为什么会死循环？

**面试者：**
因为 useEffect 依赖变化会重新执行。

如果 effect 内部又修改了依赖就会无限更新。本质是上是依赖更新导致effect执行又导致重新 更新 而又触发 effect执行而无限循环。

```js
useEffect(() => {
  setCount(count + 1)
}, [count])
```

------

## 6. React 为什么需要 key？

**面试者：**
React diff 时需要判断节点是否复用。key 本质是节点唯一标识。如果没有 key：React 可能错误复用 DOM。

导致：输入框错乱，状态异常以及不必要重渲染
