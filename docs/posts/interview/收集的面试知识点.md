# 收集的面试知识点

## JavaScript

### ES6

ES6 的全称是 ECMAScript 2015，是 JavaScript 语言的一个大的版本标准。它的核心概念是**让 JavaScript 更适合写大型应用**——比如引入了块级作用域（let/const）来替代 var 的函数作用域，用类和模块来解决之前原型链和立即执行函数模拟模块的混乱。还有箭头函数在概念上是更轻量的函数，同时固定 this 指向外层。Promise 则是把异步流程从回调嵌套变成链式调用。所以 ES6 不只是加一堆 API，而是**改变了 JavaScript 的编程模型**：从基于函数和原型，走向更接近传统面向对象和函数式混合的现代语言。

### HTML5和CSS3的新特性

嗯，HTML5 和 CSS3 算是现代前端的基础了，我简单说一下我用过觉得比较实用的那些。

HTML5 的话，最直观的是语义化标签，比如 `<header>`、`<footer>`、`<nav>`、`<article>` 这些，以前全用 div 堆，现在结构更清晰，对 SEO 和可访问性也好。再就是表单增强，像 `type="email"`、`type="number"` 还有 `placeholder`、`required` 这些属性，做简单校验不用自己写 JS 了。

媒体这块，`<video>` 和 `<audio>` 标签直接支持音视频播放，不用再靠 Flash。还有 canvas 和 SVG，canvas 适合做动态图形、图表、游戏，SVG 适合矢量图标和缩放不模糊。存储方面多了 localStorage 和 sessionStorage，比 cookie 好用太多，容量也大。还有 Web Workers，可以用多线程跑一些耗时计算，不阻塞页面。另外就是地理定位、拖拽 API、history.pushState 做单页路由，这些都是 HTML5 带来的。

CSS3 的话，最常用的就是圆角 `border-radius`、阴影 `box-shadow`、线性渐变 `linear-gradient`，还有 transition 和 transform，配合起来可以做简单的动画，不用写 JS。复杂一点的用 `@keyframes` 加 animation。布局上 Flex 和 Grid 绝对是革命性的，以前 float+clearfix 那种写法基本淘汰了。响应式靠 `@media` 查询，可以根据屏幕宽度调整样式。还有自定义属性，也就是 CSS 变量，配合主题切换很好用。新加的选择器也很多，比如 `:nth-child`、`:not`、`::before` 这些伪类伪元素，以前很多需要 JS 改 UI 的地方现在纯 CSS 就能搞定。

总之 HTML5 让浏览器更像应用平台，CSS3 让界面更灵活好看。当然有些新特性兼容性要查一下，但现在大部分现代浏览器都支持得不错。

### 纯函数及副作用

在 React 高阶组件（HOC）的语境中，“副作用”指的是 **对函数外部环境产生可观察变化** 的操作。具体来说，一个纯函数（没有副作用的函数）应满足：

1. **相同的输入总是产生相同的输出**：给定相同的组件参数，HOC 总是返回相同的新组件。
2. **不修改输入参数**：不改变传入的原始组件（例如不向它的原型添加方法、不修改它的静态属性、不直接改写它的 props）。
3. **不依赖或改变外部状态**：不读写函数外部的变量（全局、闭包中的可变变量、localStorage、DOM 等），不执行网络请求、定时器、日志输出等 I/O 操作。

### 原型与原型链

JS中构造函数内部有 prototype 属性，其属性值是对象，这个对象包含了可以由该构造函数的所有实例共享的属性和方法。

使用构造函数新建的对象内部将包含一个指针，指向构造函数的 prototype 属性对应的值，在 ES5 中这个指针被称为对象的原型。

浏览器中实现了 利用对象的**proto** 属性来访问它构造函数的 prototype 属性对应的值。ES5 中新增了一个 Object.getPrototypeOf() 方法，可以通过这个方法来获取对象的原型。

所以说原型指的是某对象的构造函数的所有实例共享的属性和方法的指针，当访问一个对象的属性时，如果这个对象内部不存在这个属性，那么它就会去它的原型对象里找这个属性，这个原型对象又会有自己的原型，于是就这样一直找下去，也就是原型链的概念。

先记住**JS 原型核心 3 条铁律**（所有原型问题都靠它）：

1. 实例的 `__proto__` **严格等于** 其构造函数的 `prototype`
2. 函数的 `prototype` 对象，默认自带一个 `constructor` 属性，**指回函数本身**
3. 所有对象的原型链终点，都是 `Object.prototype.proto__` 是 `null`
4. 所有的`构造函数.prototype` 本身是一个**对象字面量 / 普通对象**，所有**普通对象**的 `__proto__` 都指向 `Object.prototype`

```js
function Person(name) {
  this.name = name;
}
// 修改原型，即对p的构造函数的prototype 指针新加一个getName属性指向
Person.prototype.getName = function () {};
var p = new Person("hello");
p.constructor = Person;
console.log(p.__proto__ === Person.prototype); // true
console.log(p.__proto__.constructor.prototype.__proto__=== Object.prototype); // true

```



### 为什么循环中不用数组索引做key

因为key要保证元素的唯一性和稳定性，如果用索引做key，数组中做了删除等操作，后面的key都会变，会导致页面性能受到很大影响

页面节点顺序变化的话性能消耗最小，节点更新其次，节点添加和删除最大。

同时用index做节点可能会影响子节点的状态，被删后不能保持用户输入等。

### this的指向问题

普通函数的 this 由调用方式决定，谁调用就指向谁；

对象方法调用时 this 指向该对象；而箭头函数没有自己的 this，它会捕获定义时外层作用域的 this，因此不会随着调用对象变化而改变。

#### 1. 普通函数调用：谁调用，`this` 就指向谁

普通函数直接调用时，`this` 指向调用它的对象。谁调用这个函数，函数内部的 `this` 就指向谁。

```js
function sayName() {
  console.log(this.name);
}

const person = {
  name: "Tom",
  sayName
};

person.sayName(); // Tom这里 `sayName` 是通过 `person.sayName()` 调用的，所以 `this` 指向 `person`。

// 但是如果直接调用,此时没有明确的调用对象，
const fn = person.sayName;
//在浏览器非严格模式下，`this` 指向 `window`；在严格模式下，`this` 为 `undefined`。反正输出都是undefined
fn(); 
```

------

#### 2. 函数作为对象方法调用：`this` 指向这个对象

当一个函数被挂载到对象上，并通过对象调用时：

```js
const user = {
  name: "张三",
  sayHello() {
    console.log(this.name);
  }
};

user.sayHello(); //张三
//同理如果
const fn = user.sayHello;
//虽然函数原本属于 `user`，但是调用方式变了，已经不是对象调用，因此 `this` 不再指向 `user`。
fn();
```



------

#### 3. 箭头函数：没有自己的 `this`，继承外层作用域的 `this`

箭头函数不会创建自己的 `this`，它会沿用定义位置外层作用域的 `this`。

```js
const obj = {
  name: "Tom",
  sayHello: () => {
    console.log(this.name);
  }
};

obj.sayHello();
```

但是实际上箭头函数没有自己的 `this`，它会向外层寻找。这里外层是全局作用域，所以为浏览器的window.name，因此输出通常为空。

------

#### 4. 箭头函数在类中：继承类实例的 `this`

```js
class Person {
  constructor(name) {
    this.name = name;
  }

  sayHello = () => {
    console.log(this.name);
  }
}

const p = new Person("Tom");

p.sayHello(); //输出Tom
```

箭头函数定义在类实例环境中，它继承的是实例的 `this`。相当于：

```js
this.sayHello = () => {
    console.log(this.name);
}
```

这里的 `this` 指向当前实例对象。

------

#### 总结

| 调用方式                | this 指向                                   |
| ----------------------- | ------------------------------------------- |
| 普通函数调用 `fn()`     | window（非严格模式）/ undefined（严格模式） |
| 对象方法调用 `obj.fn()` | obj                                         |
| 构造函数调用 `new Fn()` | 新创建的实例                                |
| call/apply/bind 调用    | 手动指定对象                                |
| 箭头函数                | 继承定义位置外层的 this                     |



### 异步与事件循环

在前端开发中，**异步（Asynchronous）** 是指程序的某些操作不会立即完成，而是会在未来某个时间点完成，同时不会阻塞后续代码的执行。与之相对的是**同步（Synchronous）**：任务按顺序依次执行，前一个任务没完成，后面的代码就无法运行。

**为什么前端需要异步？**

- **JavaScript 是单线程的**（浏览器中每个渲染进程会有一个主线程用于执行 JS 代码、处理 UI 事件、布局等）。如果所有操作都同步执行，遇到耗时任务（如网络请求、定时器、文件读取），整个页面就会“卡死”，无法响应用户点击、滚动等交互。
- **异步** 允许这些耗时任务在后台进行，等任务完成后通过回调、Promise 等方式通知主线程，期间主线程可以继续处理其他事情（比如保持界面流畅）。当然要调用栈为空，即主线程空闲后才会后由事件循环取出执行

**前端常见的异步场景**

| 场景             | 示例代码                                      | 说明                     |
| :--------------- | :-------------------------------------------- | :----------------------- |
| 定时器           | `setTimeout(() => console.log('done'), 1000)` | 1秒后才执行回调          |
| 网络请求         | `fetch('/api').then(res => res.json())`       | 等待服务器响应           |
| 事件监听         | `button.addEventListener('click', handler)`   | 用户点击时才执行 handler |
| 文件读取（Node） | `fs.readFile('./a.txt', callback)`            | 读磁盘不阻塞主线程       |
| 动画帧           | `requestAnimationFrame(update)`               | 在下一帧绘制前执行       |

**异步的核心机制：事件循环（Event Loop）**

同步代码会立即执行，异步任务则会被放入“任务队列”，等待主线程空闲后由事件循环取出执行。简单的理解：

1. 执行同步代码（主调用栈）。
2. 遇到异步 API（如 `setTimeout`、`fetch`），交给浏览器或 Node 的底层模块处理，完成后将回调放入任务队列。
3. 主栈清空后，事件循环从队列中取出回调执行。

```javascript
console.log('1'); // 同步

setTimeout(() => {
  console.log('2'); // 异步，1秒后放到队列
}, 1000);

console.log('3'); // 同步，同步任务执行完后，执行队列中的console.log('2');

// 输出顺序：1 → 3 → 2
```

**web  worker**

在 HTML 页面中，如果在执行脚本时，页面的状态是不可相应的，直到脚本执行完成后，页面才变成可相应。web worker 是运行在后台的 js，独立于其他脚本，不会影响页面的性能。 并且通过 postMessage 将结果回传到主线程。这样在进行复杂操作的时候，就不会阻塞主线程了。

### 异步编程实现方式以及优缺点

异步编程这块，JavaScript 里其实有好几种实现方式，各有各的适用场景和毛病。

最早也是最原始的就是**回调函数**，比如 `setTimeout`、`fs.readFile` 这种。它的优点就是简单直接，没有任何语法开销。但缺点很明显，一旦异步任务多了就陷入“回调地狱”，一层套一层，代码横着长，很难维护，而且错误处理很麻烦，每个回调里都要自己判断错误。

后来就有了 **Promise**，它把回调变成了链式调用，比如 `fetch().then().catch()`，这样代码就平整多了，不用再嵌套那么深。Promise 的好处是可以统一用 `.catch` 处理错误，还支持同时等待多个异步任务比如 `Promise.all`。缺点呢，就是语法上还是有一堆 `.then` 和函数，尤其是稍微复杂一点的流程，代码读起来还是不够直观，而且 Promise 本身一旦创建就会立即执行，不能中途取消，这些都是痛点。

再后来 **Generator** 函数也被拿来处理异步，比如配合 `co` 库就能实现类似同步的写法。但是 Generator 本质上不是为异步设计的，需要手动调用 `next()` 或者依赖其他库去驱动，用起来比较别扭，所以现在基本不太直接用它来做异步了。

现在最主流的就是 **async/await**，它其实就是 Promise 的语法糖。优点非常明显，就是用同步的写法写异步代码，不用再写一堆 `.then`，`try...catch` 也能直接用，代码读起来特别顺。缺点倒是很少，主要就是要注意 `await` 只能在 async 函数里用，而且如果不小心把 `await` 忘了，得到的就是一个 Promise 对象而不是实际的值，容易出 bug。另外多个互不依赖的异步任务如果用 `await` 串行写，效率会低，需要配合 `Promise.all` 来并行。

除了这些，还有**事件监听**和**发布订阅**，比如 `EventEmitter` 或者 `addEventListener`，它们适合那种需要多次触发、或者解耦的场景，但用来做一次性异步返回就不太方便，因为需要自己去管理回调的注册和触发，比较松散。

总结一下，我现在写异步基本只用 `async/await`，遇到并行就用 `Promise.all`，实在需要加超时或者取消才会再往下封装一层。Promise 也会写，但主要是配合 `async/await` 用。回调只用在极少数特定 API 里，比如 `addEventListener` 这种本来就依赖回调的。



### 异步编程中的事件循环

嗯，事件循环可以说是 JavaScript 异步编程的底层机制。因为 JS 是单线程的，一次只能做一件事，那像定时器、网络请求这些不能干等着吧？所以需要一个调度系统，事件循环就是干这个的。

我理解它大概是这样运作的：JS 执行时会把同步任务立即执行，遇到异步任务就交给浏览器或其他环境去处理，等异步任务有了结果，就把它的回调函数扔进一个“任务队列”里。主线程执行完所有同步代码后，再去任务队列里拿回调来执行。这个过程不断重复，就像个循环，所以叫事件循环。

这里关键点是任务有优先级，分成了宏任务和微任务。宏任务包括整体 script 代码、setTimeout、setInterval、I/O 操作这些；微任务主要是 Promise.then、MutationObserver、queueMicrotask 这些。每次事件循环会先执行一个宏任务，**然后把这个宏任务执行期间产生的所有微任务都清空**，然后再取下一个宏任务。

这就解释了为什么 Promise.then 会比 setTimeout 先执行——因为微任务的执行时机是在当前宏任务的末尾，而下一个宏任务得排在后面。同样，有时候用 setTimeout 0 只是把任务扔到下一个宏任务，不一定是“立即”，因为得等当前微任务都清完。

我平时写代码会利用这个机制来做一些“延迟”或者“优先级控制”。比如想等当前同步代码和所有微任务执行完再干点啥，可以扔到 setTimeout 或者 MessageChannel 里。另外像 Vue 的 nextTick 其实也是利用了微任务或宏任务来实现的。

总之，事件循环就是单线程下的异步调度器，宏任务和微任务的区分决定了回调的执行顺序。理解了这个，很多异步顺序的坑就能避免了。

### 数据类型判断

JavaScript 中判断数据类型有很多种方式，不同场景会选择不同的方法。

最基础的是 `typeof`，它适合判断基本数据类型，比如 `string`、`number`、`boolean`、`undefined`、`symbol`、`bigint` 和 `function`。不过它对引用类型判断能力有限，像数组、对象、日期都会返回 `object`，而且 `null` 由于历史遗留问题也会返回 `object`。

如果需要精确判断引用类型，我一般会使用 `Object.prototype.toString.call()`，它能够返回类似 `[object Array]`、`[object Date]`、`[object RegExp]` 等结果，因此判断最准确，也是很多框架底层常用的方法。

对于数组，也可以直接使用 `Array.isArray()`，这是 ES5 提供的官方方法，简单且可靠。

另外，判断对象是否属于某个构造函数还可以使用 `instanceof`，例如 `arr instanceof Array`。不过它是根据原型链判断的，跨 iframe 等场景可能会失效，因此更适合判断对象实例，而不是做通用类型检测。

至于**伪数组（Array-like）**，它本质上不是数组，而是具有类似数组结构的对象，一般具有两个特点：一是拥有按数字递增的索引属性，二是具有 `length` 属性，例如 `arguments`、DOM 的 `NodeList`、`HTMLCollection` 等。

判断一个对象是不是伪数组，通常会检查它是否是对象、是否具有合法的 `length` 属性、是否不是数组。例如：

```js
obj != null &&
typeof obj === 'object' &&
typeof obj.length === 'number' &&
!Array.isArray(obj)
```

如果需要把伪数组转换成真正的数组，我一般会使用 `Array.from()` 或者展开运算符 `[...]`，这样就可以使用数组的各种方法了。



### 内存泄漏与发生原因

内存泄漏简单说就是：**程序用完了某块内存，但没有还给操作系统，导致这块内存永远被占着，用着用着可用内存越来越少，页面就越来越卡，最后崩掉。**

JavaScript 有垃圾回收机制，按理说不应该泄漏，但问题在于**有些内存虽然已经没用了，但垃圾回收器认为它还有用，就不会收**。

我碰到过几次，大概这几种情况最容易出问题：

**一个是全局变量**。不小心挂在 window 上，或者忘了加 `let`/`const`，相当于声明了一个全局变量。这个变量除非页面关闭，永远不会被回收。

**再就是定时器或者事件监听没清**。比如用 `setInterval` 搞了一个轮询，但页面组件销毁了没有 `clearInterval`，这个定时器还在跑，它引用了组件里的一些数据或者 DOM，那这些数据就永远释放不掉。事件监听也是一样，绑了 `window.addEventListener`，组件卸载时没移除，回调函数一直存在，引用的东西也一起留着。

**还有闭包**，虽然闭包很好用，但用不好也容易出事。比如一个函数内部返回另一个函数，返回的函数引用了外部函数的某个大数组。只要你拿着返回的函数不释放，那个大数组就一直存在。有时候你只是忘了把执行完的变量置 `null`，它就一直挂在那。

**DOM 引用也是个坑**。比如用变量存了一个 DOM 节点，后来页面把这个节点删了，但你的变量还留着引用，那这个 DOM 节点占的内存也不会被回收。还有一种情况是给 DOM 附加了属性，存了个很大的对象，节点删了但属性还在。

**在 Vue 和 React 里**，如果不注意 `beforeDestroy` 或者 `useEffect` 的清理函数，上面说的定时器、事件监听、订阅这些都很容易泄漏。跨组件通信的事件总线比如 EventBus，如果你监听了但组件销毁时没取消，回调函数会一直挂在总线上，整个组件都被带着收不掉。

所以我现在写代码的习惯是：**只要开了定时器、绑了全局事件、订阅了外部数据源，就一定在组件销毁时清理**。还有避免随手往 `window` 上挂东西，尽量用 `const`/`let` 控制作用域。怀疑内存泄漏时就用 Chrome DevTools 的 Memory 面板拍个快照，对比一下就能定位到问题所在。

### 闭包是什么，讲讲闭包的垃圾处理机制

#### 原理

闭包简单说就是**：一个函数能够记住并访问它定义时的作用域，即使这个函数在外面执行**。比如**函数里面 return 一个函数**，内部函数用了外部函数的变量，这就产生闭包。核心特性:

1. **函数嵌套**：外部函数包裹内部函数。
2. **引用外部变量**：内部函数使用了外部函数的局部变量。
3. **外部返回内部函数**（或其他方式使其逃逸），从而形成闭包。

```js
function outer() {
  let message = "Hello, 闭包！";  // 外部变量
  function inner() {
    console.log(message);        // 内部函数引用外部变量
  }
  return inner;
}
//计数器闭包，会自己回收
setTimeout(function() {
    console.log(i);   // 输出 3,3,3
  }, 100);

let myClosure = outer();       // myClosure 就是 inner 函数 + message 变量
myClosure();                     // 输出: Hello, 闭包！
myClosure = null                 //垃圾回收
```

#### 闭包作用

- 函数工厂，外部函数return一个内部函数，内部函数实际执行依赖外部函数收到的参数，因此每次调用外部函数时，传的参数不同，所得到的函数的作用也不同。
- 数据封装：将外部函数内定义的变量单独封装入内部函数，使其不受外部的影响。

#### 垃圾回收机制

垃圾回收机制呢，正常情况下，函数执行完它的局部变量就会被标记可回收。但闭包不一样——**因为内部函数还引用着外部函数的变量，所以这些变量不能被释放，会一直留在内存里，直到那个内部函数自己也被回收了。**

这既是优势（可以私有化变量、做缓存），也是隐患：如果闭包一直没被销毁，比如挂到了 window 上或者一直没解除引用，就会造成内存泄露。所以注意在组件卸载时要把引用置 null，或者事件监听要解绑。 

#### **循环中的经典闭包问题及其解决**

```js
// 问题：var 导致共享同一个 i
for (var i = 0; i < 3; i++) {
  setTimeout(function() {
    console.log(i);   // 输出 3,3,3
  }, 100);
}

// 解决：用闭包捕获每次的 i
for (var i = 0; i < 3; i++) {
  (function(j) {
    setTimeout(function() {
      console.log(j);   // 输出 0,1,2
    }, 100);
  })(i);
}

// 更简单的解决：用 let
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);  // 0,1,2
}
```



用到场景：就是在按钮点击回调中保存一些状态，又不想把变量放在全局，例如外部函数中保存一个count，然后内部函数作为事件处理器，这样count不会泄露到全局，又能正确记住状态

#### **闭包对于写 Hook 的帮助？**

Hooks 本质上就是依赖闭包来捕获每一次渲染的状态。每次组件渲染都会生成一个独立的闭包，里面的变量和事件处理函数都“冻结”在那一刻。这就解释了为什么 `useEffect` 的依赖项里缺了某个变量，就会拿到过期的值——闭包捕获的是旧的那个值。

闭包也让 Hooks 的写法更自然。比如自定义 Hook 内部可以封装私有的状态和副作用，对外只暴露接口，实现逻辑复用。但同时也要注意闭包陷阱，比如在 `useEffect` 里用到了外部变量但没加到依赖数组，就会拿到旧值，造成 bug。所以 `eslint-plugin-react-hooks` 会要求依赖项写完整，就是为了避免闭包捕获旧值的问题。

总结一下：useState 的“异步”是批处理优化，闭包是 Hooks 的运行基础，理解这两点就能避免很多关于“拿不到最新值”和“useEffect 不更新”的困惑。



### window.resize 与 ResizeObserver 

这两个都是监听尺寸变化的手段，但最大的区别在于**监听对象和作用域**完全不同。

**`window.resize`** 只能监听浏览器窗口大小变化，而且它有一个很烦人的毛病：触发非常频繁，只要拖动边框就会狂轰滥炸，所以实际用的时候必须配合防抖或节流，不然性能直接炸。另外它只能拿到窗口的宽高，如果想知道某个具体元素尺寸变了，还得在里面手动去查 `getBoundingClientRect`，这会强行触发回流，带来额外的性能开销。

**`ResizeObserver`** 就灵活多了，它可以监听任意 DOM 元素的尺寸变化，不只是窗口。而且它专门为响应式设计，回调里会返回 `contentRect` 和精确到小数点的盒模型尺寸（比如 `borderBoxSize`），最实用的地方在于它不会像 `window.resize` 那样在滚动或动画时疯狂触发，而是在渲染的 layout 阶段之后统一派发，性能开销小很多，还能避免你手动去读尺寸导致强制回流。

**但说到真实落地，这里有几个坑一定要处理**

- 首先是得防止内存泄漏。用了他们之后记得remove掉，防止观察者盯着已经被移除的 DOM，既浪费内存又会报错。
- **回调里别直接 setState 改数据**，回调中直接修改响应式数据会导致组件重绘，而重绘又改变尺寸，很容易死循环。我一般的做法是在回调里先用 `requestAnimationFrame` 去读值，如果还需要后续操作，再用防抖包一层，避免高频更新。
- **要处理观察的元素被删除或隐藏的情况。** 如果元素被 `display: none`，`ResizeObserver` 还能观测到尺寸变为 0，但如果没有触发 disconnect，它依然占用资源。我习惯在 `MutationObserver` 或者组件生命周期里判断元素是否存在，如果不存在就主动 disconnect。

## Vue

### Vue2和Vue3的区别

Vue2 和 Vue3 最大的区别在于**底层响应式实现、开发方式和性能**。

首先，**响应式原理不同**。Vue2 基于 `Object.defineProperty` 对对象属性进行劫持，它只能监听已经存在的属性，因此新增、删除属性以及数组部分操作需要使用 `Vue.set`、`Vue.delete` 等 API。Vue3 则基于 `Proxy` 实现响应式，可以直接监听整个对象，对属性的新增、删除以及数组、Map、Set 等数据结构都能更好地支持，功能更完善。

其次，**开发方式不同**。Vue2 主要使用 **Options API**，把 `data`、`methods`、`computed`、`watch` 等逻辑分散在不同选项中；Vue3 新增了 **Composition API**，通过 `setup`、`ref`、`reactive`、`computed` 等组合式 API，把同一业务逻辑聚合在一起，更利于代码复用和大型项目维护。

第三，**性能更好**。Vue3 重写了虚拟 DOM 和 Diff 算法，引入了静态提升、Patch Flag、Tree Shaking 等优化，减少了运行时的计算和打包体积，因此渲染性能和首屏加载速度都优于 Vue2。

第四，**TypeScript 支持更完善**。Vue3 底层使用 TypeScript 重构，类型推导更加友好，而 Vue2 对 TypeScript 的支持相对较弱。

另外，Vue3 还新增了很多实用能力，比如 **Fragment** 支持组件多个根节点、**Teleport** 可以将组件渲染到指定 DOM 节点、**Suspense** 支持异步组件加载，这些都是 Vue2 不具备的特性。

总的来说，**Vue2 更适合维护老项目，而 Vue3 在性能、开发体验、TypeScript 支持以及可维护性方面都有明显提升，因此目前新项目基本都会优先选择 Vue3。**



### Vuex的使用以及与Pinia的区别

Vuex 和 Pinia 都是 Vue 的状态管理库，用来管理多个组件之间共享的数据，比如用户信息、权限、主题配置、菜单状态等，避免通过 Props 或事件层层传递数据。

以 Vuex 为例，它采用集中式状态管理，核心包括 **State、Getter、Mutation、Action 和 Module**。其中 **State** 用来存储全局状态，**Getter** 用来派生数据，**Mutation** 是唯一修改 State 的方式，必须是同步操作；**Action** 用来处理异步逻辑，再通过提交 Mutation 修改状态；当项目较大时，还可以使用 **Module** 对状态进行拆分管理。

我在项目中一般会将用户信息、登录状态、权限菜单等放到 Vuex 中，通过组件调用 `dispatch` 执行异步操作，再 `commit` 修改状态，实现整个应用的数据共享。

不过现在 Vue3 项目更多会使用 **Pinia**，因为它是 Vue 官方推荐的新一代状态管理方案，相比 Vuex 更轻量、更符合 Composition API 的开发方式。

两者主要有几个区别：

第一，**API 更简洁**。Vuex 需要区分 State、Getter、Mutation、Action，而 Pinia 去掉了 Mutation，可以直接在 Action 中修改状态，代码更加直观。

第二，**TypeScript 支持更好**。Pinia 是用 TypeScript 编写的，类型推导几乎开箱即用，而 Vuex 使用 TS 时需要编写较多的类型声明。

第三，**模块化方式不同**。Vuex 使用 Module 管理多个模块，而 Pinia 每个 Store 都是独立模块，不需要再嵌套 Module，维护起来更简单。

第四，**Composition API 支持更好**。Pinia 可以直接在 `setup` 中使用，与 Vue3 的开发模式更加契合。

因此，如果是 **Vue2 项目**，我一般会使用 Vuex；如果是 **Vue3 新项目**，我会优先选择 Pinia，因为它代码更简洁、类型支持更完善，开发体验也更好。

## React

### React基础

React并不是将click事件绑定到了div的真实DOM上，而是在document处监听了所有的事件，当事件发生并且冒泡到document处的时候，React将事件内容封装并交由真正的处理函数运行。

`this.props`是组件之间沟通的一个接口，原则上来讲，它只能从父组件流向子组件。React具有浓重的函数式编程的思想。[React中的props为什么是只读的？](https://www.xhblog.cc.cd/posts/interview/React框架面试题.html#_10-react中的props为什么是只读的)



### React函数式组件用的比类组件多

现在肯定函数组件用得多了，新项目基本全是函数组件。区别的话，核心就三点：

1. **写法**：类组件要 `this`、要 `bind`、要写 `render`，很啰嗦；函数组件就是一个函数，直接 `return JSX`。
2. **状态**：类组件用 `this.state` 和 `this.setState`；函数组件用 `useState`。
3. **生命周期**：类组件拆成 `componentDidMount`、`componentDidUpdate` 等；函数组件就用一个 `useEffect` 搞定，更聚合。

为啥函数组件多？因为 Hooks 出来后，代码复用（自定义Hook）特别方便，而且没有 `this` 的坑。现在 React 官方文档也都是函数组件的例子，类组件基本只出现在老项目里了。



### 回答一下 useState 和 useEffect 的使用

`useState` 就是用来声明状态变量的。比如 `const [count, setCount] = useState(0)`，`count` 是当前值，`setCount` 是更新它的函数。**注意状态改变会触发组件重新渲染**。

`useEffect` 用来处理副作用，比如发请求、操作 DOM、订阅。它接受两个参数：一个是回调函数，一个是依赖数组。依赖数组控制什么时候重新执行。

- 依赖数组空 `[]`：只在组件挂载时执行一次，相当于 `componentDidMount`。
- 依赖数组不传：每次渲染都执行（**一般不用，性能差**）。
- 依赖数组有变量 `[count]`：`count` 变化时执行。

另外 `useEffect` 的回调可以返回一个清理函数，在组件卸载或下次执行前调用，用来清除定时器、取消订阅。

我平时用的话：发请求用空依赖，监听某个 prop 变化就把那个变量放依赖数组里。

### **useState 为什么是异步更新？**

准确来说，`setState` 不是真正的异步，而是**批处理**。React 为了性能，会把多个 state 更新合并成一次渲染，而不是每次 setState 都立刻重新渲染。在 React 合成事件和生命周期钩子里，它会开启批量更新模式，等所有同步代码执行完再统一更新。但在 `setTimeout`、原生事件或者 Promise 里，批处理会失效，会同步更新——所以它更像是“延迟更新”而不是“异步”。

### **为什么调用 setState 后不能立刻拿到最新值？**

因为新 state 要等到下次渲染才会生效。`setState` 的本质是触发一次重新渲染，而不是直接修改当前闭包里的 `state` 变量。所以你在当前函数作用域里拿到的 `state` 还是旧的快照值。如果想要在更新后基于新值做操作，可以用 `useEffect` 监听这个 state，或者用函数式更新 `setCount(prev => prev + 1)` 来保证基于最新值计算。

## 框架原理

### 不同框架的响应式

如果让我讲**原生 HTML/JS、Vue、React 是如何实现响应式的**，我会这样回答：

响应式其实就是一个目标：**当数据发生变化的时候，页面能够自动跟着更新，而不是开发者自己去操作 DOM。** 这也是现代前端框架最核心的能力之一，只不过不同技术实现这个目标的方式不一样。

最开始使用原生 HTML 和 JavaScript 开发的时候，其实是没有响应式这个概念的。我们通常会维护一份数据，然后写一个 `render` 函数负责把数据渲染到页面上。每次数据发生变化，比如 `count++`，还需要开发者自己去调用一次 `render()`，由 `render()` 再去找到对应的 DOM 节点，通过 `innerHTML`、`textContent` 等 API 把页面更新掉。如果开发者忘记调用 `render()`，那么数据虽然已经变了，但是页面完全不会更新。所以原生 JavaScript 属于一种命令式开发，开发者需要明确告诉浏览器什么时候更新、更新哪个 DOM。

后来大家就开始思考，既然每次修改数据都要手动调用 `render`，那为什么不能让数据自己知道什么时候发生了变化呢？这也是 Vue 响应式思想的来源。Vue2 使用的是 `Object.defineProperty` 对对象的每一个属性进行劫持，为每个属性定义 `getter` 和 `setter`。当组件渲染时，模板中访问了哪些数据，就会触发对应属性的 `getter`，Vue 会在这个阶段完成依赖收集，把当前负责渲染的 Watcher 与这些数据建立关联。当数据被修改时，就会进入对应属性的 `setter`，`setter` 会通知所有依赖这个数据的 Watcher 重新执行渲染函数，生成新的虚拟 DOM，再通过 Diff 和 Patch 的过程把变化同步到真实 DOM 上。整个过程中，开发者只需要修改数据，而不需要关心 DOM 的更新细节。

不过 `Object.defineProperty` 本身也存在一些局限，比如它只能监听已经存在的属性，不能监听对象新增或删除的属性，因此 Vue2 才会提供 `Vue.set` 这样的 API 来解决这个问题。另外，由于它需要递归地给对象的每一个属性都添加 `getter` 和 `setter`，初始化成本也比较高。

到了 Vue3，响应式底层换成了 ES6 的 `Proxy`。相比 `Object.defineProperty`，`Proxy` 是直接代理整个对象，而不是一个个属性，所以新增、删除属性以及数组索引变化都能够天然监听。同时 Vue3 将原来的 `Dep` 和 `Watcher` 机制进一步抽象成了 `track` 和 `trigger`。当响应式数据被读取时，`track` 会收集当前副作用函数；当数据发生修改时，`trigger` 会找到依赖这个数据的副作用函数重新执行，从而完成页面更新。由于 `Proxy` 不需要在初始化阶段遍历所有属性，而且功能更完整，所以 Vue3 的响应式能力和性能都比 Vue2 更优秀。

React 的思路则完全不同。很多人第一次学习 React 会觉得它是不是也用了 `Proxy` 或者数据劫持，但实际上并没有。React 根本不会监听普通变量，例如你定义一个 `let count = 0`，即使执行 `count++`，React 也完全不知道数据发生了变化，因为这个变量只是普通 JavaScript 变量，并不属于 React 管理。React 真正管理的是组件的 State，所以只有调用 `setState` 或者 Hooks 中的 `setCount`，React 才会认为状态发生了变化。此时 React 会把这次更新封装成一个 Update，交给调度器 Scheduler，根据优先级安排更新时机，然后重新执行整个组件函数，得到一棵新的虚拟 DOM 树，再与旧的虚拟 DOM 进行 Diff，只把真正发生变化的部分更新到真实 DOM 中。

所以从本质上来说，Vue 和 React 都是数据驱动视图，只是驱动方式不同。Vue 更强调**数据本身具有响应能力**，数据一旦发生变化，就能够自动通知依赖它的组件更新，因此开发者只需要修改数据即可。而 React 并不会去监听数据，而是把组件看成一个状态到 UI 的映射关系，也就是经常说的 **UI = f(State)**。每当状态发生变化时，它就重新执行组件函数，根据新的状态重新计算出新的 UI，再通过 Diff 算法完成最小化更新。因此 Vue 更偏向于“响应式数据驱动”，React 更偏向于“状态驱动渲染”，虽然实现机制完全不同，但最终目的都是在开发者尽量少操作 DOM 的前提下，高效地完成页面更新。

### 生命周期

**React** 通常将组件生命周期分为三个阶段：

- 装载阶段（Mount），组件第一次在DOM树中被渲染的过程；
- 更新过程（Update），组件状态发生变化，重新更新渲染的过程；
- 卸载过程（Unmount），组件从DOM树中被移除的过程；

**Vue** 的生命周期，简单说就是一个组件从创建、挂载、更新到销毁的过程，Vue 在里面放了 8 个钩子。我最常用的是 `created` 和 `mounted`。
`created` 的时候数据已经响应式了，但还没 DOM，所以我一般在这里发请求、初始化一些数据。
`mounted` 才拿到真实 DOM，这时候适合去操作第三方插件，比如 echarts 初始化。
然后 `beforeDestroy` 很重要，我经常在这里清定时器、解绑全局事件，不然 SPA 里切来切去会内存泄露。
还有一个容易被忽略的点：父子组件嵌套时的顺序。父组件先走完 `beforeCreate`、`created`、`beforeMount`，然后才去递归子组件，等所有子组件 `mounted` 之后，父组件才会 `mounted`。
如果用了 `keep-alive`，就会多出 `activated` 和 `deactivated`，原来的 `beforeDestroy` 就不会触发了。
基本上，我开发中主要记住：发请求在 `created`，操作 DOM 在 `mounted`，清副作用在 `beforeDestroy`，就够了。

### 进入页面发起请求的位置差异

React 里进入页面发请求，核心就一句话**：在组件挂载完成后发**。

类组件用 `componentDidMount`：

```js
componentDidMount() {
  fetch('/api/list').then(res => this.setState({ data: res }))
}
```

函数组件（主流）用 `useEffect`，依赖项传空数组：

```js
useEffect(() => {
  fetch('/api/list').then(res => setData(res))
}, [])  // 空数组 = 只在挂载时执行一次
```





### 组件间通讯方式与跨组件传值呢

组件间通信这个我分两类说吧**：常见通用方式**和**跨组件场景**。

**通用方式（适用于任意关系）**：

- **父子**：父传子用 `props`，子传父用 `$emit`（Vue）或回调函数（React）。
- **爷孙/深层嵌套**：Vue 用 `provide/inject`，React 用 `Context`。
- **兄弟**：一般通过共同父组件中转，或者事件总线（Vue 有 `$bus`，React 自己实现事件库）。
- **全局状态**：Vuex/Pinia（Vue）、Redux/Zustand（React），适合复杂大项目。

**跨组件传值的具体实现（不依赖状态管理库）**：

- **Vue**：`provide/inject` 最直接，祖先提供，后代注入，响应式需要提供 ref 或 reactive。
- **React**：`Context` + `useContext`，Provider 包裹子孙。
- **事件总线**：Vue 可以挂一个全局 `mitt` 或 `emitter`；React 可以自己用 `useRef` 封装 `EventEmitter`，但不常见。

我自己的选型习惯：简单父子用 props/emit，相隔一两层用 provide/inject 或 Context，再复杂的集中用 Pinia/Zustand。事件总线容易造成混乱，现在基本不主动用了。

#### 示例

特别是单页面应用，数据和修改数据的方法最好都放在父组件，由父组件提供方法和数据，子组件只需要进行消费就好，否则数据容易混乱。

以下是 **Vue 3** 和 **React** 中跨组件传递数据的标准示例，均包含响应式处理。

---

**Vue 3 示例**（props/`$emit` + 事件监听）

```vue
<!-- 父组件 -->
//这里定义了所监听的事件名称为update，监听到这个事件后执行handleUpdate函数
<ChildComponent :message="parentMsg" @update="handleUpdate" /> 

<!-- 子组件 -->
<script setup>
const props = defineProps(['message'])
//这里子组件提交这个update事件，会触发父组件的监听从而调用父组件的函数
const emit = defineEmits(['update','事件2',...])  //这里是声明子组件会向外触发哪些事件
emit('update', 'newValue')  //这里是实际触发事件并传递数据，emit(事件名称,数据)，
</script>
```

**React 示例**（props/传递回调函数）

```jsx
// 父组件
const Parent = () => {
  const [msg, setMsg] = useState('hello')
  // 父组件将自己的状态修改函数作为回调传给子组件
  return <Child message={msg} onUpdate={setMsg} />
}

// 子组件
const Child = ({ message, onUpdate }) => {
  // 子组件通过调用父组件传入的回调函数，将新数据传回父组件
  return <button onClick={() => onUpdate('newValue')}>
    {message}
  </button>
}
```

**核心对照**

| 方向     | Vue                | React                        |
| :------- | :----------------- | :--------------------------- |
| 父→子    | `props`            | `props`（单向数据流）        |
| 子→父    | `$emit` + 事件监听 | 传递回调函数（通常setState） |
| 跨层级   | `provide/inject`   | `Context`                    |
| 全局状态 | Vuex / Pinia       | Redux / Zustand              |

这两种模式都体现了**单向数据流**的理念：父组件通过props向下传递数据，子组件通过事件或回调向上通知变化，避免数据随意修改导致的混乱。

##### Vue 3：`provide` / `inject`（响应式）

```vue
<!-- 祖先组件：App.vue -->
<template>
  <div>
    <p>祖先组件中的数据：{{ user.name }} ({{ user.age }}岁)</p>
    <button @click="updateName">改名</button>
    <ParentComponent />
  </div>
</template>

<script setup>
import { provide, reactive, ref } from 'vue'
import ParentComponent from './ParentComponent.vue'

// 响应式数据（使用 reactive 或 ref）
const user = reactive({ name: '张三', age: 25 })

// 也可以提供方法
const updateName = () => {
  user.name = '李四'
}

// 向所有后代提供数据
provide('user', user)
provide('updateName', updateName)
</script>
```

```vue
<!-- 后代组件：DeepChild.vue -->
<template>
  <div style="border:1px solid #ccc; margin:10px; padding:10px;">
    <h3>后代组件</h3>
    <p>接收到的名字：{{ user.name }}</p>
    <p>年龄：{{ user.age }}</p>
    <button @click="updateName">调用祖先方法修改名字</button>
  </div>
</template>

<script setup>
import { inject } from 'vue'

// 注入祖先提供的数据（可以设置默认值）
const user = inject('user')
const updateName = inject('updateName')

// 也可以直接修改 user.name（因为是 reactive，会同步到祖先）
// 但推荐通过提供的方法来修改，保持数据流清晰
</script>
```

##### React：Context + useContext（响应式）

```jsx
// 1. 创建 Context
import { createContext, useContext, useState } from 'react'

const UserContext = createContext(null)

// 2. 祖先组件（Provider）
function App() {
  const [user, setUser] = useState({ name: '张三', age: 25 })

  const updateName = () => {
    setUser(prev => ({ ...prev, name: '李四' }))
  }

  return (
    <UserContext.Provider value={{ user, updateName }}>  //此处提供数据
      <div>
        <p>祖先中的数据：{user.name} ({user.age}岁)</p>
        <button onClick={updateName}>改名</button>
        <ParentComponent />
      </div>
    </UserContext.Provider>
  )
}

// 3. 后代组件（消费）
function DeepChild() {
  const { user, updateName } = useContext(UserContext)

  return (
    <div style={{ border: '1px solid #ccc', margin: 10, padding: 10 }}>
      <h3>后代组件</h3>
      <p>接收到的名字：{user.name}</p>
      <p>年龄：{user.age}</p>
      <button onClick={updateName}>调用祖先方法修改名字</button>
    </div>
  )
}
```

#### 关键点对比

| 特性              | Vue 3                        | React                                  |
| ----------------- | ---------------------------- | -------------------------------------- |
| 提供数据          | `provide(key, value)`        | `<Context.Provider value={}>`          |
| 消费数据          | `inject(key)`                | `useContext(Context)`                  |
| 响应式原理        | 使用 `ref` / `reactive` 包裹 | 使用 `useState` 返回的 state 和 setter |
| 默认值 / 后备内容 | `inject(key, defaultValue)`  | `createContext(defaultValue)`          |

两种方式都能实现任意层级的组件通信，且能保持数据响应式/状态驱动。 



### 受控和非受控表单

受控和非受控，其实是 React 里的概念，但放到原生 HTML 里也好理解，主要区别就是**表单的值由谁管**。

**受控表单**：值交给 JavaScript 控制。比如 input 的 `value` 绑定一个变量，每次输入触发（input.addEventListener('input',func）)更新变量。这样变量和输入框始终同步，方便做实时校验、条件禁用、动态改变值等。缺点是要写事件处理。

**非受控表单**：值交给 DOM 自己管。你不需要手动设置 `value`，只需要给个 `ref`，需要的时候再取出 DOM 里的值。比如 `document.getElementById` 或 React 的 `useRef`。优点是代码简单，适合简单表单、文	件上传（file input 必须是非受控）。

原生 HTML 里其实大部分是非受控的，但你可以用 JS 去控制它的值，那就变成受控了。React 推荐用受控，因为状态更可控。实际开发中，复杂表单用受控，简单场景或者文件上传用非受控。

## 浏览器与网络

### 登录鉴权

1. 用户输入账号密码，后端返回public key 前端接收后利用其对输入进行加密发送，后端校验成功后返回一个 token。
2. 前端把 token 存到 localStorage 或者 httpOnly cookie 里，后续每个请求都带着 token（一般放请求头 Authorization）。
3. 登录成功后接着请求后端拿当前用户的权限菜单列表，然后动态生成路由和侧边栏菜单。

至于菜单持久化，主要是解决**刷新页面后路由和菜单丢失**的问题。我们项目的做法：

- 登录成功后，把后端返回的菜单数据（原始树结构）和用户权限码同时存一份到 `sessionStorage`（不是 localStorage，因为不想多标签页共享，除非业务需要）。
- 在路由守卫（`beforeEach`）里优先判断 pinia/vuex 里有没有菜单数据，没有就去 `sessionStorage` 里取，有的话直接恢复路由（调用 `addRoute` 批量添加）。
- 如果 `sessionStorage` 里也没有，才重新请求后端接口。
- 同时 pinia 里会保存一份，配合 `pinia-plugin-persist` 自动持久化到 localStorage，刷新后直接还原，这样就不用在守卫里反复读 sessionStorage。

另外退出登录时要同时清掉 `sessionStorage` 和 store 里的菜单数据，以及 token，再跳回登录页。

主要记住：菜单等非敏感数据做持久化，避免刷新白屏；token 也要持久化，但退出时一定要清理干净。

### axios取消请求

能取消，用new AbortController() 一个控制器对象，然后调用对象的abort方法就能取消请求。

取消请求常用于tab栏切换，用户切换tab时会触发多个请求，但是请求响应的顺序不同，容易导致页面错乱。

所以说如果用户切换到a又快速切换到b这个tab，那就得将对于a的内容的请求给取消掉。 



### Cookie与双token

目前主流的登录认证方案通常采用 **Access Token + Refresh Token** 的双 Token 模式。

服务器登录成功后会将 **Refresh Token** 通过 `HttpOnly Cookie`（后端调用`res.cookie()`返回cookie）返回给浏览器，浏览器会自动保存，并且在向 Cookie 所属域名发起请求时自动携带。

由于 `HttpOnly` 属性限制了 JavaScript 对 Cookie 的读写，因此即使发生 XSS 攻击，恶意脚本也无法窃取 Refresh Token，从而提升了安全性。不过，仅使用 Cookie 容易受到 CSRF（跨站请求伪造）攻击，因此通常会配合 `SameSite`（以及必要时的 `Secure`）等属性限制跨站请求携带 Cookie，降低 CSRF 风险。

另一方面，现在常见的做法是将**Access Token** 存储在 `localStorage`（或内存）中，用于接口鉴权，并设置较短的有效期；当 Access Token 过期时（后端会返回 **401 Unauthorized**），前端通过自动携带的 `HttpOnly Refresh Token` 向服务器申请新的 Access Token，实现无感刷新，在兼顾用户体验的同时提高整体认证体系的安全性。

一般来说：

| Token             | 常见有效期                    | 作用                                      |
| ----------------- | ----------------------------- | ----------------------------------------- |
| **Access Token**  | 5～30 分钟（常见 15 分钟）    | 用于接口鉴权，过期后失效                  |
| **Refresh Token** | 7～30 天（常见 7 天或 30 天） | 用于刷新 Access Token，过期后需要重新登录 |

现在很多系统不会让 Refresh Token 一直使用到过期，而是在**每次刷新 Access Token 时，同时签发一个新的 Refresh Token，并立即使旧的 Refresh Token 失效**。

### 登录状态保持

对于登录的token来保持登陆状态的，普通项目存 `localStorage`，配合请求拦截器自动塞进 `Authorization` 头，最简单，刷新页面不丢失，多标签页共享。适合大部分后台管理系统，而**更严格场景（如金融、网银）**：用 `httpOnly cookie` + `Secure` + `SameSite=Strict`，ji即短生命周期Access Token（可放localStorage）+ 刷新Token（放HttpOnly Cookie）的模式，后端设置，前端拿不到，防 XSS 偷 token。缺点是受 cookie 大小限制且需要后端配合。

短生命周期access_token（可放localStorage）+ refresh_token（放HttpOnly Cookie）的模式工作流程：登录成功后，后端同时返回两个 token。前端把 access_token存 localStorage，refresh 由后端自动写到 cookie。每次请求带着 access。access 过期后，前端调用 /refresh 接口，浏览器自动带上 cookie 里的 refresh_token，后端校验后返回新 access。前端存好新 access，继续请求。如果 refresh 也过期或失效，就跳回登录。

这个流程中的 `refresh_token`，依然保存在浏览器本地的 Cookie 存储中（属于该域名）。用 F12 开发者工具的 \**Application → Cookies\** 面板，**是看得到的**（包括名称、值、过期时间、`HttpOnly` 标记等）。只是 `HttpOnly` 标记让前端 JavaScript 无法通过 `document.cookie` 读取它。这个场景下后续每次请求这个cookie对应的path(`/refresh` 接口)，浏览器自动带上 这个cookie（即带上refresh_token用于获取新的access_token） 。



### 前端四类本地存储的区别和作用

- **Cookie**：容量最小（4KB），会随每次 HTTP 请求自动带过去，所以适合存 session ID 这类需要传给后端的数据。缺点是有安全风险（XSS/CSRF），而且同步读写、操作 API 也比较老。生命周期可以设置过期时间。Cookie 相比localStorage，随每个HTTP请求**自动发送**，影响性能。现在除了严格场景（如金融、网银）用 `httpOnly cookie`其他用的很少了。

- **localStorage**：容量大（5-10MB），**永久**存储（除非手动清除），不会自动发给后端。适合存不敏感的长久数据，比如用户偏好、主题。同步 API，简单用 `setItem/getItem`。

- **sessionStorage**：和 localStorage 基本一样，区别是**只在当前标签页/会话窗口有效**，关闭标签页就没了。适合存临时性的表单草稿、页面内的中间状态。

- **IndexedDB**：容量极大（几百MB甚至更多），支持索引、事务、异步 API，适合存大量结构化数据，比如离线缓存、复杂数据（像用户聊天记录、文件切片）。但 API 较复杂，通常需要封装或用 `idb` 库。

总结一下：需要传给后端用localStorage +请求头或者 Cookie（很少了）；简单本地存储用 localStorage 或 sessionStorage；需要存储大量数据或复杂查询时用 IndexedDB。日常开发中，localStorage 用得最多。

## CSS

### Flex 和 Grid布局

Flex 和 Grid 都是 CSS 布局方案。Flex 是一维的，管一行或者一列；Grid 是二维的，同时管行和列。

**Flex**：主轴和交叉轴。常用属性有 `justify-content`（主轴对齐）、`align-items`（交叉轴对齐）、`flex-wrap` 是否换行。适合做导航栏、列表项、水平垂直居中（最简单就是 `display: flex; align-items: center; justify-content: center`）。

**Grid**：先定义行列，比如 `grid-template-columns: 1fr 1fr 1fr` 分三列。再放子元素进去，自动或手动定位。适合做整体页面布局、复杂网格、卡片墙。

平时我大多数场景用 Flex，因为简单够用；遇到需要行和列同时对齐（比如日历、图片墙）就用 Grid。两者也能配合使用，不冲突。

- **Flex**：导航栏、按钮组、列表项、水平或垂直的简单排列。  
- **Grid**：整体页面布局、相册墙、仪表盘、需要行列对齐的复杂结构。

我们可以用一个生活化的类比来理解 **一维** 和 **二维** 的区别：

- **Flex（一维）**：就像 **一条手链上的珠子**。  
  所有的珠子只能沿着这根线（主轴）排队——要么横着排，要么竖着排。  
  你可以决定珠子之间的间隔、对齐方式（靠左、居中、靠右），但你不能同时规定“第几行第几列”。  
  如果珠子太多，可以换行（`flex-wrap: wrap`），但换行后的“行”只是自动折行，不是真正的二维网格控制。

- **Grid（二维）**：就像 **棋盘或电影院座位表**。  
  你事先画好了行和列的网格（例如 3 行 × 4 列），每个棋子或观众都知道自己具体在第几行、第几列。  
  你可以同时控制行和列的尺寸、对齐，甚至让某个元素横跨多行多列。

---

通过代码对比

**Flex（一维）**：只管理单方向排列  

```css
.container {
  display: flex;
  flex-direction: row;     /* 水平排列，只控制这一行 */
  justify-content: space-between; /* 水平方向对齐 */
  align-items: center;          /* 垂直方向（副轴）统一拉伸或居中，但不能独立控制每行位置 */
}
```

**Grid（二维）**：同时管理行和列  ，`fr` 让列宽按份数瓜分**剩余空间**，实现弹性比例布局

```css
.container {
  display: grid;
  grid-template-rows: 100px 200px;   /* 定义第1行高100px，第2行高200px */
  grid-template-columns: 1fr 2fr;    /* 定义总共将列分为三份，第1列占1份，第2列占2份 */
  gap: 10px;
}
.item {
  grid-row: 2 / 3;      /* 放在第2行 */
  grid-column: 1 / 2;   /* 放在第1列 */
}
```

Grid 容器在定义行和列后，会自动生成**网格线**，从 `1` 开始编号。

- **行网格线**：水平方向的线，从上往下编号 1, 2, 3, ...
- **列网格线**：垂直方向的线，从左往右编号 1, 2, 3, ...

每个单元格由**两条相邻的行线**和**两条相邻的列线**围成。

假如有一个 `2行 × 2列` 的网格：

```text
列线：1     2     3
     |     |     |
行1  |  A  |  B  |
线2  +-----+-----+
行2  |  C  |  D  |
线3  +-----+-----+
```

- 单元格 **A** 位于：行线 1 → 2，列线 1 → 2
- 单元格 **B** 位于：行线 1 → 2，列线 2 → 3
- 单元格 **C** 位于：行线 2 → 3，列线 1 → 2
- 单元格 **D** 位于：行线 2 → 3，列线 2 → 3

------

#### 你的代码含义

```css
.item { 
  grid-row: 2 / 3;    /* 起始行线 = 2，结束行线 = 3 → 占据第2行 */
  grid-column: 1 / 2; /* 起始列线 = 1，结束列线 = 2 → 占据第1列 */
}
```

所以这个 `.item` 被放置到 **第2行、第1列** 的单元格（上图中的 **C**）。



### 垂直居中

1. **Flex**：父容器 `display: flex; align-items: center;` 一行搞定，最推荐。
2. **绝对定位 + transform**：元素 `position: absolute; top: 50%; transform: translateY(-50%);` 适合脱离文档流或弹窗场景。
3. **Grid**：父容器 `display: grid; align-items: center;` 跟 Flex 类似，但整体布局用 Grid 时顺手用。

另外还有老办法：`line-height` 等于高度（仅限单行文本），`display: table-cell` + `vertical-align: middle`（兼容旧浏览器）。现在基本不再手写了，Flex 足够覆盖大部分需求。



### 前端怎么做移动端适配的？

我们项目做移动端适配，主要用的是 vw 方案。就是通过 postcss 插件，用npm安装完再在项目根目录的 `postcss.config.js` 里配置一下，构建的时候就会自动生效。自动把设计稿上的 px 转成 vw，比如 375 宽的设计稿，一个 16px 的按钮转出来大概 4.27vw 这样。然后页面里设置一下 viewport 就 ok 了。当然对个别需要精确缩放的元素，用 flex 或百分比补一下。之前也用过 rem 配合 flexible.js，但感觉还是 vw 更省事，不用 js 动态算根字体，兼容性也够。另外字体和间距基本都靠插件自动转，个别需要微调的地方再用 flex 或者百分比补一下。整体来说开发时直接写 px，打包完自动适配，挺方便的。

页面 `meta` 设置 `viewport`，宽度等于设备宽度，初始缩放 1。字体用 `vw` 或 `rem` 都行，我们统一走插件转换。

## 性能优化

### 前端页面性能指标

前端性能指标主要可以分为**加载性能、交互性能和稳定性**三大类，其中最重要的是 Google 提出的 **Core Web Vitals（核心网页指标）**。

首先是 **LCP（Largest Contentful Paint）**，表示最大内容元素渲染完成的时间，比如页面最大的图片或标题什么时候显示出来，它反映的是页面的加载速度，一般要求 **2.5 秒以内**。

其次是 **INP（Interaction to Next Paint）**，这是目前替代 FID 的指标，用来衡量用户点击、输入等操作后，页面多久能够完成响应并更新界面，它反映的是页面的交互流畅度，通常希望控制在 **200 毫秒以内**。

第三个是 **CLS（Cumulative Layout Shift）**，表示页面布局偏移，比如图片加载后把按钮顶下去、文字突然移动等，它衡量的是页面的视觉稳定性，一般要求 **小于 0.1**。

除此之外，还有一些常见的性能指标：

- **FCP（First Contentful Paint）**：首次内容绘制，表示页面第一次渲染出文本或图片的时间。
- **TTFB（Time To First Byte）**：首字节时间，反映服务器响应速度和网络延迟。
- **TTI（Time To Interactive）**：页面达到可交互状态的时间，不过现在使用频率已经没有以前高了。

实际开发中，我通常会结合 **Chrome DevTools 的 Performance 面板**、**Lighthouse**，以及线上监控平台（如 Sentry、阿里 ARMS 等）来采集这些指标，并针对性地进行优化，例如资源懒加载、图片压缩、代码分割、减少主线程阻塞、开启缓存等，从而提升页面的加载速度和用户体验。

### 首屏加载性能优化

- 文件优化，没变化的代码尽量统一到一个文件，例如第三方库的引入文件，这样的在文件缓存的时候可极大减少下载时间
- 下载优化，服务器下发文件的时候可以通过gzip等方式压缩文件大小。同时协议版本最好http2，下载性能更好且请求头体积更小
- 缓存策略，服务器进行精准恰当的缓存策略，分强缓存（浏览器直接取缓存文件）和协商缓存（请求协商后无修改就直接用本地缓存）
- 动画，用骨架屏或者适当的loading动画，优化用户体验

### 前端埋点

前端埋点就是**在客户端采集用户行为或应用运行数据，并上报到服务端的过程**。它是做产品数据分析、异常监控、用户画像和体验优化的基础。

**埋点的类型**，我一般分成三类：

- **代码埋点**：在业务代码里手动嵌入上报逻辑，比如按钮点击、页面浏览时调用 `track('event_name', { extra: xxx })`。优点是精准可控，缺点是侵入性强，每次加埋点都要改代码发版本。
- **可视化埋点**：通过可视化工具圈选页面元素，自动生成埋点配置，不需要开发介入。适合运营或产品同学自己配置，但灵活性有限，复杂场景覆盖不了。
- **无痕埋点**：也叫全埋点，自动采集所有交互事件和页面变化，不用手动写代码。优点是覆盖全，缺点是数据量太大，而且无法区分哪些是关键行为，容易浪费存储和带宽。

**请求方式**也是三类：

- 图片上报，利用图片的url上加上埋点信息来get请求发送埋点，优点是无跨域问题，简单直接，缺点是无法压缩或加密，上报长度有 限制
- 正常请求上报，能加密能压缩，但是有跨域问题，且**页面关闭后无法发送埋点信息**。
- **sendBeacon**：利用这个浏览器专门上报埋点的api进行上报，不能加密不能压缩，不能扩展。但是由于进程是依赖浏览器的，所以说页面关闭后不关浏览器也可以上报。

**上报时机**，主流的有三种：

- **即时上报**：事件发生时立刻发送请求。适合关键事件（比如支付成功），但并发高的时候对服务器压力大。
- **批量上报**：把多条日志存在本地队列，积攒到一定数量或间隔时间再一次性发送。可以减少请求次数，提升性能，是现在的主流方式。
- **离线上报**：网络不可达时先存 localStorage 或 IndexedDB，等网络恢复后再重新发送，保证数据不丢。

实际项目中，通常组合使用：关键路径用即时上报，普通行为用批量上报，配合 `navigator.sendBeacon` 在页面关闭时发送最后一批数据，避免请求被浏览器取消。埋点 SDK 还要考虑采样率、去重、隐私脱敏以及尽量减少对主线程的性能影响。

### 大量数据一次性渲染不卡顿

这个问题我们之前遇到过类似的场景，十万条数据一次性渲染，直接塞 DOM 里肯定会卡死。核心思路就是 **不要一次性全渲染**，用分片或者虚拟滚动来分批处理。

**方案一：虚拟滚动**

这是最推荐的方式，就是只渲染可视区域内的数据，监听滚动动态加载和卸载列表项。外部用一个容器撑开滚动条，内部通过计算滚动位置决定显示哪些数据，复用固定数量的 DOM 节点。比如用 `react-window` 或者 `vue-virtual-scroller` 这些库，十万条数据也能丝滑滚动，因为 DOM 节点始终只有几十个，性能是最好的。

**方案二：分片渲染**

如果不能用虚拟滚动，或者列表结构比较复杂，就用` requestIdleCallback`分批渲染。

`requestIdleCallback` 的思路是：**利用浏览器空闲时间来执行低优先级的任务**。它会在每一帧渲染完成后，如果还有剩余时间，就去执行你传入的回调函数。浏览器会通过参数告诉你当前还有多少空闲时间 `deadline.timeRemaining()`，你可以自己决定用多少时间来分批渲染。这个方式特别适合数据渲染的场景，因为它不会影响动画、滚动、输入这些高优先级操作。

不过它也有局限性。一是兼容性，部分浏览器和 Safari 不支持，需要 polyfill。二是它不保证什么时候执行，如果浏览器一直忙（比如高频率动画或者大量计算），任务可能被无限推迟。

**方案三：document.createDocumentFragment**

如果数据不太多但 DOM 操作本身是瓶颈，可以用 `DocumentFragment` 先把所有节点在内存里组装好，然后一次性插入 DOM，减少回流次数。但十万条数据即使这样操作，构建节点本身的时间也够长的，所以只能作为辅助手段。

**方案四：服务端分页**

如果数据量真的太大，客户端渲染不是最优解，后端接口本身就应该支持分页或者滚动加载，让前端按需请求，而不是一次性返回十万条。这是最根本的解决方案，但受限于业务需求，有时候后端没法改。

**其他优化手段：**
- 对数据进行裁剪，比如只渲染必要的字段，不要挂一堆冗余数据到 DOM 上
- 用 `will-change: transform` 或 `content-visibility: auto` 提示浏览器优化渲染
- 数据不变的情况下可以用 `Object.freeze` 冻结数据，防止 Vue 或 React 做响应式代理造成额外开销

总结一下：**虚拟滚动是处理大量数据渲染的首选**，配合后端分页和前端裁剪，十万条数据其实不算大问题。如果业务场景必须一次性展示全部数据，那就用时间分片给用户一个加载中的体验，至少不会白屏。



##  工程化

### 首选vite进行项目打包，相比webpack有什么优势？

嗯，这个问题其实挺实际的，因为现在新项目确实越来越多选 Vite 而不是 Webpack。我先说结论吧：开发阶段我会首选 Vite，因为它的冷启动和热更新体验比 Webpack 好太多了。

Webpack 最大的特点是功能非常全，生态强大，什么 loader、plugin 都有，几乎所有前端资源都能处理。而且它经过这么多年沉淀，坑基本都被踩平了，大型项目的配置虽然复杂但非常灵活。缺点也明显，就是开发时启动慢，尤其在项目比较大的时候，要整个应用打包完才能看到页面，改一行代码也要重新编译相关模块，随着项目膨胀热更新越来越慢。

Vite 的思路不一样。它利用浏览器原生 ES Module 的能力，开发时不需要打包，直接启动一个开发服务器，按需编译你访问的那个文件。所以冷启动几乎秒开，不管项目有多大。热更新也是，因为只更新被改动的模块，速度非常快，基本是立即生效。再加上配置文件简单很多，开箱即用、对 Vue、React 支持也很好，新手很容易上手。

缺点的话，Vite 的生产环境打包其实底层还是用的 Rollup，生态和插件虽然也在快速发展，但跟 Webpack 那么多年的积累比，有些小众的场景可能找不到现成的插件。另外在老旧的浏览器上可能需要额外做降级处理，虽然可以通过插件解决，但 Webpack 那边更成熟一点。

所以我现在的习惯是：新项目、中小型项目、或者对开发体验要求比较高的，我首选 Vite。如果是特别复杂的、需要很多定制化打包逻辑的老项目，或者团队已经深度依赖 Webpack 生态的，那还是继续用 Webpack 更稳妥。不过趋势上看，Vite 越来越主流，我个人的话能选 Vite 就选 Vite。



### Vite 和 webpack 的热更新有什么区别

热更新（HMR）本质上是修改文件后只更新修改的部分不刷新整个页面

Webpack : 找依赖 --> 重新打包 --> 替换bundle 

Webpack 和 Vite 都支持 HMR，但实现机制不同。Webpack 在开发模式下仍然需要维护整个项目的依赖图，修改文件后会重新编译变更模块，更新对应的 Chunk，并生成 `hot-update.js` 文件，再通过 **WebSocket** 通知浏览器完成模块替换。虽然不需要重新打包整个项目，但随着项目规模增大，依赖分析和模块重建的成本也会增加，因此热更新时间会变长。

Vite 则基于浏览器原生 ES Module，在开发环境下不进行整体打包，而是按需提供模块。当某个文件发生变化时，Vite 只重新编译这个模块（通常使用速度很快的 esbuild 完成转换），通过 **WebSocket** 通知浏览器重新请求该模块，并利用 ESM 完成模块替换，**不需要重新生成 Bundle 或 Chunk**。因此，Vite 的热更新粒度更细、响应速度更快，尤其在大型项目中优势更加明显。

Vite 不全面打包，而是精准打包，且依赖原生ESM的能力实现单文件的替换，所以快，

而Webpack需要分析依赖链，对依赖链所有文件进行打包并重新生成Bundle文件并替换Bundle文件，
