## 动态路由及鉴权

### Vue-router

我们项目用的是 Vue-Router。动态路由的核心思路是**：前端定义好所有需要权限的路由配置，但不在启动时全部挂载，而是登录后根据后端返回的权限列表动态添加**。

具体步骤：

1. 路由分两块**：静态路由**（登录页、404 等无需权限的）和**动态路由**（需要权限的页面，在 `meta` 里加 `roles` 或 `permission` 标记）。
2. 登录成功后，后端返回当前用户的权限码列表（比如 `['user:add', 'order:list']`）。
3. 在路由守卫 `beforeEach` 里做判断：如果没添加过动态路由，就调用一个 `generateRoutes` 方法，把动态路由数组过滤出用户有权限的那些，然后 `router.addRoute` 逐一添加（Vue 3 的 `addRoute` 支持嵌套）。
4. 同时把过滤后的路由存到 Vuex/Pinia 里，用来生成侧边栏菜单。菜单组件递归渲染这个树形结构，根据 `meta.hidden` 字段决定是否显示。

注意两个坑：

- 动态添加完后要调用 `next({ ...to, replace: true })`，确保路由完整后再进入页面，否则可能匹配不到。
- 刷新页面时动态路由会丢失，所以在 `App.vue` 的 `created` 或 `setup` 里要重新拉取权限、重新添加路由，一般配合 `localStorage` 存一下权限码，避免每次都白屏。

鉴权菜单其实和动态路由用的是同一份数据，只是展示上再处理一下标题、图标、排序这些。

### React-router

React Router 里动态路由的核心思路也是一样**：登录后根据权限动态添加路由**。不过 React Router 的 API 跟 Vue Router 不太一样。

我们项目用的 v6，做法是：

1. 定义一份全量的路由配置（比如 `allRoutes` 数组），每个路由加一个 `auth` 或 `roles` 字段。
2. 登录后拿权限列表，过滤出有权限的路由。
3. 用 `createHashRouter` 或 `createBrowserRouter` 创建路由实例，把过滤好的路由传进去。不用像 Vue 那样手动 `addRoute`，而是直接生成一个新 router 实例。
4. 但是刷新会丢失，所以需要在根组件里用 `useEffect` 监听 token 变化，重新拉权限、重新生成路由，再用 `useNavigate` 跳转。

**鉴权菜单**：把过滤后的路由树存到 context 或 Redux 里，递归渲染菜单组件，根据 `hidden` 字段隐藏菜单项。

跟 Vue 的主要区别：

- Vue Router 支持动态 `addRoute`，可以在运行时不断追加。React Router v6 没有直接的 `addRoute`，要么提前用生成的路由实例整个替换，要么用 `<Routes>` 结合条件渲染（但这样会在组件里写很多权限判断，比较乱）。
- 所以我们通常用第二种：定义路由表后，用 `useRoutes` 或者直接传入 `router` 对象。菜单生成逻辑和 Vue 几乎一样。

简单说：都是“路由表和权限比对”+“菜单递归渲染”，只是框架 API 不同。

## 页面请求大规模并发的问题

### 背景

大项目涉及，比如数据采集平台或者低代码编辑平台，请求的内容得有序且相对稳定的发送到后端

### 方案

#### 请求队列封装

用常规的方案，封装请求队列，大批流量来之后进入队列 ，通过窗口大小进行限制，只对窗口大小内的几个请求进行处理请求完成后出队。

```js
q = [];
q.push("入队");
q.shift(); //出队
```

```js
class requestQueue {
  constructor(maxCount) {
    this.maxCount = maxCount; // 最大并发数
    this.currentCount = 0; // 当前并发请求数
    this.queue = []; // 请求队列
  }

  add(request) {
    return new Promise((resolve, reject) => {
      this.queue.push({ request, resolve, reject });
      this.processQueue();
    });
  }

  processQueue() {
    // 情况1：当前并发数已达上限，不处理
    if (this.currentCount >= this.maxCount) {
      return;
    }

    // 情况2：队列为空，不处理
    if (this.queue.length === 0) {
      return;
    }

    // 从队列头部取出一个请求
    const { request, resolve, reject } = this.queue.shift();

    // 并发数+1
    this.currentCount++;

    // 执行请求
    Promise.resolve(request())
      .then((result) => {
        // 请求成功
        resolve(result);
      })
      .catch((error) => {
        // 请求失败
        reject(error);
      })
      .finally(() => {
        // 无论成功失败，当前请求结束，并发数-1
        this.currentCount--;

        // 继续处理队列中的下一个请求
        this.processQueue();
      });
  }

  // 可选：获取当前队列状态
  getStatus() {
    return {
      currentCount: this.currentCount,
      queueLength: this.queue.length,
      maxCount: this.maxCount,
    };
  }

  // 可选：清空队列
  clear() {
    this.queue = [];
  }
}

// 使用示例
const queue = new requestQueue(3); // 同时最多3个请求

// 添加请求
for (let i = 0; i < 10; i++) {
  queue
    .add(() => {
      return fetch(`/api/data/${i}`).then((res) => res.json());
    })
    .then((data) => {
      console.log("请求成功:", data);
    })
    .catch((err) => {
      console.log("请求失败:", err);
    });
}

// 文件上传切片示例
const uploadQueue = new requestQueue(5); // 5个并发

chunks.forEach((chunk) => {
  uploadQueue
    .add(() => {
      return uploadChunk(chunk); // 上传单个切片
    })
    .then((result) => {
      console.log(`切片 ${chunk.index} 上传成功`);
    });
});
```

#### 防抖/节流方式进行处理

**防抖原理**（debounce）：事件触发后延迟执行，若在延迟时间内再次触发则重新计时。适用于输入框搜索、窗口resize等场景（确保在指定时间内函数只执行一次，用户多次快速点击后 ，延迟执行，期间重新触发则重置延迟，相当于限制请求发送的频次）。

**节流原理** （throttle）：规定时间内只执行一次事件处理函数（时间戳（规定时间戳内只能执行一次），定时器（延迟执行））。适用于滚动加载、按钮防重复点击、游戏动画等场景。（多次请求会处理，但是会限制请求发送的频率）

以下是纯 JavaScript 的防抖和节流函数代码示例：

##### 防抖（Debounce）

```javascript
// 防抖：延迟执行，期间重新触发则重置延迟
function debounce(fn, delay) {
  let timer = null;
  return function (...args) {
    if (timer) clearTimeout(timer);
    timer = setTimeout(() => {
      fn.apply(this, args);
    }, delay);
  };
}

// 使用示例
const handleInput = debounce((value) => {
  console.log("搜索:", value);
}, 500);

document.getElementById("search").addEventListener("input", (e) => {
  handleInput(e.target.value);
});
```

##### 节流（Throttle）- 时间戳版

```javascript
// 节流：立即执行，规定时间内只执行一次
function throttle(fn, delay) {
  let lastTime = 0;
  return function (...args) {
    const now = Date.now();
    if (now - lastTime >= delay) {
      lastTime = now;
      fn.apply(this, args);
    }
  };
}

// 使用示例
const handleScroll = throttle(() => {
  console.log("滚动位置:", window.scrollY);
}, 200);

window.addEventListener("scroll", handleScroll);
```

##### 节流（Throttle）- 定时器版

```javascript
// 节流：延迟执行，停止触发后还会执行一次
function throttleTimer(fn, delay) {
  let timer = null;
  return function (...args) {
    if (timer) return;
    timer = setTimeout(() => {
      fn.apply(this, args);
      timer = null;
    }, delay);
  };
}

// 使用示例
const handleResize = throttleTimer(() => {
  console.log("窗口大小:", window.innerWidth);
}, 300);

window.addEventListener("resize", handleResize);
```

##### 完整节流（首尾都执行）

```javascript
// 节流：第一次立即执行，最后一次也会执行
function throttleFull(fn, delay) {
  let lastTime = 0;
  let timer = null;

  return function (...args) {
    const now = Date.now();
    const remaining = delay - (now - lastTime);

    if (remaining <= 0) {
      if (timer) {
        clearTimeout(timer);
        timer = null;
      }
      lastTime = now;
      fn.apply(this, args);
    } else if (!timer) {
      timer = setTimeout(() => {
        lastTime = Date.now();
        timer = null;
        fn.apply(this, args);
      }, remaining);
    }
  };
}
```

##### 实际应用场景示例

```javascript
// 1. 输入框搜索防抖
const searchInput = document.querySelector("#search");
const searchDebounce = debounce(async (keyword) => {
  const res = await fetch(`/api/search?q=${keyword}`);
  const data = await res.json();
  console.log("搜索结果:", data);
}, 500);
searchInput.addEventListener("input", (e) => searchDebounce(e.target.value));

// 2. 按钮点击节流（防止重复提交）
const submitBtn = document.querySelector("#submit");
const throttleSubmit = throttle(() => {
  console.log("提交表单");
  // 发送请求...
}, 1000);
submitBtn.addEventListener("click", throttleSubmit);

// 3. 滚动加载节流
let page = 1;
const loadMore = throttle(async () => {
  const res = await fetch(`/api/list?page=${page}`);
  const data = await res.json();
  page++;
  console.log("加载更多数据:", data);
}, 1000);
window.addEventListener("scroll", () => {
  if (window.innerHeight + window.scrollY >= document.body.offsetHeight - 100) {
    loadMore();
  }
});
```

**区别总结**：

- **防抖**：连续触发只执行最后一次（搜索输入、窗口resize）
- **节流**：间隔时间内只执行一次（滚动加载、按钮防连点）

## Vue3中常见工具库以及外部防抖和节流函数

有的，在 Vue3 + TypeScript 项目里，防抖和节流通常不需要自己从头写。最主流的方式是直接使用工具库，主要有两个不错的选择：

### 🛠️ 推荐方案一：VueUse

**VueUse** 是一个专门为 Vue3 设计的 Composition API 工具集，和 TypeScript 配合起来非常顺畅，是目前最推荐的方式。

-  ·······**安装**：

  ```bash
  npm install @vueuse/core
  ```

- **防抖 (Debounce)**：使用 `useDebounceFn`。适合处理输入框搜索、表单验证，等用户停止操作后再执行。

  ```vue
  <script setup lang="ts">
  import { ref } from "vue";
  import { useDebounceFn } from "@vueuse/core";

  const searchQuery = ref("");

  // 声明一个防抖函数：延迟 500ms 执行，意思是我如果连续输入了两个1那只会在500毫秒内打印出一个11
  const debouncedSearch = useDebounceFn(() => {
    console.log("发起搜索请求:", searchQuery.value);
  }, 500);
  </script>

  <template>
    <input v-model="searchQuery" @input="debouncedSearch" />
  </template>
  ```

- **节流 (Throttle)**：使用 `useThrottleFn`。适合处理滚动加载、窗口大小调整、按钮连点等需要持续响应但需限制频率的场景。

  ```vue
  <script setup lang="ts">
  import { useThrottleFn } from "@vueuse/core";
  
  // 声明一个节流函数：每 1000ms 最多执行一次
  const throttledScroll = useThrottleFn(() => {
    console.log("处理滚动事件:", window.scrollY);
  }, 1000);
  </script>
  
  <template>
    <div @scroll="throttledScroll">滚动内容</div>
  </template>
  ```

### 📚 推荐方案二：Lodash

**Lodash** 也是一个非常经典的工具库。如果你的项目已经在用 Lodash，或者不想引入新的依赖，用它就很方便。为了更好的 TypeScript 支持和 Tree-Shaking（按需加载），建议安装 `lodash-es`。

- **安装**：

  ```bash
  npm install lodash-es
  npm install -D @types/lodash-es  # TypeScript 项目需要安装类型声明
  ```

- **防抖与节流**：

  ```vue
  <script setup lang="ts">
  import { ref, onUnmounted } from "vue";
  import debounce from "lodash-es/debounce";
  import throttle from "lodash-es/throttle";
  
  const searchQuery = ref("");
  
  const fetchData = () => {
    console.log("请求数据:", searchQuery.value);
  };
  
  // 使用 Lodash 防抖，500ms 延迟
  const debouncedFetch = debounce(fetchData, 500);
  
  // 使用 Lodash 节流，1000ms 间隔
  const throttledClick = throttle(() => {
    console.log("按钮被点击（节流）");
  }, 1000);
  
  // 在组件卸载时取消，避免内存泄露
  onUnmounted(() => {
    debouncedFetch.cancel();
    throttledClick.cancel();
  });
  </script>
  ```

### 💡 总结对比

| 特性                     | VueUse                                             | Lodash                                                       |
| :----------------------- | :------------------------------------------------- | :----------------------------------------------------------- |
| **Vue 集成度**           | **高**，专为 Vue3 Composition API 设计，使用更自然 | **一般**，通用 JS 库，需注意手动管理 `cancel` 等生命周期     |
| **体积（Tree-Shaking）** | **极小**，按需引入，只打包你用到的函数             | **较小**，`lodash-es` 支持按需引入，但整体库体积比 VueUse 大 |
| **API 风格**             | **Vue 风格**，响应式友好                           | **通用 JS 风格**，对 Vue 无特殊适配                          |

**简单来说**：

- 如果追求**与 Vue3 的最佳集成、极致的轻量级和开发体验**，首选 **VueUse**。
- 如果项目已经**重度使用 Lodash**，或者你需要一个**一劳永逸的通用工具库**，**Lodash** 是稳妥的选择。

## 大文件上传

### 背景

我做项目的时候，做AI工作流的时候，涉及到用户自定义模型的上传，或者遇到大规模的知识库文件的上传，可能比较大上传时间长，有时候因为网络断开或者网络波动导致文件的上传失败或者用户不小心关闭了网页导致上传终止（断点续传），这类问题。

### **专业术语**

- 断点续传
- 断开重连重传
- 切片上传

### 方案

1. 前端将文件进切片，切分为一个个chunk
2. 切片要进行取名，利用hash值进行取名，index进行顺序标记，然后将切片传递给后端
3. 后端对切片进行组合

### 加料

前端切片的话，主进程做很卡顿，利用web-worker多线程切片，处理完后发送给主进程。切完后将blob存储到indexedDB，下次用户来后嗅探未完成的切片，有就尝试继续上传。利用websocket进行实时通知，以及请求序列的控制wss

### 断点续传核心：

这三个概念是断点续传的核心，我给你逐一解释：

---

#### 1. IndexedDB - 浏览器本地数据库

**是什么**：
浏览器自带的 NoSQL 数据库，可以在用户硬盘上存储大量数据（远大于 localStorage 的 5MB 限制）

**在断点续传中的作用**：

```javascript
// 用户上传到一半关闭页面 - 切片保存在 IndexedDB
// 下次打开页面 - 从 IndexedDB 读取未完成的切片继续上传
```

**为什么用它**：

- 存切片文件（Blob 类型）
- 容量大（几百 MB 没问题）
- 持久化（用户不主动清除就一直在）

---

#### 2. wss - WebSocket Secure

**是什么**：
基于 TCP 的全双工通信协议，wss 是它的加密版本（类似 https）

**特点**：

- 实时双向通信（不像 HTTP 必须客户端先请求）
- 长连接（一次连接持续通信）

**在断点续传中的作用**：

```javascript
// 场景1：服务端主动通知"你的第5个切片我收到了"
ws.onmessage = (e) => {
  if (e.data.type === "chunk_received") {
    // 可以把这个切片从待上传队列移除
  }
};

// 场景2：连接断了立刻知道
ws.onclose = () => {
  // 立即暂停上传，保存进度，准备重连
};
```

**用 HTTP 不行吗？**

- HTTP 只能客户端问、服务端答
- WebSocket 可以服务端主动告诉客户端"别传了，我在重启"

---

#### 3. Hash 命名 - 文件指纹

**是什么**：
通过算法（如 MD5）计算出的唯一字符串，类似文件的"身份证号"

```javascript
// 示例
文件 "cat.jpg" 内容 → 计算 → "d41d8cd98f00b204e9800998ecf8427e"
```

**在断点续传中的作用**：

**a. 文件级别的 hash**（整个文件的指纹）

```javascript
// 目的：秒传 + 避免重复上传
fileHash = md5(整个文件)

// 上传前先问后端：这个文件你有了吗？
POST /check-file { fileHash: "d41d8c..." }
// 后端返回：已有 → 秒传成功；部分有 → 传缺失的切片
```

**b. 切片级别的 hash**（每个切片的指纹）

```javascript
// 切片命名格式：fileHash-index-chunkHash
chunkName = `${fileHash}-${index}-${md5(chunk)}`

// 为什么还要切片的hash？
- 防止传输过程中数据损坏
- 后端合并前校验："这个切片完整吗？"
```

**hash 怎么算**：

```javascript
// 使用 spark-md5 库，分块计算避免卡顿
const spark = new SparkMD5();
// 分块读取文件，逐步更新 hash
spark.append(chunkData);
```

---

#### 一句话总结

- **IndexedDB**：把切片存用户硬盘，页面关了也不丢
- **wss**：实时知道上传状态，断网立刻感知
- **hash**：文件的身份证，用来秒传和校验数据完整性

## setTimeout 与异步

`setTimeout` 实现异步的核心原理，不是靠它自己“计算时间”，而是靠浏览器的**事件循环机制**和**多线程模型**。

可以分两层来理解它的运作方式：

### 1. 核心流程：把任务“寄存”出去

JavaScript 引擎是单线程的，它没办法一边计时一边执行代码。所以 `setTimeout` 的计时工作其实是交给**浏览器（或 Node.js）的定时器线程**去做的。

具体的执行流水线如下：

- **第一步：交单**。
  JS 引擎执行到 `setTimeout(fn, 1000)`，它只做一件事**：把 `fn` 连同“1000ms 后执行”这个指令，提交给浏览器的定时器线程**。交完立马接着执行下一行代码（这就是“非阻塞/异步”的来源）。

- **第二步：后台计时**。
  浏览器的定时器线程开始计时。此时 **JS 主线程是空闲的吗？不，它正在疯狂执行 `setTimeout` 后面的同步代码**。

- **第三步：排队等号**。
  1000ms 到了，定时器线程不会直接把 `fn` 塞给 JS 主线程（那会打断当前执行，导致不可预测的错误）。它会**把 `fn` 放进一个叫“任务队列”的地方**，让它排队。

- **第四步：事件循环召唤**。
  JS 主线程把手头的同步代码全执行完了，执行栈清空。这时，**事件循环** 就会去任务队列里看一眼：“咦，有活干了”，然后把 `fn` 从队列里拽出来，推到执行栈里执行。

### 2. 为什么有时候 1000ms 不精准？

正是因为上面的**第三步排队机制**，导致了延迟只多不少。

如果 `setTimeout` 到点了，但主线程正在死循环或者处理一个长达 2 秒的复杂计算，`fn` 就必须在队列里干等着。

```javascript
setTimeout(() => {
  console.log("我等了很久才出来");
}, 0); // 即便是 0ms 延迟

// 假设这是一个耗时 2 秒的循环
for (let i = 0; i < 10000000000; i++) {
  // 主线程被堵在这里 2 秒
}

// 输出顺序：
// 1. 循环结束
// 2. 才会输出 '我等了很久才出来' （实际延迟 2000ms+，而不是 0ms）
```

### 3. 进阶视角：`setTimeout` 定义了一个“宏任务”

在面试或深入研究时，可以记住这个概念：`setTimeout` 产生的任务属于 **宏任务**。事件循环每一轮只会从任务队列里**取一个**宏任务来执行。

如果你遇到更精细的任务调度（比如需要在一个动画帧之后立刻执行），仅仅理解 `setTimeout` 还不够，可能会用到 `Promise` 或 `requestAnimationFrame`。需要我展开讲讲它们和 `setTimeout` 在调度时机上的具体区别吗？

## Promise.resolve

`Promise.resolve` 在 Promise 体系中扮演着两个核心角色**：创建一个立即成功的 Promise**，以及**在异步流程中标准化任意值为 Promise**。

简单来说，它的作用是**：把一个值（可能是普通值、thenable 对象，或另一个 Promise）包装成一个 Promise 对象，并确保最终的行为符合 Promise 规范。**

我们可以分两层来理解它的具体行为：

### 1. 静态方法：`Promise.resolve(value)`

这是最常用的形式，它会根据传入的参数类型，产生不同的结果：

| 参数类型          | 行为                                                                                                                                       | 代码示例                                                                                                         |
| :---------------- | :----------------------------------------------------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------------- |
| **普通值**        | 返回一个**状态为 Fulfilled** 的 Promise，其 `[[PromiseResult]]` 即为该值。                                                                 | `Promise.resolve('hello').then(v => console.log(v))` // 输出 hello                                               |
| **Promise 实例**  | 具有**幂等性**。如果传入的是一个 Promise，它会直接返回这个**完全相同的 Promise 实例**，不会做任何包装或修改。                              | `const p1 = new Promise(...);` <br> `const p2 = Promise.resolve(p1);` <br> `console.log(p1 === p2);` // **true** |
| **Thenable 对象** | 尝试调用该对象的 `then` 方法，并**根据 thenable 的最终状态来决定返回的 Promise 状态**。常用于兼容旧式异步库（如 jQuery 的 `$.Deferred`）。 | `Promise.resolve({ then: (res, rej) => res(10) })`                                                               |

### 2. 构造函数内的参数名：`resolve`

在 `new Promise((resolve, reject) => {})` 中，这个 `resolve` 是构造函数传给执行器的**函数引用**。

它的职责是**：将 Promise 的状态从 Pending 改为 Fulfilled**。

```javascript
new Promise((resolve, reject) => {
  setTimeout(() => {
    // 这里的 resolve 就是用来改变状态的函数
    resolve("异步操作成功");
  }, 1000);
});
```

**注意**： 调用此处的 `resolve` 并不会阻塞后续代码的执行（除非使用 `await`），它只是发出了一个“状态变更指令”。

### 3. 进阶用法：在 `async` 函数中的隐形转换

这其实是你最常使用 `Promise.resolve` 逻辑的场景，只是它被语法糖隐藏了。

在一个 `async` 函数中，**`return` 的值会被自动调用 `Promise.resolve` 进行包装**。

```javascript
async function getData() {
  // 尽管你返回的是普通对象，但它实际返回的是 Promise.resolve({ name: 'Li' })
  return { name: "Li" };
}

// 所以你可以使用 .then
getData().then((user) => console.log(user));
```

### 4. 实践应用场景

掌握 `Promise.resolve` 后，最实用的场景是**统一异步/同步接口**：

```javascript
function getUserFromCache(id) {
  const cached = localStorage.getItem(id);
  if (cached) {
    // 即便是同步读取的数据，也包装成 Promise 返回
    // 这样调用方永远可以用 .then 或 await 来处理，不用区分数据类型
    return Promise.resolve(JSON.parse(cached));
  } else {
    // 异步请求
    return fetch(`/api/user/${id}`).then(r => r.json()】);
  }
}
```

## 前端实现页面截图（canvas）

### 一、 背景与痛点

在前端业务中，将 DOM 内容转化为图片的需求非常普遍。结合场景，主要有以下几类：

1.  **内容预览与分享**：例如飞书文档列表页，希望不点进详情就能看到内容缩略图；或者将页面特定区域（如排行榜、战报）保存为 PNG 分享到社交平台。
2.  **业务凭证留存**：例如医疗行业的电子化验单、挂号单、处方笺；电商的订单详情、物流底单。这些内容通常由 CSS 动态渲染，需要将其固化为不可篡改的图片格式（PNG/PDF）提供给用户打印或存档。
3.  **设计交付与走查**：设计师或产品经理看到线上 Bug 或优秀案例，想一键截图导出并附带 CSS 样式信息，用于修改文档或设计稿。

### 二、 技术方案对比

针对“内容导出为 PNG”这一核心目标，前端主要有以下三种技术路径：

| 方案                          | 原理                                                                         | 优点                                                   | 缺点                                                                                                          | 适用场景                                         |
| :---------------------------- | :--------------------------------------------------------------------------- | :----------------------------------------------------- | :------------------------------------------------------------------------------------------------------------ | :----------------------------------------------- |
| **1. Canvas 原生绘制**        | 利用 Canvas API `drawImage` 将图片、视频帧画进去，或 `fillText` 绘制文字。   | 性能极好，灵活度极高，可精确控制像素级输出。           | 对于复杂的 DOM 结构（如 `div` 嵌套、`flex` 布局）需手动换算坐标重绘，**代码侵入性强，无法直接“截取”网页**。   | 纯图片合成、海报编辑器（Canvas 就是画板）。      |
| **2. html2canvas (主流选择)** | 遍历 DOM 树，解析每个节点的 Computed Style，将其“翻译”成 Canvas 的绘制指令。 | **非侵入式**，只需传入 `selector`，自动解析 CSS。      | 遇到 `box-shadow`、`border-radius` 叠加、`SVG` 或 `Canvas` 内嵌内容时**可能有解析偏差或空白**；性能开销较大。 | **业务后台截图、单据导出、缩略图生成**。         |
| **3. Puppeteer (无头浏览器)** | 启动一个真实的 Chrome 内核（Headless），打开网页后调用 `Page.screenshot()`。 | **还原度 100%**，CSS 新特性、WebGL、字体渲染完全一致。 | **极重**（需启动浏览器进程），无法在前端浏览器直接运行，必须部署在 **Node.js 服务端**。                       | 服务端批量生成 PDF、页面监控、高精度设计稿还原。 |

> **补充说明**：“无头表格”或“无头 UI 库”通常是指类似 `jsPDF` 结合 Canvas 或直接在 Node 端通过虚拟 DOM 计算坐标的方案，但普适性和还原度不如 Puppeteer。

### 三、 实际落地设计：通用截图工具协议

为了实现“通用性”和“按需截图”，我们需要设计一个规范的工具函数。补充具体协议如下：

**1. 模式选择：函数式 vs 组件式**

- **函数式（推荐）**：接收参数，返回 Promise。灵活，不侵入视图层。
- **组件式**：封装一个 `<ScreenShot>` 组件，利用插槽包裹内容。适合特定区域的显隐控制。

**2. 隐藏 Canvas 的策略（隐藏画布技术）**
为了保证截图过程不影响用户当前页面的视觉，通常将解析用的 Canvas 放在屏幕外：

```css
.offscreen-canvas {
  position: fixed;
  top: -9999px;
  left: -9999px;
  visibility: hidden;
  pointer-events: none;
}
```

### 四、 代码编写示例（TypeScript + html2canvas）

以下是实现一个具备**通用性**、**防抖**、**CDN 上传**和**下载回退**的 Hook：

```typescript
import html2canvas from "html2canvas";
import { useDebounceFn } from "@vueuse/core"; // 结合你之前问的防抖

interface ScreenshotOptions {
  selector: string; // 对应标签选择器
  scale?: number; // 清晰度，建议 2-3
  backgroundColor?: string;
  fileName?: string;
}

export const useScreenshot = () => {
  // 1. 核心截图逻辑
  const capture = async (options: ScreenshotOptions): Promise<File | null> => {
    const { selector, scale = 2, backgroundColor = "#ffffff" } = options;

    const element = document.querySelector(selector);
    if (!element) {
      console.error(`[Screenshot] 选择器 "${selector}" 未找到元素`);
      return null;
    }

    try {
      // 2. 调用 html2canvas 解析 DOM -> Canvas
      const canvas = await html2canvas(element as HTMLElement, {
        scale, // 提高分辨率，解决模糊问题
        backgroundColor, // 避免 PNG 透明背景导致的显示异常
        allowTaint: false, // 安全限制，防止跨域图片污染
        useCNAME: true, // 尝试用 CNAME 解决图片跨域
        logging: false, // 关闭调试日志
      });

      // 3. Canvas 转 Blob (为上传 CDN 做准备)
      return new Promise((resolve) => {
        canvas.toBlob((blob) => {
          if (blob) {
            const file = new File(
              [blob],
              `${options.fileName || "screenshot"}.png`,
              { type: "image/png" },
            );
            resolve(file); // 外部函数由于 await 了这个 Promise，会拿到 resolve(file) 传递出来的 file
          } else {
            resolve(null);
          }
        }, "image/png");
      });
    } catch (error) {
      console.error("截图失败:", error);
      return null;
    }
  };

  // 4. 上传 CDN 逻辑 (模拟)
  const uploadToCDN = async (file: File): Promise<string> => {
    const formData = new FormData();
    formData.append("file", file);
    // 实际项目中替换为你的上传接口
    // const res = await axios.post('/api/upload', formData);
    // return res.data.url;

    // 模拟返回 URL
    return URL.createObjectURL(file);
  };

  // 5. 本地下载兜底
  const downloadImage = (file: File) => {
    const url = URL.createObjectURL(file);
    const link = document.createElement("a");
    link.href = url;
    link.download = file.name;
    link.click();
    URL.revokeObjectURL(url);
  };

  // 6. 暴露给业务用的防抖方法 (处理高频点击)
  const handleScreenshot = useDebounceFn(async (options: ScreenshotOptions) => {
    const file = await capture(options);
    if (!file) return;

    // 假设业务需要上传到 CDN（Content Delivery Network） 获取永久链接
    const cdnUrl = await uploadToCDN(file);
    console.log("截图已上传 CDN:", cdnUrl);

    // 同时保留本地下载能力
    // downloadImage(file);
  }, 500);

  return {
    capture,
    handleScreenshot,
  };
};
```

### 五、 避坑指南（你内容中隐含的难点补充）

1.  **Canvas 污染（Tainted Canvas）**：
    - **现象**：画布上有跨域图片，调用 `toBlob` 时报错。
    - **解决**：后端图片必须配置 CORS 响应头（`Access-Control-Allow-Origin`），且前端加载图片时加 `crossorigin="anonymous"` 属性。

2.  **滚动条与溢出隐藏**：
    - **现象**：截图中包含了浏览器的滚动条，或者 `overflow: auto` 区域只截到了可视区。
    - **解决**：在调用 `html2canvas` 前，临时修改 DOM 样式 `overflow: visible` 和 `height: auto`，截图后再复原。

3.  **Canvas 模糊问题**：
    - **原因**：屏幕 DPI 与 Canvas 默认像素比不一致。
    - **解决**：代码中已包含 `scale: 2` 或 `scale: window.devicePixelRatio`，将画布物理像素放大，再通过 CSS 缩小显示（或直接下载大图）。

## 移动端适配问题

### 方案对比

- **多端分离**：不同端加载不同页面，隔离彻底但维护成本高。
- **动态加载 CSS**：根据 UA 判断加载不同样式文件，易产生样式冲突。
- **响应式布局**：一套代码通过媒体查询适配，维护成本最低，推荐使用。
- **CSS 预处理器**：配合响应式使用，通过变量和 Mixin 提升开发效率。

### 解决方案

#### 1. 设置视窗

```html
<meta
  name="viewport"
  content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no"
/>
```

- `width=device-width`：视窗宽度等于设备宽度，是 rem 方案生效的前提。
- `viewport-fit=cover`：适配 iPhone 刘海屏，让背景色延伸到安全区外。

#### 2. 掌握媒体查询

```css
body {
  font-size: 16px;
}
@media (min-width: 780px) and (max-width: 1024px) {
  body {
    font-size: 18px;
  }
}
```

**补全建议**：采用移动优先原则（`min-width`），从小屏向大屏递进覆盖。

#### 3. 掌握 Flex 布局

- **主轴方向**：`flex-direction` 决定排列方向（row / column）。
- **次轴方向**：与主轴垂直的方向。
- **对齐方式**：`justify-content` 控制主轴，`align-items` 控制次轴。
- **弹性属性**：`flex: 1` 表示元素会按比例分配剩余空间。

**关键补全**：移动端 Flex 容器内文字溢出时，需加 `min-width: 0` 防止撑开布局。

```css
.flex-item {
  flex: 1;
  min-width: 0;
  overflow: hidden;
  text-overflow: ellipsis;
  white-space: nowrap;
}
```

#### 4. 图片响应式处理

```html
<picture>
  <source srcset="img-tablet.jpg" media="(min-width: 780px)" />
  <source srcset="img-desktop.jpg" media="(min-width: 1080px)" />
  <img src="img-mobile.jpg" alt="默认图片" />
</picture>
```

**补全建议**：结合 `srcset` 的 `2x`、`3x` 标识，可同时适配 Retina 高清屏：

```html
<source srcset="img@2x.jpg 2x, img@3x.jpg 3x" />
```

#### 5. rem 和 em 单位

```css
html {
  font-size: 16px;
}
@media (min-width: 780px) and (max-width: 1024px) {
  html {
    font-size: 18px;
  }
}
.header {
  height: 1rem; /* 默认 16px，屏幕 780px~1024px 时为 18px */
}
```

**关键区分**：

- **rem**：相对于根元素 `html` 的 `font-size`，适合全局统一缩放。
- **em**：相对于当前元素或父元素的 `font-size`，适合组件内部局部缩放。

**补全建议**：现代方案可用 **vw** 替代 rem，无需 JS 动态计算：

```css
.header {
  height: 13.33vw; /* 设计稿 750px 下，100px 对应 13.33vw */
}
```

### 移动端坑点

- **1px 边框变粗**：Retina 屏下用伪元素 + `transform: scaleY(0.5)` 实现物理 1px。
- **键盘弹起遮挡**：`position: fixed` 在软键盘弹出时会失效，关键输入框需避开底部。
- **安全区适配**：结合 `env(safe-area-inset-bottom)` 为 iPhone 底部留出安全距离。

## 访问端识别

### 背景

一个链接访问页面，需要同时适配 PC 端和 Mobile 端，根据设备类型展现不同的页面布局或内容。

### 方案核心思路

1. **先识别**：判断当前访问设备是 PC 还是 Mobile。
2. **后渲染**：根据识别结果，加载对应组件或执行跳转逻辑。

### 识别端：User-Agent 方案

**核心 API**：`navigator.userAgent`

```js
console.log(navigator.userAgent);
// 移动端典型特征：Mobi、Android、iPhone、iPad

function isMobile() {
  return /Mobi|Android|iPhone|iPad/i.test(navigator.userAgent);
}

if (isMobile()) {
  // 跳转移动端页面
  window.location.href = "/mobile";
}
```

**补全建议**：更完善的 UA 正则（覆盖主流设备）：

```js
function isMobile() {
  return /Android|webOS|iPhone|iPad|iPod|BlackBerry|IEMobile|Opera Mini/i.test(
    navigator.userAgent,
  );
}
```

### 框架中使用：跨组件传递设备信息

**React 实现**（Context 跨层级传递）：

```jsx
// 1. 创建 Context
const DeviceContext = React.createContext();

// 2. 顶层注入
<DeviceContext.Provider value={{ isMobile: isMobile() }}>
  <App />
</DeviceContext.Provider>;

// 3. 子组件使用
function Header() {
  const { isMobile } = useContext(DeviceContext);
  return <div>{isMobile ? "移动端" : "PC端"}</div>;
}
```

**Vue 实现**（Provide / Inject 依赖注入）：

```js
// 1. 顶层注入（重要 API）
provide("isMobile", isMobile());

// 2. 子组件使用
const isMobile = inject("isMobile");
```

### 服务端识别（SSR 场景）

当项目涉及服务端渲染（Next.js / Nuxt）时，仅靠前端 UA 识别会导致 **hydration 不匹配** 。

**Node.js 端识别**：

```js
// 从请求头获取 User-Agent
export function getServerSideProps({ req }) {
  const ua = req.headers["user-agent"] || "";
  const isMobile = /Mobi|Android/i.test(ua);
  return { props: { isMobile } };
}
```

### 注意事项

1. **UA 可被伪造**：User-Agent 仅作参考，不能用于权限校验。
2. **iPadOS 13+ 特例**：iPad 新版系统 UA 与 Mac 相同，需配合 `navigator.maxTouchPoints` 辅助判断。

```js
const isIPad = /Macintosh/i.test(ua) && navigator.maxTouchPoints > 1;
```

3. **屏幕尺寸辅助**：结合 `window.innerWidth` 作为降级方案（`width <= 768` 视为移动端）。

### 渲染策略选择

| 策略             | 实现方式                              | 适用场景               |
| :--------------- | :------------------------------------ | :--------------------- |
| **跳转分流**     | 识别后重定向到不同域名（`m.xxx.com`） | 两套完全独立的项目     |
| **动态组件**     | 同页面内根据 `isMobile` 渲染不同组件  | 一套代码，组件级适配   |
| **CSS 媒体查询** | 组件相同，仅样式变化                  | 布局差异小，仅视觉调整 |

---

### 其他方案

最好是用响应式来做，通过响应式来实现不同端的网页适配，或者通过查询访问媒体的最小宽度（useMediaQuery）来进行端的判断（不太准确）

## QPS 达到峰值

### 背景

当前端的应用的每秒查询次数(QPS)达到峰值会对服务端产生较大的压力，可能导致系统崩溃，我们需要进行请求的降低或者优化管理请求的发出来减轻负担。

### 方案

## 图片懒加载方案对比

以下是对图片懒加载方案的总结与代码示例，每种方案附有简洁的注释和用法说明。

### 方案一：`loading='lazy'`（原生 HTML 属性）

最简洁的懒加载方式，浏览器自动判断图片进入可视区时加载，无需额外 JavaScript。

```html
<!-- 只需要在 img 标签上加上 loading="lazy" 即可 -->
<img src="example.jpg" alt="懒加载图片" loading="lazy" width="600" height="400">

<!-- 注释：
  - 兼容性：Chrome 77+、Edge 79+、Firefox 75+，Safari 15.4+ 开始支持
  - 优点：零 JS 开销，浏览器自动优化
  - 缺点：较旧的浏览器不支持；无法精细控制占位样式；图片加载前占位区域无背景等
-->
```

### 方案二：Intersection Observer API（现代推荐方式）

利用浏览器提供的 `IntersectionObserver` 监测图片与视窗的交集，当图片进入可视区时动态替换 `src`。

```html
<!-- 占位图使用小体积图片或透明图，真实地址放在 data-src 中 -->
<img class="lazy" data-src="real-image.jpg" src="placeholder.png" alt="懒加载图片" width="600" height="400">

<script>
  // 创建一个观察器，当目标元素与视窗交叉比例 > 0 时触发
  const observer = new IntersectionObserver((entries, observer) => {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        const img = entry.target;
        // 将 data-src 中的真实地址赋给 src
        img.src = img.dataset.src;
        // 加载完成后取消观察该图片，避免回调重复执行
        observer.unobserve(img);
      }
    });
  });

  // 获取所有需要懒加载的图片并开始观察
  document.querySelectorAll('img.lazy').forEach(img => observer.observe(img));
</script>

<!-- 注释：
  - 优点：性能高（不占用主线程滚动事件），代码简洁，兼容性好（IE 不兼容，可 polyfill）
  - 缺点：需要了解 API，但比原始滚动监听更加现代和高效
  - 可额外配置 rootMargin 提前加载，如 { rootMargin: "100px" }
-->
```

### 方案三：滚动事件 + 手动计算（最原始的做法）

监听 `scroll` 事件，通过 `getBoundingClientRect()` 判断图片是否已进入视窗，性能较差，需要添加节流处理。

```html
<img class="lazy" data-src="real-image.jpg" src="placeholder.png" alt="懒加载图片" width="600" height="400">

<script>
  function lazyLoad() {
    // 获取所有未加载的懒加载图片
    const images = document.querySelectorAll('img.lazy:not([data-loaded])');
    images.forEach(img => {
      // 获取元素相对于视窗的位置
      const rect = img.getBoundingClientRect();
      // 判断是否进入视口（上边界 < 窗口高度 且 下边界 > 0）
      if (rect.top < window.innerHeight && rect.bottom > 0) {
        img.src = img.dataset.src;
        img.setAttribute('data-loaded', 'true'); // 标记已加载，避免重复操作
      }
    });
  }

  // 滚动时加载，必须加节流/throttle 减少计算频率
  let ticking = false;
  window.addEventListener('scroll', () => {
    if (!ticking) {
      requestAnimationFrame(() => {
        lazyLoad();
        ticking = false;
      });
      ticking = true;
    }
  });

  // 页面加载时立即检查一次，避免首屏图片不加载
  window.addEventListener('load', lazyLoad);
</script>

<!-- 注释：
  - 缺点：滚动事件触发频繁，即使节流依然会反复执行判断，性能较差
  - 优点：兼容性极佳，适用于所有浏览器
  - 生产环境不推荐，除非需要兼容极旧浏览器且无法使用 polyfill
-->
```

### 方案四：使用现成的第三方库（lazysizes / lazyload）

成熟的库封装了懒加载逻辑，并提供占位、响应式图片、出错重试等高级功能。

```html
<!-- 以 lazysizes 为例，先引入它的 JS 文件 -->
<script src="https://cdn.jsdelivr.net/npm/lazysizes@5.3.2/lazysizes.min.js" async></script>

<!-- 将图片的 class 设为 lazyload，真实地址放在 data-src 或 data-srcset 中 -->
<img data-src="real-image.jpg" src="placeholder.jpg" class="lazyload" alt="懒加载图片" width="600" height="400">

<!-- 注释：
  - lazysizes 自动监听滚动/交互，无需手动写任何加载逻辑
  - 支持响应式图片（data-srcset），动画进入效果，以及 Intersection Observer 后备方案
  - 类似的库还有：vanilla-lazyload、lozad.js 等
  - 优点：功能全面，可靠性高，适合快速开发
  - 缺点：需要额外引入第三方 JS，增加页面依赖
-->
```

---

### 方案对比总结

| 方案                  | 实现难度 | 性能               | 兼容性                  | 适用场景                               |
| --------------------- | -------- | ------------------ | ----------------------- | -------------------------------------- |
| `loading='lazy'`      | 极低     | 最优（浏览器原生） | 现代浏览器              | 简单项目，不在意旧浏览器支持           |
| Intersection Observer | 中       | 高                 | IE 不支持，其余主流均可 | 现代前端项目，兼顾性能与可控           |
| scroll 事件 + 手算    | 中       | 低（滚动卡顿风险） | 所有浏览器              | 极少数需要兼容古董浏览器的场景         |
| 第三方库 (lazysizes)  | 低       | 高                 | 自动降级                | 快速开发，需要强大功能（如响应式图片） |

> 建议：新项目优先使用 `loading="lazy"` 或 Intersection Observer。若需要精细控制占位、背景色、图片渐现动画等，可采用 lazysizes 这类成熟库。原始 scroll 监听方式仅供理解原理，实践中避免使用。





## 上拉加载下拉刷新

### 实现逻辑

这个 demo 实际做的是“下拉刷新 + 上拉加载更多”，不是“下拉加载 + 上拉刷新”。整体思路很典型：用一个可滚动容器承载列表，在顶部放一个“下拉刷新条”，底部放一个“加载更多提示”，然后分别监听 `touch` 和 `scroll` 事件来触发两套逻辑。

**整体结构**
在 [下拉加载上拉刷新.html](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:51>) 里，`.scroll-container` 是滚动容器；顶部的 `.pull-refresh` 是刷新提示条，初始高度为 `0`；中间的 `#feedList` 放列表；底部的 `#loadMoreTrigger` 放“加载更多”状态文案。  
数据层很简单，核心状态都在 [318 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:318>) 附近：
`allItems` 存当前列表数据，`isLoadingMore` 和 `isRefreshing` 防止并发触发，`hasMore` 控制是否还能继续加载，`MAX_ITEMS_LIMIT` 是最大条数上限。

**下拉刷新是怎么实现的**
下拉刷新主要靠 `touchstart / touchmove / touchend` 三个事件，代码在 [515 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:515>) 到 [593 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:593>) 一段。

1. `onTouchStart`
在 [515 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:515>) 记录手指起点 `touchStartY`，并记住当前滚动位置 `scrollTop`。

2. `onTouchMove`
在 [524 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:524>) 计算 `deltaY = currentY - touchStartY`。  
只有满足这几个条件才认为是“下拉刷新手势”：
- 当前已经滚到顶部：`scrollTop <= 0`
- 手指向下拖：`deltaY > 0`
- 当前不在刷新中

满足后会：
- `e.preventDefault()` 阻止浏览器原生回弹
- 根据拖动距离动态修改顶部刷新条高度：`pullRefreshBar.style.height = ...`
- 当高度超过阈值 `pullThreshold = 60` 时，文案切成“松手刷新”，图标旋转 180 度
- 没超过阈值时，文案保持“下拉刷新”

这里还有两个小设计：
- `newHeight = deltaY * 0.8`，做了阻尼效果，拖动越长视觉上不会 1:1 增长
- `refreshBarMaxHeight = 70`，限制最大拉伸高度

3. `onTouchEnd`
在 [567 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:567>) 判断：如果当前确实在 pull，且刷新条高度超过阈值，就调用 `performRefresh()`；否则直接收起刷新条。

4. `performRefresh`
刷新执行逻辑在 [424 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:424>)：
- 先把 `isRefreshing = true`
- 固定刷新条高度为 `56px`，显示“刷新中”
- `await delay(600)` 模拟接口请求
- 重新生成一批新数据：`generateBatch(INITIAL_COUNT)`
- 直接覆盖旧列表：`allItems = [...newData]`
- 调 `renderFullList()` 整体重渲染
- 最后收起刷新条，恢复状态

所以“下拉刷新”的本质是：  
`顶部拉伸反馈 + 超阈值判定 + 松手触发刷新 + 用新数据整体替换列表`

**上拉加载更多是怎么实现的**
这部分靠 `scroll` 事件，核心在 [596 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:596>) 的 `onScrollHandler()`。

1. 监听滚动接近底部
在 [604 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:604>) 用这个条件判断是否快到底了：

```js
scrollHeight - scrollTop - clientHeight <= 70
```

意思是：内容总高度 - 已滚动距离 - 可视区高度 <= 70px  
也就是“离底部还剩 70px 时，就提前触发加载”。

2. 防抖
它在 [599 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:599>) 到 [612 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:612>) 用了一个 `setTimeout(..., 50)`，相当于做了一个很轻量的防抖，避免滚动时频繁计算。

3. `loadMoreData`
真正加载逻辑在 [452 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:452>)：
- 如果正在加载或正在刷新，直接返回
- 如果 `hasMore` 为假，或者已经到 `MAX_ITEMS_LIMIT`，则显示“没有更多”
- 否则设置 `isLoadingMore = true`
- 底部 UI 切到 loading 状态：`updateLoadMoreUI(true, true)`
- `await delay(500)` 模拟接口请求
- 随机生成 5 到 8 条新数据
- `allItems.push(...newItems)` 追加到数组末尾
- `appendItemsToEnd(newItems)` 只把新增项插入 DOM 尾部，而不是整表重绘
- 如果达到上限，关闭 `hasMore`

所以“上拉加载”的本质是：  
`监听滚动接近底部 + 异步取新数据 + 追加到现有列表尾部 + 更新底部状态`

**这份 demo 值得注意的设计点**
- 下拉刷新和上拉加载互斥：`isRefreshing` / `isLoadingMore` 防止同时触发。
- 刷新是“整体替换数据”，加载更多是“尾部追加数据”。
- 顶部用 `touch` 事件，因为它要拿到“手指向下拖”的过程；底部用 `scroll` 事件，因为只关心“是否快滚到底了”。
- `resetPullRefreshBar()` 在 [416 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:416>) 统一负责把刷新条复位，这样状态收口比较干净。
- `watchGlobalLocks()` 在 [624 行](</abs/path/d:/Code/VuePress/vuepress_blog/tests/下拉加载上拉刷新.html:624>) 会在刷新时把 `overscrollBehavior` 改成 `none`，避免刷新过程中出现浏览器原生回弹干扰。







## windows对象上频繁绑定内容的风险

这个问题其实在实际开发中挺容易踩坑的。频繁往 `window` 上绑定东西，最直接的风险就是**全局命名空间污染**。

比如说你在一个模块里写了 `window.currentUser = ...`，另一个同事不知道，也写了个 `window.currentUser`，后面的人就等着熬夜吧，互相覆盖、取值莫名其妙出错。而且这种隐式依赖还特别难追踪，不知道这个变量是谁挂在 window 上的，代码维护起来很痛苦。

另一个就是**内存泄漏**。如果你往 window 上绑了一个很大的对象或者数组，比如存了上千条列表数据，或者挂了一个 DOM 元素的引用，然后页面切换或者组件销毁的时候忘记清掉，这些东西就永远占着内存，因为 window 一直活着，它引用的东西回收不掉。尤其 spa 应用里，反复进出页面，内存会一直涨。

还有就是**性能和可测试性**。频繁操作 window 对象，特别是事件监听这种，比如 `window.addEventListener('scroll', ...)`，如果忘记解绑，会越来越累积，每次滚动都要执行一堆回调，页面会变卡。而且写单元测试的时候，依赖全局 window 的代码很难测，每次都要 mock，环境也很容易相互干扰。

所以我的习惯是尽量不直接往 window 上挂东西，真的需要全局数据的话，用模块导出、状态管理工具比如 Vuex 或者 Pinia。实在要用，也加个**命名空间**，比如 `window.__APP__ = { ... }`，集中管理，别散得到处都是。事件监听也记得在组件销毁时解绑，或者用 AbortController 统一取消。



## Monorepo主要解决了什么问题，Monorepo和git子模块有什么区别

嗯，Monorepo 这个词就是“单一代码库”的意思，它主要是解决**多项目管理的复杂度**。当你有多个相互关联的项目，比如一个前端项目拆成了几个包，或者一个产品同时有 web、app、后台服务，如果每个项目单独一个 git 仓库，管理起来会越来越头疼。

Monorepo 解决的核心痛点有几个：首先是**代码复用和依赖管理**。在多仓库模式下，公共组件或者工具函数想要共享，就得发布到 npm 私有源，或者用 git 子模块，维护起来很麻烦。Monorepo 里所有代码都在一个仓库，直接 `import` 就行，不用发包，改一个公共函数所有项目立刻生效。

其次是**统一工具链和配置**。比如 eslint、prettier、typescript、构建脚本这些，多仓库的话每个仓库都要配一遍，而且容易版本不一致。Monorepo 可以一套配置管所有子项目，升级也只需要改一次。

还有就是**跨项目的原子提交和协作**。多仓库如果一次需求要同时改 A 和 B 两个仓库，得先改 A 发版，再改 B 引新版本，中间还有版本号管理、依赖升级这些流程。Monorepo 里一次 commit 就能同时改多个包，CI 也能一次跑完所有测试，不会出现“改了接口但调用方没跟上”的问题。

至于**和 git 子模块的区别**，子模块是 git 提供的一种方式，让你在一个仓库里引用另一个独立的 git 仓库。它的特点是：子仓库还是独立的，有自己独立的 commit 历史、分支、远程地址。主仓库只记录子模块的某个 commit 哈希。它的缺点是：操作很麻烦，更新子模块要进到子目录里 pull，克隆主仓库要加 `--recursive` 才能拉子模块，新手很容易忘记。而且跨模块改代码极其繁琐，因为要分别 commit、分别 push。所以子模块更适合那种“很少改动、只是引用的外部依赖”，比如一个第三方库或者公共配置模板。

Monorepo 并不是用子模块实现的，而是用像 **pnpm workspace**、**Turborepo**、**Nx**、**Lerna** 这类工具，把多个项目放在同一个仓库的不同目录，同时通过软链接和缓存来解决构建效率问题。它和子模块的本质区别是：Monorepo 里的所有代码版本是一致的，改起来是原子操作；子模块是分仓库管理的，改起来像跨仓库协作。

所以总结一下，Monorepo 适合内部多个项目紧密协作的场景，子模块更适合引用外部稳定依赖的场景。

## 什么是SSE

SSE 全称是 **Server-Sent Events**，它是一种基于 HTTP 的服务器推送技术，允许服务器主动向浏览器发送数据，而不用等浏览器先发请求。

它的工作方式很简单：浏览器通过 `EventSource` API 向服务器发起一个连接，这个连接保持打开状态，服务器可以随时往这个连接里写入数据，浏览器以事件的形式接收。

**SSE 的特点**主要有这几个：它是单向的，服务器到浏览器，浏览器不能通过同一个连接往服务器发数据；基于 HTTP 协议，不需要特殊协议比如 WebSocket 那种握手；自动重连，连接断开后浏览器会自己尝试重新连接；支持自定义事件类型和消息 ID，客户端可以断点续传。

**和 WebSocket 的区别**，WebSocket 是双向实时通信，适合聊天、游戏这种场景；SSE 更轻量，只需要服务器推送数据的话用它就够了，比如股票行情、通知提醒、日志流。WebSocket 需要单独协议，部署稍复杂；SSE 直接用 HTTP，很多 CDN 和代理都支持。

**在浏览器端的用法**也很简单：`new EventSource('/stream')`，然后监听 `onmessage` 或按事件类型监听。服务器需要设置响应头 `Content-Type: text/event-stream`，数据格式是 `data: 内容\n\n`，多个字段用 `\n` 分隔。

**限制**也有：同一个域最多并发 6 个连接，而且 IE 和旧版 Edge 不支持，需要 polyfill。另外只能传文本，二进制数据需要自己编码。

总结一下，SSE 适合做**单向的、实时性要求不是极高**的推送场景，比如消息通知、监控看板、AI 对话里的流式响应。相比轮询它更省资源，相比 WebSocket 它更简单。

## 用户反馈白屏

这个问题其实挺实际的，线上出白屏是大概率会遇到的事情。我一般会按步骤排查，先快速定位问题，再决定怎么修。

首先我会自己先复现一下，看看是不是特定环境才出现。比如换个浏览器、开个无痕模式、清一下缓存，或者用手机访问试试。有时候是用户浏览器版本太低，不支持某些 ES6 语法，或者扩展插件冲突了。

如果复现不了，就让用户提供一些信息，比如浏览器版本、操作步骤、控制台有没有报错。如果能拿到报错截图最好，拿不到就看他能不能打开开发者工具，或者用我们埋点系统上报的错误日志。

一般来说白屏的原因大概分几类：

**一个是 JS 报错导致渲染中断**。比如某个组件里用了某个变量，但数据没拿到变成了 undefined，调用它的属性就报错了，React 或者 Vue 会直接把整块 UI 炸掉。这种看控制台报错就能定位，通常是兼容性问题或者后端返回的数据结构异常。解决办法就是代码里多判空，或者用错误边界组件兜一下。

**再就是路由或者入口文件的问题**。比如单页应用打包后的 JS 文件加载失败了，可能是 CDN 挂了、资源路径不对、或者用户网络环境屏蔽了某个域名。这个可以在 Network 面板看有没有 404，或者用 try-catch 配合动态 import 做降级。

**还有 CSS 阻塞的情况**，比如某个 CSS 文件加载失败导致样式没出来，但内容其实在，只是看不见。这种看元素结构能发现文字其实是存在的，只是颜色白色或者被挡住了。

**另外就是白屏骨架屏没切走**，比如异步请求一直 pending 或者卡住了，loading 状态没结束，用户看到的还是加载中的占位图。这种要检查接口是不是挂了，或者超时时间设置得太短。

**最极端的情况是内存爆了**，页面加载大量数据或者有内存泄漏，导致页面崩溃白屏。这种需要看 Performance 面板或者 Memory 面板去排查。

所以我的习惯是项目里加上错误边界、全局的 try-catch、还有资源加载失败的回调，配合 Sentry 这种错误监控工具，能提前发现大部分白屏问题。如果是正在排查线上问题，就先用抓包工具或者远程调试，实在不行就回滚版本，然后慢慢看日志。



## 请列举至少3种会阻塞浏览器渲染的资源，并说说如何进行针对性优化。

这个问题挺实际的，浏览器渲染被阻塞，通常是因为某些资源占用了主线程或者阻止了 DOM 树的构建。我遇到的比较常见的有这么几种：

**第一种是 CSS 文件**。CSS 被浏览器认为是渲染阻塞资源，因为浏览器需要构建 CSSOM（CSS 对象模型），而构建 DOM 树的时候如果遇到外部 CSS，会等 CSS 下载并解析完才继续渲染。如果 CSS 文件太大或者网络慢，白屏时间就会变长。针对 CSS 的优化，我一般会压缩 CSS 文件，把首屏关键的样式内联到 HTML 的 `<head>` 里，非关键的 CSS 用 `media` 属性区分，比如 `media="print"`，或者用 `preload` 提前加载，再或者通过动态加载的方式异步加载非首屏的 CSS。

**第二种是同步加载的 JavaScript**。默认情况下，浏览器遇到 `<script>` 标签会暂停 DOM 解析，先下载并执行 JS，执行完才继续往下解析。如果 JS 文件很大或者逻辑复杂，页面就会卡住。优化方案主要是给 `<script>` 加上 `defer` 或者 `async`。`defer` 是按顺序异步下载，等 DOM 解析完后执行；`async` 是下载完立即执行，适合独立不依赖 DOM 的脚本。另外就是把 JS 放在 `<body>` 底部，但更推荐用 `defer`。还有一个常见优化是代码分割，不要让一个 JS 文件太大。

**第三种是字体文件**。字体通常用 `@font-face` 引入，浏览器在字体下载完之前，会先用降级字体显示，或者干脆不显示文字（FOIT，Flash of Invisible Text）。这个虽然不是严格意义上的“阻塞渲染”，但是会阻塞文字的渲染，用户体验上也像卡住了。优化方式我常用 `font-display: swap`，让浏览器先用系统字体显示，等字体下载完再替换，就不会出现文字消失的尴尬。另外可以预加载关键字体，比如 `<link rel="preload" as="font">`，或者把字体转成 base64 内联，但只适合小文件。

除了这三种，**图片** 不会阻塞渲染，但会阻塞 `load` 事件，优化主要靠懒加载。**如果页面有大段的内联脚本或者样式**，虽然不是外部资源，但解析和执行的耗时同样会阻塞主线程，解决办法是把不影响首屏的逻辑拆出去或者异步化。

总结下来，我的优化思路就是：首屏关键的资源提前加载并压缩，非关键的异步化或延迟加载，能用 `defer` 和 `async` 就用，字体用 `swap` 兜底。然后配合 Chrome DevTools 的 Performance 面板看一下实际阻塞情况，再针对性地调整。

## 解释JavaScript中的事件委托event delegation的定义，以及其带来的优点。

事件委托，简单说就是**不直接把事件监听绑定到每个目标元素上，而是绑定到它们的父元素或者祖先元素上，利用事件冒泡的机制，让父元素统一处理子元素的事件**。

举个例子，如果一个列表里有 100 个按钮，常规做法是循环每个按钮加监听，费内存还麻烦。事件委托的做法是给父容器加监听，点击按钮时事件会冒泡到父容器，父容器通过 `event.target` 判断实际点的是哪个按钮，再执行对应的逻辑。

这样做带来的好处主要有三个：

**第一个是性能好**。减少事件监听器的数量，不需要为每个子元素都绑定函数，尤其是动态列表或者数据量很大的时候，内存占用和初始化时间都会明显降低。

**第二个是支持动态元素**。如果子元素是后来通过 JS 动态添加的，不用再重新绑监听，因为事件绑定在父元素上，新加的子元素只要在父容器内，点它自然就能触发。传统方式每新增一个元素就要重新绑一次，很容易漏。

**第三个是代码更简洁**。不用写循环绑定的逻辑，也避免了频繁解绑和重新绑定的麻烦，管理起来更容易，尤其是组件销毁时清理监听也只需要清理一个父容器的监听就行。

不过事件委托也有两个小坑要注意。一是事件必须支持冒泡，像 `focus`、`blur` 这类不冒泡的事件就不太合适。二是判断具体目标元素时，可能需要通过 `closest` 或者判断类名来确保点的是你真正想要处理的那个子元素，避免父元素自己的点击也触发逻辑。

总的来说，事件委托在需要监听大量子元素或者动态增删子元素的场景下，是非常推荐的做法。	

## 请解释Vue组件的生命周期的各个阶段及对应的应用场景

Vue 组件的生命周期，大概分成四个阶段：创建、挂载、更新、销毁。每个阶段 Vue 都会给我们抛一些钩子函数，方便在特定时机塞自己的逻辑。

**创建阶段**，主要调 `beforeCreate` 和 `created`。`beforeCreate` 的时候，组件实例刚初始化完，数据观测和事件配置都还没好，所以这里拿不到 data，也拿不到 methods，一般很少用，偶尔用来混入一些插件级别的初始化。到了 `created`，响应式数据、计算属性、方法都准备好了，但 DOM 还没生成，所以这里最适合做的是**异步请求**、初始化非响应式的变量、或者从 `localStorage` 读点东西。这个阶段不能操作 DOM，因为压根没有。

**挂载阶段**，有 `beforeMount` 和 `mounted`。`beforeMount` 是在模板编译完成、虚拟 DOM 生成但还没替换真实 DOM 的时候，平时用得不多。`mounted` 是组件第一次被挂载到真实 DOM 之后，这个时候可以**操作 DOM** 了，也适合初始化那些依赖 DOM 的第三方库，比如 ECharts、swiper，或者获取元素宽高、绑定事件监听等。

**更新阶段**，组件的数据变化时触发，有 `beforeUpdate` 和 `updated`。`beforeUpdate` 在数据改完但 DOM 还没重新渲染之前，可以用来记录旧的 DOM 状态，比如滚动位置。`updated` 是在 DOM 重新渲染完成后，如果有些逻辑依赖更新后的 DOM，可以放这里。不过要注意，`updated` 里尽量不要改数据，不然容易死循环。实际开发中这两个钩子用得不算多，因为很多需求用 computed 或 watch 就能解决。

**销毁阶段**，对应 `beforeDestroy` 和 `destroyed`。`beforeDestroy` 在实例被销毁之前调用，这时候实例还是完整的，**最重要的工作就是清理**：清定时器、解绑全局事件、取消订阅、销毁第三方实例，不然会导致内存泄漏。`destroyed` 是组件销毁完后调用，此时所有指令和事件监听都已被移除，一般很少需要手动写逻辑。

还有一个特殊情况，如果组件被 `<keep-alive>` 缓存了，会多出 `activated` 和 `deactivated`。`activated` 在组件每次激活（进入视图）时触发，`deactivated` 在组件停用（离开视图）时触发。缓存的组件不会真正走 `beforeDestroy`，所以类似定时器的清理可能就得放在 `deactivated` 里。

总结一下记忆方法：请求放在 `created`，操作 DOM 放 `mounted`，清理工作放 `beforeDestroy`，缓存组件用 `activated`/`deactivated`。



## 什么场景下需要封装组件？封装组件的判断标准是什么？

这个问题其实挺实在的，因为日常开发里面，组件封装很容易走两个极端：要么不敢抽，全写在一个文件里；要么过度封装，为了复用而复用反而更乱。

我自己判断要不要封装一个组件，一般会看几个信号。

**最直接的信号就是代码重复**。同一段模板、同一套逻辑，在项目里出现了两次以上，我就会考虑抽成组件。比如各种弹窗，确认框、表单弹窗，结构都差不多，只是标题和内容不一样，这种很适合封装。但也要注意别因为两次就急着抽，可能第三次的时候需求就跑偏了，等到第三次或者第四次再抽也不迟。

**第二个信号是业务逻辑的独立性**。有些模块虽然只在一个地方用，但它本身的逻辑特别复杂，比如富文本编辑器、日历选择器、代码编辑器这种，单独抽出来能大大降低主文件的复杂度，维护起来也清晰。这种按“高内聚低耦合”来拆，即使不复用，封装也有价值。

**第三个信号是变化的频率**。如果某块代码总是因为改需求而变来变去，把它抽出来，通过 props 或者插槽来控制变化，改的时候只动这个组件，不影响父组件。反过来，如果一块代码很稳定基本不改，放在哪里都行，没必要硬抽。

**第四个是跨项目复用的潜力**。如果某个功能不依赖具体业务，比如带校验的输入框、分页组件、懒加载图片，这些属于通用 UI 组件，甚至可以抽到组件库里，未来别的项目也能直接用。

**判断标准**，我觉得可以问自己几个问题：

- 这个组件是否有一个清晰的输入输出，也就是 props 和事件边界是否明确？
- 它是否只做一件事，并且这件事跟父组件没关系？
- 如果以后需求变了，修改这个组件会不会影响到父组件或者其他地方？（影响越小说明封装的越好）
- 复用它的成本，是否比重新写一份还低？

**不建议封装的场景**：如果一段代码只有两个地方用，但两个地方的样式和行为完全不一样，强行抽一个组件会导致 props 满天飞、if 条件一堆，反而更难维护。这时候不如保留两份，各写各的，等相似度稳定了再重构。

总的来说，我的原则是“先写后抽，不提前设计”。先写在一个地方，当出现第二个类似场景，再看看能不能抽象；等到第三个出现时再动手。过早抽象比不抽象更容易出问题。



## 前端错误监控系统

> 你正在构建一个前端错误监控系统，请说明你会如何处理异常捕获？如何设计日志上报方案？请描述主要的实现思路。

这个问题我正好在项目里做过类似的。前端错误监控，核心就是两步：先把异常抓到，再把错误信息上报到服务端。我分开说。

**异常捕获这部分，主要覆盖三种场景。**

第一种是**运行时错误**，比如变量未定义、调用不存在的方法。在 Vue 里可以用 `Vue.config.errorHandler` 全局捕获组件内的错误；在 React 里用 `ErrorBoundary` 组件包裹业务模块。另外还加一个 `window.onerror`，专门捕获没有被框架兜住的 JS 错误，能拿到错误信息和行列号。不过 `onerror` 拿不到资源加载失败的错误，那种得用 `window.addEventListener('error')` 单独监听，而且判断 `target` 是 `script` 或 `link` 之类的元素才上报。

第二种是**Promise 的未捕获 reject**，会走 `window.addEventListener('unhandledrejection')`，能捕获到 `fetch` 接口异常或者 `Promise.reject` 没补 `.catch` 的情况。这个很容易漏，一定要加上。

第三种是**资源加载错误**，上面提到了，图片、脚本、样式文件 404 或者加载超时，走 `error` 事件兜住。

另外还有一些**接口异常**，一般在请求库的拦截器里统一处理，比如 axios 的响应拦截器里判断状态码非 2xx 就上报。这种不属于 JS 运行时错误，但是也纳入监控范围。

**日志上报方案的设计，核心原则是：不阻塞主线程，不频繁发请求，尽量不丢数据。**

我一般这样设计。每次捕获到错误，先封装成一个标准格式，包含错误信息、文件名、行列号、用户 ID、页面 URL、浏览器信息、操作步骤（可以配合录屏或者用户行为栈）。然后把这条日志放进一个队列里，不是马上发。

队列会用 `requestIdleCallback` 或者 `setTimeout` 延迟发送，避免和关键渲染任务抢资源。也可以利用 `navigator.sendBeacon`，尤其适合在页面 unload 或者跳转前发日志，因为常规的 `fetch` 可能会被浏览器取消。`sendBeacon` 的好处是异步、不阻塞页面关闭、浏览器保证会发出去。

如果日志量比较大，我还会做合并上报。比如队列积攒了 10 条日志或者过了 5 秒钟，就打包成一个请求一次性发给后端，减少请求次数。后端收到后按时间落库或者推入消息队列。

**还有个坑是防止上报死循环**。比如上报逻辑本身出了错，又触发新的上报，那就会递归爆炸。所以要在上报的 `catch` 里做标记，避免重复上报自己产生的错误。

**整体实现思路**大概是这样：
- 在入口文件初始化监控 SDK，挂载全局的各种错误监听
- 捕获后过滤重复错误（相同错误和行列号短时间内不重复报）
- 放入队列，定时或积攒批量发送
- 发送失败时存入 `localStorage`，下次页面启动时重新尝试上报
- 生产环境可以用 `Sentry` 这类成熟方案，自己做的好处是可以定制业务上下文，比如带上用户操作埋点串，方便复盘

当然，我们项目规模不是特别大的话，其实直接上 Sentry 更省事，自己造轮子主要是为了数据走内网合规，或者需要深度定制用户行为轨迹。



## 在技术方案选型的评分矩阵建立过程中，你是如何确定各个评估维度的权重分配的？

这个问题挺考验实际经验的。评分矩阵的权重分配，说到底是**把团队和项目的偏好量化**的过程。我一般会分几步走。

**第一步，先列出所有可能相关的评估维度**。比如性能、开发成本、学习曲线、社区活跃度、文档质量、可扩展性、稳定性、安全性、运维成本、与现有技术栈的契合度等。这个阶段不设限，先把能想到的都列出来，然后和团队一起过一遍，合并同类项，去掉明显不相关的。

**第二步，确定核心的“必保项”和“加分项”**。有些维度是硬性门槛，比如对安全性要求极高的金融项目，安全维度权重就要很高，甚至设置为一票否决。再比如做内部管理系统，开发速度比性能更重要，权重就会偏向开发成本和学习曲线。这一步需要明确项目的核心诉求是什么，是快速上线？是长期维护？还是高并发场景？核心诉求决定了权重的基调。

**第三步，用层次分析法（AHP）或者更简单的两两比较法来定权重的数值**。我会列一个表格，把所有维度两两比较，问自己“A 比 B 重要多少”，比如“性能比开发成本重要一点点”，或者“学习曲线比社区活跃度重要很多”。通过这种两两对比，可以算出一个相对权重值，避免拍脑袋。如果团队人多，会让前端、后端、运维、产品一起打分，取平均值，减少个人偏见。

**第四步，根据项目阶段和风险偏好微调**。比如项目探索期，技术选型可能有很大变动，那么灵活性和生态的权重会提高；项目已经进入稳定期，迁移成本很高，那么与现有系统的兼容性权重就要拉高。有时候还会给新技术的“不确定性”加一个惩罚系数，因为新东西虽然看起来很美好，但藏着很多坑。

**第五步，权重不是一次定死的**。在评分过程中，如果发现某个方案在某项上得分特别极端（比如性能特别好但文档一塌糊涂），我会反问自己：这个维度的权重真的合理吗？有没有可能我低估了文档的重要性？然后适当调整。选型结束后，还会复盘权重的合理性，为下次决策积累经验。

**一个具体例子**，我之前做前端监控 SDK 的选型，评估了 Sentry、自己封装、还有商业产品。项目要求是数据必须内网存储，合规要求很高，所以“数据可控性”和“自研成本”权重给到 40%，而“功能丰富度”只给 15%。最后权重分配表经过评审后，大家一起打分，算出来自己封装虽然开发成本高，但数据可控这项满分，综合得分最高。后来确实也选了自己做，和预期一致。

总结一下，权重分配不是纯数学问题，核心是**团队对业务优先级达成共识**。用矩阵是为了让决策有据可查，避免“我觉得这个好”这种主观争论。





## 外层大盒子包含小盒子，小盒子设置 margin-top:10px 的呈现效果

父元素和第一个/最后一个子元素之间，垂直方向上的 margin 会发生合并，小盒子带着大盒子一起往下移动了 10px。

这个问题涉及到 CSS 的一个经典现象——**外边距塌陷**，也叫外边距合并。

如果你在一个大盒子里面放一个小盒子，然后给小盒子设置 `margin-top: 10px`，你可能会发现：小盒子没有往下移动 10px，反而带着大盒子一起往下移动了 10px。大小盒子之间并没有出现预期的 10px 间距。

这是因为**父元素和第一个/最后一个子元素之间，垂直方向上的 margin 会发生合并**。浏览器会取父元素和子元素之间较大的那个 margin 值，应用到父元素的外侧。你给子元素设了 `margin-top`，它想把自己顶下去，但父元素没有内边距或边框隔开，这个 margin 就直接“穿透”了父元素，作用到了父元素的外面。

要实现大小盒子之间隔开 10px 的效果，有几种常用方法：
- 给父元素设置 `overflow: hidden` 或 `overflow: auto`
- 给父元素设置 `border-top: 1px solid transparent`
- 给父元素设置 `padding-top: 10px`，子元素就不用设 margin 了
- 用 Flex 或 Grid 布局，这两种布局不会产生 margin 塌陷问题



## 数组常见方法，哪些改变原数组哪些不改变

**数组的常见方法**我平时用得比较多的有这些：`shift`、`unshift`、`push`、`pop`、 用来在头尾增删元素；`map`、`filter`、`reduce`、`forEach` 用来遍历和处理数据；`find`、`findIndex`、`indexOf`、`includes` 用来查找；`slice` 用来截取；`splice` 用来增删改；`concat` 合并数组；`join` 转字符串；`sort` 排序；`reverse` 反转；还有 `some`、`every` 用来做条件判断。

**slice 和 splice 的区别**，很多人会搞混。slice 是截取，不改变原数组，返回一个新数组。用法是 `arr.slice(start, end)`，包含 start 不包含 end，不传参数就复制整个数组。splice 是增删改，会改变原数组，返回被删掉的元素组成的新数组。用法是 `arr.splice(start, deleteCount, ...items)`，从 start 位置删 deleteCount 个，然后再把 items 插进去。记忆方法：splice 多一个 p，代表 "pull out" 或者 "put in"，会动原数组。

**哪些会改变原数组，哪些不会**，这也是面试常问的。改变原数组的：`push`、`pop`、`shift`、`unshift`、`splice`、`sort`、`reverse`、`fill`、`copyWithin`。不改变原数组的：`slice`、`concat`、`map`、`filter`、`reduce`、`forEach`（它不返回新数组但也不改变原数组，除非回调里主动改）、`find`、`findIndex`、`indexOf`、`includes`、`join`、`some`、`every`、`flat`、`flatMap`。还有一个 `toSorted`、`toReversed` 这些 ES2023 新加的方法，它们不会改变原数组，相当于返回排序或反转后的新数组，但目前兼容性还不够广。

总结一句：记不准的时候，可以默认字符串方法都不改原字符串（字符串是不可变的），数组方法就要小心了，用之前最好确认一下文档。

## Vue组件的通信方式有哪些

Vue 组件通信方式还蛮多的，不同关系用不同方法。

**父子组件**是最常见的场景。父传子用 `props`，子传父用 `$emit`，这是最基础也最推荐的方式。父组件也可以通过 `ref` 直接调用子组件的方法，但一般用在确实需要主动触发子组件行为的场景，比如表单校验。

**爷孙组件**或者深层嵌套的组件，可以用 `provide` 和 `inject`。祖先组件通过 `provide` 提供数据，后代组件通过 `inject` 注入。这样不用一层层传 props。但需要注意的是，`provide` 默认不是响应式的，如果想要响应式，需要传 `ref` 或 `reactive` 对象。

**兄弟组件**没法直接通信，通常通过共同的父组件来中转，也就是子传父、父再传给另一个子。或者用**事件总线**，在 Vue 2 里可以用 `new Vue()` 作为事件中心，`$on` 监听、`$emit` 触发。Vue 3 里推荐用 `mitt` 库来代替。不过事件总线会让数据流向变乱，项目大了不太好维护。

**全局状态管理**是解决跨层级复杂通信的最终方案。Vue 2 用 Vuex，Vue 3 用 Pinia。适合多个组件共享同一份数据、或者状态改动频繁的场景，比如用户信息、购物车、主题配置。

另外还有一些场景化的方式：`$parent` 和 `$children` 可以访问父组件和子组件实例，但耦合太强，基本不推荐。`$attrs` 可以透传父组件传递但子组件没有声明的 props，配合 `inheritAttrs` 禁掉自动继承，适合做高阶组件。插槽也是一种通信方式，父组件通过插槽把内容传递给子组件，本质是 HTML 结构的传递。

总结一下我的习惯：父子用 props/emit，爷孙用 provide/inject，全局用 Pinia，兄弟或者简单场景可以用事件总线但尽量少用。组件通信太多太乱的时候，就想想是不是该抽一个 Pinia 模块了。



## Vue Router的常见模式及区别， History模式刷新时需要配置什么

Vue Router 有两种常见模式：**hash 模式**和**history 模式**。

**hash 模式**是默认的，URL 里带一个 `#`，比如 `http://example.com/#/user`。`#` 后面的部分叫 hash，它的特点是：hash 变化不会触发浏览器向服务器发请求，页面不会刷新，但会触发 `hashchange` 事件，Vue Router 监听到这个事件就切换组件。所以 hash 模式不需要服务器做任何配置，直接就能跑，兼容性也好，能到 IE 9。缺点就是 URL 里带个 `#`，不好看，而且 hash 部分的参数不会被发送到服务端，SEO 略差一点。

**history 模式**基于 HTML5 的 History API，URL 里没有 `#`，看起来就是正常的路径，比如 `http://example.com/user`。它通过 `pushState` 和 `replaceState` 来切换路由，不会刷新页面。但问题在于：当用户直接访问 `http://example.com/user` 或者刷新页面时，浏览器会真的向服务器发请求，请求这个路径。如果服务器没有针对这个路径做配置，就会返回 404。

**所以 history 模式需要服务器做兜底配置**。核心思路就是：所有路由地址都返回同一个 HTML 页面（通常是 `index.html`），由前端路由自己去匹配组件。具体配置方式根据服务器不同：

- **Nginx** 里用 `try_files $uri $uri/ /index.html;`，意思是先找对应文件，找不到就返回 index.html。
- **Apache** 用 `mod_rewrite`，写 `.htaccess` 把请求重写到 index.html。
- **Node.js（Express）** 可以用 `connect-history-api-fallback` 中间件。

另外，history 模式部署到非根路径时，还需要在 Vue Router 里配置 `base` 选项，比如 `base: '/app/'`，服务器也要同步调整。

总结一下：开发环境用 hash 省事，生产环境用 history 更好看，但别忘了配服务器。如果配不了服务器就老老实实用 hash。



## 请解释一下 HTTP 缓存 有哪些类型，它们分别是如何工作的？

HTTP 缓存我理解主要分为两类：**强缓存**和**协商缓存**。它们都是用来减少重复请求、加快页面加载的，但工作方式不太一样。

**强缓存**是最直接的，浏览器向服务器请求资源时，服务器会返回一个过期时间，比如 `Cache-Control: max-age=3600`，告诉浏览器这个资源在未来一小时内可以直接用本地缓存，压根不会发请求到服务器。还有旧一点的 `Expires`，指定一个绝对时间点。强缓存命中时，网络面板会看到 `200 (from disk cache)` 或 `memory cache`。缺点是过期之前服务器更新了资源，浏览器也不知道，只能等时间到了再去拿新版本。

**协商缓存**则是“每次都去问问服务器”，但服务器不一定返回完整资源。浏览器发请求时会带上一些标识，比如 `If-Modified-Since` 带上上次修改时间，或者 `If-None-Match` 带上上次返回的 ETag（资源唯一标识）。服务器收到后比对一下：如果资源没变，就返回 `304 Not Modified`，告诉浏览器继续用缓存，这就省了传输完整资源的时间；如果变了，就正常返回 200 和新资源。协商缓存能保证客户端和服务器资源始终一致，但比强缓存多了一次请求开销。

实际项目中通常两者配合使用：强缓存兜底命中率，协商缓存保证一致性。比如用 `Cache-Control: max-age=86400, must-revalidate`，意思是一天内走强缓存，一天后资源过期了再走协商缓存去问服务器。而 `ETag` 比 `Last-Modified` 更精确，适合文件内容频繁变化但修改时间没变的情况，比如周期性重新生成但内容相同的配置。



## 请解释一下 DOM 中的事件冒泡和事件捕获是什么，以及事件委托(事件代理) 的原理和优势？

事件冒泡和事件捕获是 DOM 事件传播的两个阶段。**事件捕获**是从 window 往下一直走到目标元素，**事件冒泡**是从目标元素往上一直回到 window。先捕获后冒泡，中间还有一个“目标阶段”就在目标元素上。

**事件委托**就是利用了事件冒泡的机制。你不需要给每个子元素单独绑定事件，而是把监听器挂到它们的父元素或者祖先元素上，通过判断 `event.target` 来决定怎么处理。比如一个动态增删的列表，给父级 `ul` 绑定点击事件，不管后面加多少 `li`，点上去都能触发，不用再重新绑。

它的**优势**很明显：一是**性能好**，减少监听器的数量，尤其在子元素非常多的时候内存占用和初始化时间都降下来了；二是**支持动态元素**，新增的子元素无需手动绑事件；三是**代码简洁**，清理事件也只需清理父级的一个监听，不用遍历销毁。

不过也有注意事项，比如 `focus`、`blur` 事件不冒泡，得用捕获阶段或者特殊处理。另外 `event.target` 可能不是你要的子元素，比如点到了 `li` 里的 `span`，这时可能需要用 `closest` 向上找一下。还有就是委托层级太深或者页面事件过多时，可能会有轻微的性能开销，但实际应用中基本可以忽略。



##  请解释一下 虚拟DOM(Virtual DOM) 是什么，以及它的工作原理和主要优势？

虚拟 DOM 简单说就是用 **JavaScript 对象来模拟真实 DOM 树**的一个轻量表示。比如真实 DOM 节点有几十上百个属性，而虚拟 DOM 只保留几个关键属性，比如 `tag`、`props`、`children`。

它的工作流程大概是这样的：每次状态变化时，框架会重新生成一棵新的虚拟 DOM 树，然后用 **diff 算法** 比较新旧两棵树，找出哪里变了，最后只把有差异的部分（patch）更新到真实 DOM 上。

主要优势有几个：首先是 **避免频繁操作真实 DOM**，因为真实 DOM 操作开销很大，而虚拟 DOM 在内存中计算差异，再批量更新，性能更好；其次是 **跨平台**，因为虚拟 DOM 是对真实 DOM 的抽象，可以根据不同平台渲染成不同的东西，比如 RN 里渲染成原生组件；还有 **开发体验** 也更好，开发者只需要声明式地写 UI，不用手动操作 DOM 和操心性能细节。

当然虚拟 DOM 也不是银弹，首次渲染时它比直接操作真实 DOM 要多一层计算，而且 diff 算法也有开销。但对于频繁更新的交互页面，它的优势还是很明显的。



## 请简单说一下 Vue 中的 diff 算法 是如何工作的，它有哪些核心优化策略？

Vue 中的 diff 算法主要用于更新阶段，对比新旧虚拟 DOM 树，找出最小化的变更并应用到真实 DOM 上。它的核心思路是**同层比较**，不跨层级对比，这就把复杂度从 O(n³) 降到了 O(n)。

具体工作流程是这样：当组件重新渲染时，会生成新的虚拟 DOM 树，然后和旧树一起传给 `patch` 函数。先比较两个根节点，如果节点类型不一样（比如 `div` 变成了 `p`），就直接替换整个子树，不再往下对比。如果节点类型相同，就继续比较它们的属性和子节点。

对于子节点的比较，Vue 做了一些关键优化。它假设列表中的节点有唯一的 `key`，通过双端比较（头头、尾尾、头尾、尾头）来尽可能复用相同 `key` 的节点，减少移动和创建操作。这就是为什么要给 `v-for` 加 `key`，并且要稳定唯一、不要用 `index`，否则 diff 算法可能会误判导致性能下降甚至状态错乱。

```vue
<!-- 错误：用 index，因为它会随列表变化而改变，失去 key 的意义 -->
<div v-for="(item, index) in list" :key="index"></div>
<!-- 正确：用唯一 id，不能随机数，每次渲染都变，完全失效，也不能不写key导致就地复用，容易出 BUG -->
<div v-for="item in list" :key="item.id"></div>
```

Vue 2 用的是双端 diff，Vue 3 在此基础上做了优化，加上了静态标记和区块提升，编译时就把静态节点和动态节点区分开，更新时只追踪动态节点。这也是 Vue 3 性能更好的原因之一。

总的来说，Vue 的 diff 策略就是：同层对比 + key 优化 + 静态标记，在保证算法效率的同时兼顾了开发体验。



## 假设现在让你设计一个前端日志上报 SDK，用于收集 React 项目中的错误、性能数据和用户行为（点击、路由变化等），你会如何设计它的模块划分和上报策略？不要求完整代码，讲清楚思路即可。

我会把它设计成一个**可配置、非侵入、支持批量上报**的 SDK，核心是跟业务代码解耦，不能影响主流程的性能和稳定性。

**模块划分上**，我大概会分成这么几块：

1. **初始化模块**：负责接收配置，比如上报地址、采样率、是否开启性能监控、用户信息等。同时做环境检测，判断是浏览器还是 Node，有没有 `window` 对象。

2. **错误捕获模块**：封装 `window.onerror` 和 `unhandledrejection`，还会劫持 `console.error` 可选。如果是 React 项目，会提示用户用 `ErrorBoundary` 包裹根组件，SDK 提供一个高阶组件或 Hook 来上报组件树内的错误。对于异步错误和资源加载错误也会单独处理。

3. **性能采集模块**：利用 Performance API 拿 `navigation` 和 `paint` 数据，比如 DNS、TCP、首屏时间、DOM 解析耗时等。还可以通过 `PerformanceObserver` 监听 `largest-contentful-paint`、`first-input` 这些核心指标。页面关闭前可以用 `sendBeacon` 发送最后一批性能数据。

4. **用户行为模块**：采用插件化设计，可以按需开启。比如点击行为通过事件委托监听 `document` 上的点击，记录目标元素的 `data-*` 属性或路径。路由变化则劫持 `pushState`、`popstate` 事件，或者监听 React Router 的 history 实例。页面离开或者组件卸载时会 flush 队列。

5. **存储与队列模块**：所有日志先进入一个内存队列，设置最大长度（比如 30 条），超过就触发上报。同时支持降级存储，如果队列满了或者上报失败，可以存到 `localStorage` 或 IndexedDB，下次页面启动时重新尝试上报。

6. **上报模块**：支持批量上报，默认会合并多条日志为一个请求，减少连接数。上报时机可以用 `requestIdleCallback` 或者 `setTimeout` 延迟，页面 `beforeunload` 时改用 `navigator.sendBeacon` 确保数据发出。还支持重试机制，比如失败后指数退避重试 3 次。

**上报策略**上，我会做成可配置的：
- **采样上报**：高流量页面只上报一定比例，比如 10%，避免服务端压力过大。
- **隐私脱敏**：对 URL 参数、用户输入的文本做脱敏或者哈希处理，不能直接上报敏感信息。
- **数据格式**：每条日志会带上公共字段，比如应用 ID、版本号、用户标识、时间戳、会话 ID，方便后端关联查询。

最后还要考虑**性能开销**：日志的序列化尽量用 `JSON.stringify` 的浅拷贝，避免深拷贝大对象；采集用户行为时要防抖节流，不能拖慢页面交互。如果是大型项目，SDK 本身也要支持按需引入，比如只引入错误监控，不引入性能模块。

整体思路就是：非侵入、可插拔、优先保证主流程稳定，日志上报在后台悄悄完成，用户无感知。
