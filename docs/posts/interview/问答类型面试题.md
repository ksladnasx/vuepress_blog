## 动态路由及鉴权

### Vue-router

我们项目用的是 Vue-Router。动态路由的核心思路是：**前端定义好所有需要权限的路由配置，但不在启动时全部挂载，而是登录后根据后端返回的权限列表动态添加**。

具体步骤：

1. 路由分两块：**静态路由**（登录页、404 等无需权限的）和**动态路由**（需要权限的页面，在 `meta` 里加 `roles` 或 `permission` 标记）。
2. 登录成功后，后端返回当前用户的权限码列表（比如 `['user:add', 'order:list']`）。
3. 在路由守卫 `beforeEach` 里做判断：如果没添加过动态路由，就调用一个 `generateRoutes` 方法，把动态路由数组过滤出用户有权限的那些，然后 `router.addRoute` 逐一添加（Vue 3 的 `addRoute` 支持嵌套）。
4. 同时把过滤后的路由存到 Vuex/Pinia 里，用来生成侧边栏菜单。菜单组件递归渲染这个树形结构，根据 `meta.hidden` 字段决定是否显示。

注意两个坑：

- 动态添加完后要调用 `next({ ...to, replace: true })`，确保路由完整后再进入页面，否则可能匹配不到。
- 刷新页面时动态路由会丢失，所以在 `App.vue` 的 `created` 或 `setup` 里要重新拉取权限、重新添加路由，一般配合 `localStorage` 存一下权限码，避免每次都白屏。

鉴权菜单其实和动态路由用的是同一份数据，只是展示上再处理一下标题、图标、排序这些。

### React-router

React Router 里动态路由的核心思路也是一样：**登录后根据权限动态添加路由**。不过 React Router 的 API 跟 Vue Router 不太一样。

我们项目用的 v6，做法是：

1. 定义一份全量的路由配置（比如 `allRoutes` 数组），每个路由加一个 `auth` 或 `roles` 字段。
2. 登录后拿权限列表，过滤出有权限的路由。
3. 用 `createHashRouter` 或 `createBrowserRouter` 创建路由实例，把过滤好的路由传进去。不用像 Vue 那样手动 `addRoute`，而是直接生成一个新 router 实例。
4. 但是刷新会丢失，所以需要在根组件里用 `useEffect` 监听 token 变化，重新拉权限、重新生成路由，再用 `useNavigate` 跳转。

**鉴权菜单**：把过滤后的路由树存到 context 或 Redux 里，递归渲染菜单组件，根据 `hidden` 字段隐藏菜单项。

跟 Vue 的主要区别：

- Vue Router 支持动态 `addRoute`，可以在运行时不断追加。React Router v6 没有直接的 `addRoute`，要么提前用生成的路由实例整个替换，要么用 `<Routes>` 结合条件渲染（但这样会在组件里写很多权限判断，比较乱）。
- 所以我们通常用第二种：定义路由表后，用 `useRoutes` 或者直接传入 `router` 对象。菜单生成逻辑和 Vue 几乎一样。

简单说：都是“路由表和权限比对”+“菜单递归渲染”，只是框架 API 不同。



## 页面请求大规模并发的问题

### 背景

大项目涉及，比如数据采集平台或者低代码编辑平台，请求的内容得有序且相对稳定的发送到后端

### 方案

#### 请求队列封装

用常规的方案，封装请求队列，大批流量来之后进入队列 ，通过窗口大小进行限制，只对窗口大小内的几个请求进行处理请求完成后出队。

```js
q = [];q.push("入队");
q.shift() //出队
```

```js
class requestQueue {
  constructor(maxCount) {
    this.maxCount = maxCount;  // 最大并发数
    this.currentCount = 0;      // 当前并发请求数
    this.queue = [];            // 请求队列
  }
  
  add(request) {
    return new Promise((resolve, reject) => {
      this.queue.push({ request, resolve, reject });
      this.processQueue();
    });
  }
  
  processQueue() {
    // 情况1：当前并发数已达上限，不处理
    if (this.currentCount >= this.maxCount) {
      return;
    }
    
    // 情况2：队列为空，不处理
    if (this.queue.length === 0) {
      return;
    }
    
    // 从队列头部取出一个请求
    const { request, resolve, reject } = this.queue.shift();
    
    // 并发数+1
    this.currentCount++;
    
    // 执行请求
    Promise.resolve(request())
      .then((result) => {
        // 请求成功
        resolve(result);
      })
      .catch((error) => {
        // 请求失败
        reject(error);
      })
      .finally(() => {
        // 无论成功失败，当前请求结束，并发数-1
        this.currentCount--;
        
        // 继续处理队列中的下一个请求
        this.processQueue();
      });
  }
  
  // 可选：获取当前队列状态
  getStatus() {
    return {
      currentCount: this.currentCount,
      queueLength: this.queue.length,
      maxCount: this.maxCount
    };
  }
  
  // 可选：清空队列
  clear() {
    this.queue = [];
  }
}

// 使用示例
const queue = new requestQueue(3); // 同时最多3个请求

// 添加请求
for (let i = 0; i < 10; i++) {
  queue.add(() => {
    return fetch(`/api/data/${i}`).then(res => res.json());
  }).then(data => {
    console.log('请求成功:', data);
  }).catch(err => {
    console.log('请求失败:', err);
  });
}

// 文件上传切片示例
const uploadQueue = new requestQueue(5); // 5个并发

chunks.forEach(chunk => {
  uploadQueue.add(() => {
    return uploadChunk(chunk); // 上传单个切片
  }).then(result => {
    console.log(`切片 ${chunk.index} 上传成功`);
  });
});
```

#### 防抖/节流方式进行处理

**防抖原理**（debounce）：事件触发后延迟执行，若在延迟时间内再次触发则重新计时。适用于输入框搜索、窗口resize等场景（确保在指定时间内函数只执行一次，用户多次快速点击后 ，延迟执行，期间重新触发则重置延迟，相当于限制请求发送的频次）。

**节流原理** （throttle）：规定时间内只执行一次事件处理函数（时间戳（规定时间戳内只能执行一次），定时器（延迟执行））。适用于滚动加载、按钮防重复点击、游戏动画等场景。（多次请求会处理，但是会限制请求发送的频率）

以下是纯 JavaScript 的防抖和节流函数代码示例：

##### 防抖（Debounce）

```javascript
// 防抖：延迟执行，期间重新触发则重置延迟
function debounce(fn, delay) {
    let timer = null;
    return function(...args) {
        if (timer) clearTimeout(timer);
        timer = setTimeout(() => {
            fn.apply(this, args);
        }, delay);
    };
}

// 使用示例
const handleInput = debounce((value) => {
    console.log('搜索:', value);
}, 500);

document.getElementById('search').addEventListener('input', (e) => {
    handleInput(e.target.value);
});
```

##### 节流（Throttle）- 时间戳版

```javascript
// 节流：立即执行，规定时间内只执行一次
function throttle(fn, delay) {
    let lastTime = 0;
    return function(...args) {
        const now = Date.now();
        if (now - lastTime >= delay) {
            lastTime = now;
            fn.apply(this, args);
        }
    };
}

// 使用示例
const handleScroll = throttle(() => {
    console.log('滚动位置:', window.scrollY);
}, 200);

window.addEventListener('scroll', handleScroll);
```

##### 节流（Throttle）- 定时器版

```javascript
// 节流：延迟执行，停止触发后还会执行一次
function throttleTimer(fn, delay) {
    let timer = null;
    return function(...args) {
        if (timer) return;
        timer = setTimeout(() => {
            fn.apply(this, args);
            timer = null;
        }, delay);
    };
}

// 使用示例
const handleResize = throttleTimer(() => {
    console.log('窗口大小:', window.innerWidth);
}, 300);

window.addEventListener('resize', handleResize);
```

##### 完整节流（首尾都执行）

```javascript
// 节流：第一次立即执行，最后一次也会执行
function throttleFull(fn, delay) {
    let lastTime = 0;
    let timer = null;
    
    return function(...args) {
        const now = Date.now();
        const remaining = delay - (now - lastTime);
        
        if (remaining <= 0) {
            if (timer) {
                clearTimeout(timer);
                timer = null;
            }
            lastTime = now;
            fn.apply(this, args);
        } else if (!timer) {
            timer = setTimeout(() => {
                lastTime = Date.now();
                timer = null;
                fn.apply(this, args);
            }, remaining);
        }
    };
}
```

##### 实际应用场景示例

```javascript
// 1. 输入框搜索防抖
const searchInput = document.querySelector('#search');
const searchDebounce = debounce(async (keyword) => {
    const res = await fetch(`/api/search?q=${keyword}`);
    const data = await res.json();
    console.log('搜索结果:', data);
}, 500);
searchInput.addEventListener('input', (e) => searchDebounce(e.target.value));

// 2. 按钮点击节流（防止重复提交）
const submitBtn = document.querySelector('#submit');
const throttleSubmit = throttle(() => {
    console.log('提交表单');
    // 发送请求...
}, 1000);
submitBtn.addEventListener('click', throttleSubmit);

// 3. 滚动加载节流
let page = 1;
const loadMore = throttle(async () => {
    const res = await fetch(`/api/list?page=${page}`);
    const data = await res.json();
    page++;
    console.log('加载更多数据:', data);
}, 1000);
window.addEventListener('scroll', () => {
    if (window.innerHeight + window.scrollY >= document.body.offsetHeight - 100) {
        loadMore();
    }
});
```

**区别总结：**
- **防抖**：连续触发只执行最后一次（搜索输入、窗口resize）
- **节流**：间隔时间内只执行一次（滚动加载、按钮防连点）

## Vue3中常见工具库以及外部防抖和节流函数

有的，在 Vue3 + TypeScript 项目里，防抖和节流通常不需要自己从头写。最主流的方式是直接使用工具库，主要有两个不错的选择：

### 🛠️ 推荐方案一：VueUse
**VueUse** 是一个专门为 Vue3 设计的 Composition API 工具集，和 TypeScript 配合起来非常顺畅，是目前最推荐的方式。

*   ​	·······**安装**：
    ```bash
    npm install @vueuse/core
    ```

*   **防抖 (Debounce)**：使用 `useDebounceFn`。适合处理输入框搜索、表单验证，等用户停止操作后再执行。
    ```vue
    <script setup lang="ts">
    import { ref } from 'vue'
    import { useDebounceFn } from '@vueuse/core'
    
    const searchQuery = ref('')
    
    // 声明一个防抖函数：延迟 500ms 执行，意思是我如果连续输入了两个1那只会在500毫秒内打印出一个11
    const debouncedSearch = useDebounceFn(() => {
      console.log('发起搜索请求:', searchQuery.value)
    }, 500)
    </script>
    
    <template>
      <input v-model="searchQuery" @input="debouncedSearch" />
    </template>
    ```

*   **节流 (Throttle)**：使用 `useThrottleFn`。适合处理滚动加载、窗口大小调整、按钮连点等需要持续响应但需限制频率的场景。
    ```vue
    <script setup lang="ts">
    import { useThrottleFn } from '@vueuse/core'
    
    // 声明一个节流函数：每 1000ms 最多执行一次
    const throttledScroll = useThrottleFn(() => {
      console.log('处理滚动事件:', window.scrollY)
    }, 1000)
    </script>
    
    <template>
      <div @scroll="throttledScroll">滚动内容</div>
    </template>
    ```

### 📚 推荐方案二：Lodash
**Lodash** 也是一个非常经典的工具库。如果你的项目已经在用 Lodash，或者不想引入新的依赖，用它就很方便。为了更好的 TypeScript 支持和 Tree-Shaking（按需加载），建议安装 `lodash-es`。

*   **安装**：
    ```bash
    npm install lodash-es
    npm install -D @types/lodash-es  # TypeScript 项目需要安装类型声明
    ```

*   **防抖与节流**：
    ```vue
    <script setup lang="ts">
    import { ref, onUnmounted } from 'vue'
    import debounce from 'lodash-es/debounce'
    import throttle from 'lodash-es/throttle'
    
    const searchQuery = ref('')
    
    const fetchData = () => {
      console.log('请求数据:', searchQuery.value)
    }
    
    // 使用 Lodash 防抖，500ms 延迟
    const debouncedFetch = debounce(fetchData, 500)
    
    // 使用 Lodash 节流，1000ms 间隔
    const throttledClick = throttle(() => {
      console.log('按钮被点击（节流）')
    }, 1000)
    
    // 在组件卸载时取消，避免内存泄露
    onUnmounted(() => {
      debouncedFetch.cancel()
      throttledClick.cancel()
    })
    </script>
    ```

### 💡 总结对比
| 特性                     | VueUse                                             | Lodash                                                       |
| :----------------------- | :------------------------------------------------- | :----------------------------------------------------------- |
| **Vue 集成度**           | **高**，专为 Vue3 Composition API 设计，使用更自然 | **一般**，通用 JS 库，需注意手动管理 `cancel` 等生命周期     |
| **体积（Tree-Shaking）** | **极小**，按需引入，只打包你用到的函数             | **较小**，`lodash-es` 支持按需引入，但整体库体积比 VueUse 大 |
| **API 风格**             | **Vue 风格**，响应式友好                           | **通用 JS 风格**，对 Vue 无特殊适配                          |

**简单来说：**
*   如果追求**与 Vue3 的最佳集成、极致的轻量级和开发体验**，首选 **VueUse**。
*   如果项目已经**重度使用 Lodash**，或者你需要一个**一劳永逸的通用工具库**，**Lodash** 是稳妥的选择。

## 大文件上传

### 背景

我做项目的时候，做AI工作流的时候，涉及到用户自定义模型的上传，或者遇到大规模的知识库文件的上传，可能比较大上传时间长，有时候因为网络断开或者网络波动导致文件的上传失败或者用户不小心关闭了网页导致上传终止（断点续传），这类问题。

### **专业术语**

- 断点续传
- 断开重连重传
- 切片上传

### 方案

1. 前端将文件进切片，切分为一个个chunk
2. 切片要进行取名，利用hash值进行取名，index进行顺序标记，然后将切片传递给后端
3. 后端对切片进行组合

### 加料

前端切片的话，主进程做很卡顿，利用web-worker多线程切片，处理完后发送给主进程。切完后将blob存储到indexedDB，下次用户来后嗅探未完成的切片，有就尝试继续上传。利用websocket进行实时通知，以及请求序列的控制wss

###  断点续传核心：

这三个概念是断点续传的核心，我给你逐一解释：

---

#### 1. IndexedDB - 浏览器本地数据库

**是什么**：
浏览器自带的 NoSQL 数据库，可以在用户硬盘上存储大量数据（远大于 localStorage 的 5MB 限制）

**在断点续传中的作用**：

```javascript
// 用户上传到一半关闭页面 - 切片保存在 IndexedDB
// 下次打开页面 - 从 IndexedDB 读取未完成的切片继续上传
```

**为什么用它**：
- 存切片文件（Blob 类型）
- 容量大（几百 MB 没问题）
- 持久化（用户不主动清除就一直在）

---

#### 2. wss - WebSocket Secure

**是什么**：
基于 TCP 的全双工通信协议，wss 是它的加密版本（类似 https）

**特点**：
- 实时双向通信（不像 HTTP 必须客户端先请求）
- 长连接（一次连接持续通信）

**在断点续传中的作用**：
```javascript
// 场景1：服务端主动通知"你的第5个切片我收到了"
ws.onmessage = (e) => {
  if (e.data.type === 'chunk_received') {
    // 可以把这个切片从待上传队列移除
  }
}

// 场景2：连接断了立刻知道
ws.onclose = () => {
  // 立即暂停上传，保存进度，准备重连
}
```

**用 HTTP 不行吗？**
- HTTP 只能客户端问、服务端答
- WebSocket 可以服务端主动告诉客户端"别传了，我在重启"

---

#### 3. Hash 命名 - 文件指纹

**是什么**：
通过算法（如 MD5）计算出的唯一字符串，类似文件的"身份证号"

```javascript
// 示例
文件 "cat.jpg" 内容 → 计算 → "d41d8cd98f00b204e9800998ecf8427e"
```

**在断点续传中的作用**：

**a. 文件级别的 hash**（整个文件的指纹）
```javascript
// 目的：秒传 + 避免重复上传
fileHash = md5(整个文件)

// 上传前先问后端：这个文件你有了吗？
POST /check-file { fileHash: "d41d8c..." }
// 后端返回：已有 → 秒传成功；部分有 → 传缺失的切片
```

**b. 切片级别的 hash**（每个切片的指纹）
```javascript
// 切片命名格式：fileHash-index-chunkHash
chunkName = `${fileHash}-${index}-${md5(chunk)}`

// 为什么还要切片的hash？
- 防止传输过程中数据损坏
- 后端合并前校验："这个切片完整吗？"
```

**hash 怎么算**：
```javascript
// 使用 spark-md5 库，分块计算避免卡顿
const spark = new SparkMD5()
// 分块读取文件，逐步更新 hash
spark.append(chunkData)
```

---

#### 一句话总结

- **IndexedDB**：把切片存用户硬盘，页面关了也不丢
- **wss**：实时知道上传状态，断网立刻感知
- **hash**：文件的身份证，用来秒传和校验数据完整性

## setTimeout 与异步

`setTimeout` 实现异步的核心原理，不是靠它自己“计算时间”，而是靠浏览器的**事件循环机制**和**多线程模型**。

可以分两层来理解它的运作方式：

### 1. 核心流程：把任务“寄存”出去
JavaScript 引擎是单线程的，它没办法一边计时一边执行代码。所以 `setTimeout` 的计时工作其实是交给**浏览器（或 Node.js）的定时器线程**去做的。

具体的执行流水线如下：

- **第一步：交单**。
  JS 引擎执行到 `setTimeout(fn, 1000)`，它只做一件事：**把 `fn` 连同“1000ms 后执行”这个指令，提交给浏览器的定时器线程**。交完立马接着执行下一行代码（这就是“非阻塞/异步”的来源）。

- **第二步：后台计时**。
  浏览器的定时器线程开始计时。此时 **JS 主线程是空闲的吗？不，它正在疯狂执行 `setTimeout` 后面的同步代码**。

- **第三步：排队等号**。
  1000ms 到了，定时器线程不会直接把 `fn` 塞给 JS 主线程（那会打断当前执行，导致不可预测的错误）。它会**把 `fn` 放进一个叫“任务队列”的地方**，让它排队。

- **第四步：事件循环召唤**。
  JS 主线程把手头的同步代码全执行完了，执行栈清空。这时，**事件循环** 就会去任务队列里看一眼：“咦，有活干了”，然后把 `fn` 从队列里拽出来，推到执行栈里执行。

### 2. 为什么有时候 1000ms 不精准？
正是因为上面的**第三步排队机制**，导致了延迟只多不少。

如果 `setTimeout` 到点了，但主线程正在死循环或者处理一个长达 2 秒的复杂计算，`fn` 就必须在队列里干等着。

```javascript
setTimeout(() => {
  console.log('我等了很久才出来');
}, 0); // 即便是 0ms 延迟

// 假设这是一个耗时 2 秒的循环
for (let i = 0; i < 10000000000; i++) { 
  // 主线程被堵在这里 2 秒
}

// 输出顺序：
// 1. 循环结束
// 2. 才会输出 '我等了很久才出来' （实际延迟 2000ms+，而不是 0ms）
```

### 3. 进阶视角：`setTimeout` 定义了一个“宏任务”
在面试或深入研究时，可以记住这个概念：`setTimeout` 产生的任务属于 **宏任务**。事件循环每一轮只会从任务队列里**取一个**宏任务来执行。

如果你遇到更精细的任务调度（比如需要在一个动画帧之后立刻执行），仅仅理解 `setTimeout` 还不够，可能会用到 `Promise` 或 `requestAnimationFrame`。需要我展开讲讲它们和 `setTimeout` 在调度时机上的具体区别吗？

## Promise.resolve

`Promise.resolve` 在 Promise 体系中扮演着两个核心角色：**创建一个立即成功的 Promise**，以及**在异步流程中标准化任意值为 Promise**。

简单来说，它的作用是：**把一个值（可能是普通值、thenable 对象，或另一个 Promise）包装成一个 Promise 对象，并确保最终的行为符合 Promise 规范。**

我们可以分两层来理解它的具体行为：

### 1. 静态方法：`Promise.resolve(value)`
这是最常用的形式，它会根据传入的参数类型，产生不同的结果：

| 参数类型          | 行为                                                         | 代码示例                                                     |
| :---------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **普通值**        | 返回一个**状态为 Fulfilled** 的 Promise，其 `[[PromiseResult]]` 即为该值。 | `Promise.resolve('hello').then(v => console.log(v))` // 输出 hello |
| **Promise 实例**  | 具有**幂等性**。如果传入的是一个 Promise，它会直接返回这个**完全相同的 Promise 实例**，不会做任何包装或修改。 | `const p1 = new Promise(...);` <br> `const p2 = Promise.resolve(p1);` <br> `console.log(p1 === p2);` // **true** |
| **Thenable 对象** | 尝试调用该对象的 `then` 方法，并**根据 thenable 的最终状态来决定返回的 Promise 状态**。常用于兼容旧式异步库（如 jQuery 的 `$.Deferred`）。 | `Promise.resolve({ then: (res, rej) => res(10) })`           |

### 2. 构造函数内的参数名：`resolve`
在 `new Promise((resolve, reject) => {})` 中，这个 `resolve` 是构造函数传给执行器的**函数引用**。

它的职责是：**将 Promise 的状态从 Pending 改为 Fulfilled**。

```javascript
new Promise((resolve, reject) => {
  setTimeout(() => {
    // 这里的 resolve 就是用来改变状态的函数
    resolve('异步操作成功');
  }, 1000);
})
```
**注意：** 调用此处的 `resolve` 并不会阻塞后续代码的执行（除非使用 `await`），它只是发出了一个“状态变更指令”。

### 3. 进阶用法：在 `async` 函数中的隐形转换
这其实是你最常使用 `Promise.resolve` 逻辑的场景，只是它被语法糖隐藏了。

在一个 `async` 函数中，**`return` 的值会被自动调用 `Promise.resolve` 进行包装**。

```javascript
async function getData() {
  // 尽管你返回的是普通对象，但它实际返回的是 Promise.resolve({ name: 'Li' })
  return { name: 'Li' }; 
}

// 所以你可以使用 .then
getData().then(user => console.log(user));
```

### 4. 实践应用场景
掌握 `Promise.resolve` 后，最实用的场景是**统一异步/同步接口**：

```javascript
function getUserFromCache(id) {
  const cached = localStorage.getItem(id);
  if (cached) {
    // 即便是同步读取的数据，也包装成 Promise 返回
    // 这样调用方永远可以用 .then 或 await 来处理，不用区分数据类型
    return Promise.resolve(JSON.parse(cached));
  } else {
    // 异步请求
    return fetch(`/api/user/${id}`).then(r => r.json()】);
  }
}
```



## 前端实现页面截图（canvas）

### 一、 背景与痛点
在前端业务中，将 DOM 内容转化为图片的需求非常普遍。结合场景，主要有以下几类：

1.  **内容预览与分享**：例如飞书文档列表页，希望不点进详情就能看到内容缩略图；或者将页面特定区域（如排行榜、战报）保存为 PNG 分享到社交平台。
2.  **业务凭证留存**：例如医疗行业的电子化验单、挂号单、处方笺；电商的订单详情、物流底单。这些内容通常由 CSS 动态渲染，需要将其固化为不可篡改的图片格式（PNG/PDF）提供给用户打印或存档。
3.  **设计交付与走查**：设计师或产品经理看到线上 Bug 或优秀案例，想一键截图导出并附带 CSS 样式信息，用于修改文档或设计稿。

### 二、 技术方案对比
针对“内容导出为 PNG”这一核心目标，前端主要有以下三种技术路径：

| 方案                          | 原理                                                         | 优点                                                   | 缺点                                                         | 适用场景                                         |
| :---------------------------- | :----------------------------------------------------------- | :----------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------- |
| **1. Canvas 原生绘制**        | 利用 Canvas API `drawImage` 将图片、视频帧画进去，或 `fillText` 绘制文字。 | 性能极好，灵活度极高，可精确控制像素级输出。           | 对于复杂的 DOM 结构（如 `div` 嵌套、`flex` 布局）需手动换算坐标重绘，**代码侵入性强，无法直接“截取”网页**。 | 纯图片合成、海报编辑器（Canvas 就是画板）。      |
| **2. html2canvas (主流选择)** | 遍历 DOM 树，解析每个节点的 Computed Style，将其“翻译”成 Canvas 的绘制指令。 | **非侵入式**，只需传入 `selector`，自动解析 CSS。      | 遇到 `box-shadow`、`border-radius` 叠加、`SVG` 或 `Canvas` 内嵌内容时**可能有解析偏差或空白**；性能开销较大。 | **业务后台截图、单据导出、缩略图生成**。         |
| **3. Puppeteer (无头浏览器)** | 启动一个真实的 Chrome 内核（Headless），打开网页后调用 `Page.screenshot()`。 | **还原度 100%**，CSS 新特性、WebGL、字体渲染完全一致。 | **极重**（需启动浏览器进程），无法在前端浏览器直接运行，必须部署在 **Node.js 服务端**。 | 服务端批量生成 PDF、页面监控、高精度设计稿还原。 |

> **补充说明**：“无头表格”或“无头 UI 库”通常是指类似 `jsPDF` 结合 Canvas 或直接在 Node 端通过虚拟 DOM 计算坐标的方案，但普适性和还原度不如 Puppeteer。

### 三、 实际落地设计：通用截图工具协议
为了实现“通用性”和“按需截图”，我们需要设计一个规范的工具函数。补充具体协议如下：

**1. 模式选择：函数式 vs 组件式**

- **函数式（推荐）**：接收参数，返回 Promise。灵活，不侵入视图层。
- **组件式**：封装一个 `<ScreenShot>` 组件，利用插槽包裹内容。适合特定区域的显隐控制。

**2. 隐藏 Canvas 的策略（隐藏画布技术）**
为了保证截图过程不影响用户当前页面的视觉，通常将解析用的 Canvas 放在屏幕外：

```css
.offscreen-canvas {
  position: fixed;
  top: -9999px;
  left: -9999px;
  visibility: hidden;
  pointer-events: none;
}
```

### 四、 代码编写示例（TypeScript + html2canvas）

以下是实现一个具备**通用性**、**防抖**、**CDN 上传**和**下载回退**的 Hook：

```typescript
import html2canvas from 'html2canvas';
import { useDebounceFn } from '@vueuse/core'; // 结合你之前问的防抖

interface ScreenshotOptions {
  selector: string;        // 对应标签选择器
  scale?: number;          // 清晰度，建议 2-3
  backgroundColor?: string;
  fileName?: string;
}

export const useScreenshot = () => {
  // 1. 核心截图逻辑
  const capture = async (options: ScreenshotOptions): Promise<File | null> => {
    const { selector, scale = 2, backgroundColor = '#ffffff' } = options;
    
    const element = document.querySelector(selector);
    if (!element) {
      console.error(`[Screenshot] 选择器 "${selector}" 未找到元素`);
      return null;
    }

    try {
      // 2. 调用 html2canvas 解析 DOM -> Canvas
      const canvas = await html2canvas(element as HTMLElement, {
        scale,                    // 提高分辨率，解决模糊问题
        backgroundColor,          // 避免 PNG 透明背景导致的显示异常
        allowTaint: false,        // 安全限制，防止跨域图片污染
        useCNAME: true,           // 尝试用 CNAME 解决图片跨域
        logging: false,           // 关闭调试日志
      });

      // 3. Canvas 转 Blob (为上传 CDN 做准备)
      return new Promise((resolve) => {
        canvas.toBlob((blob) => {
          if (blob) {
            const file = new File([blob], `${options.fileName || 'screenshot'}.png`, { type: 'image/png' });
            resolve(file);   // 外部函数由于 await 了这个 Promise，会拿到 resolve(file) 传递出来的 file
          } else {
            resolve(null);
          }
        }, 'image/png');
      });

    } catch (error) {
      console.error('截图失败:', error);
      return null;
    }
  };

  // 4. 上传 CDN 逻辑 (模拟)
  const uploadToCDN = async (file: File): Promise<string> => {
    const formData = new FormData();
    formData.append('file', file);
    // 实际项目中替换为你的上传接口
    // const res = await axios.post('/api/upload', formData);
    // return res.data.url;
    
    // 模拟返回 URL
    return URL.createObjectURL(file);
  };

  // 5. 本地下载兜底
  const downloadImage = (file: File) => {
    const url = URL.createObjectURL(file);
    const link = document.createElement('a');
    link.href = url;
    link.download = file.name;
    link.click();
    URL.revokeObjectURL(url);
  };

  // 6. 暴露给业务用的防抖方法 (处理高频点击)
  const handleScreenshot = useDebounceFn(async (options: ScreenshotOptions) => {
    const file = await capture(options);
    if (!file) return;

    // 假设业务需要上传到 CDN（Content Delivery Network） 获取永久链接
    const cdnUrl = await uploadToCDN(file);
    console.log('截图已上传 CDN:', cdnUrl);
    
    // 同时保留本地下载能力
    // downloadImage(file);
  }, 500);

  return {
    capture,
    handleScreenshot,
  };
};
```

### 五、 避坑指南（你内容中隐含的难点补充）

1.  **Canvas 污染（Tainted Canvas）**：
    - **现象**：画布上有跨域图片，调用 `toBlob` 时报错。
    - **解决**：后端图片必须配置 CORS 响应头（`Access-Control-Allow-Origin`），且前端加载图片时加 `crossorigin="anonymous"` 属性。

2.  **滚动条与溢出隐藏**：
    - **现象**：截图中包含了浏览器的滚动条，或者 `overflow: auto` 区域只截到了可视区。
    - **解决**：在调用 `html2canvas` 前，临时修改 DOM 样式 `overflow: visible` 和 `height: auto`，截图后再复原。

3.  **Canvas 模糊问题**：
    - **原因**：屏幕 DPI 与 Canvas 默认像素比不一致。
    - **解决**：代码中已包含 `scale: 2` 或 `scale: window.devicePixelRatio`，将画布物理像素放大，再通过 CSS 缩小显示（或直接下载大图）。

## 移动端适配问题

### 方案对比
- **多端分离**：不同端加载不同页面，隔离彻底但维护成本高。
- **动态加载 CSS**：根据 UA 判断加载不同样式文件，易产生样式冲突。
- **响应式布局**：一套代码通过媒体查询适配，维护成本最低，推荐使用。
- **CSS 预处理器**：配合响应式使用，通过变量和 Mixin 提升开发效率。

### 解决方案

#### 1. 设置视窗
```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
```
- `width=device-width`：视窗宽度等于设备宽度，是 rem 方案生效的前提。
- `viewport-fit=cover`：适配 iPhone 刘海屏，让背景色延伸到安全区外。

#### 2. 掌握媒体查询
```css
body {
  font-size: 16px;
}
@media (min-width: 780px) and (max-width: 1024px) {
  body {
    font-size: 18px;
  }
}
```
**补全建议**：采用移动优先原则（`min-width`），从小屏向大屏递进覆盖。

#### 3. 掌握 Flex 布局
- **主轴方向**：`flex-direction` 决定排列方向（row / column）。
- **次轴方向**：与主轴垂直的方向。
- **对齐方式**：`justify-content` 控制主轴，`align-items` 控制次轴。
- **弹性属性**：`flex: 1` 表示元素会按比例分配剩余空间。

**关键补全**：移动端 Flex 容器内文字溢出时，需加 `min-width: 0` 防止撑开布局。
```css
.flex-item {
  flex: 1;
  min-width: 0;
  overflow: hidden;
  text-overflow: ellipsis;
  white-space: nowrap;
}
```

#### 4. 图片响应式处理
```html
<picture>
  <source srcset="img-tablet.jpg" media="(min-width: 780px)">
  <source srcset="img-desktop.jpg" media="(min-width: 1080px)">
  <img src="img-mobile.jpg" alt="默认图片">
</picture>
```
**补全建议**：结合 `srcset` 的 `2x`、`3x` 标识，可同时适配 Retina 高清屏：
```html
<source srcset="img@2x.jpg 2x, img@3x.jpg 3x">
```

#### 5. rem 和 em 单位
```css
html {
  font-size: 16px;
}
@media (min-width: 780px) and (max-width: 1024px) {
  html {
    font-size: 18px;
  }
}
.header {
  height: 1rem; /* 默认 16px，屏幕 780px~1024px 时为 18px */
}
```
**关键区分**：
- **rem**：相对于根元素 `html` 的 `font-size`，适合全局统一缩放。
- **em**：相对于当前元素或父元素的 `font-size`，适合组件内部局部缩放。

**补全建议**：现代方案可用 **vw** 替代 rem，无需 JS 动态计算：

```css
.header {
  height: 13.33vw; /* 设计稿 750px 下，100px 对应 13.33vw */
}
```

### 移动端坑点
- **1px 边框变粗**：Retina 屏下用伪元素 + `transform: scaleY(0.5)` 实现物理 1px。
- **键盘弹起遮挡**：`position: fixed` 在软键盘弹出时会失效，关键输入框需避开底部。
- **安全区适配**：结合 `env(safe-area-inset-bottom)` 为 iPhone 底部留出安全距离。



## 访问端识别

### 背景
一个链接访问页面，需要同时适配 PC 端和 Mobile 端，根据设备类型展现不同的页面布局或内容。

### 方案核心思路
1. **先识别**：判断当前访问设备是 PC 还是 Mobile。
2. **后渲染**：根据识别结果，加载对应组件或执行跳转逻辑。

### 识别端：User-Agent 方案

**核心 API**：`navigator.userAgent`

```js
console.log(navigator.userAgent);
// 移动端典型特征：Mobi、Android、iPhone、iPad

function isMobile() {
  return /Mobi|Android|iPhone|iPad/i.test(navigator.userAgent);
}

if (isMobile()) {
  // 跳转移动端页面
  window.location.href = '/mobile';
}
```

**补全建议**：更完善的 UA 正则（覆盖主流设备）：
```js
function isMobile() {
  return /Android|webOS|iPhone|iPad|iPod|BlackBerry|IEMobile|Opera Mini/i.test(navigator.userAgent);
}
```

### 框架中使用：跨组件传递设备信息

**React 实现**（Context 跨层级传递）：
```jsx
// 1. 创建 Context
const DeviceContext = React.createContext();

// 2. 顶层注入
<DeviceContext.Provider value={{ isMobile: isMobile() }}>
  <App />
</DeviceContext.Provider>

// 3. 子组件使用
function Header() {
  const { isMobile } = useContext(DeviceContext);
  return <div>{isMobile ? '移动端' : 'PC端'}</div>;
}
```

**Vue 实现**（Provide / Inject 依赖注入）：
```js
// 1. 顶层注入（重要 API）
provide('isMobile', isMobile());

// 2. 子组件使用
const isMobile = inject('isMobile');
```

### 服务端识别（SSR 场景）

当项目涉及服务端渲染（Next.js / Nuxt）时，仅靠前端 UA 识别会导致 **hydration 不匹配** 。

**Node.js 端识别**：
```js
// 从请求头获取 User-Agent
export function getServerSideProps({ req }) {
  const ua = req.headers['user-agent'] || '';
  const isMobile = /Mobi|Android/i.test(ua);
  return { props: { isMobile } };
}
```

### 注意事项
1. **UA 可被伪造**：User-Agent 仅作参考，不能用于权限校验。
2. **iPadOS 13+ 特例**：iPad 新版系统 UA 与 Mac 相同，需配合 `navigator.maxTouchPoints` 辅助判断。
```js
const isIPad = /Macintosh/i.test(ua) && navigator.maxTouchPoints > 1;
```
3. **屏幕尺寸辅助**：结合 `window.innerWidth` 作为降级方案（`width <= 768` 视为移动端）。

### 渲染策略选择
| 策略             | 实现方式                              | 适用场景               |
| :--------------- | :------------------------------------ | :--------------------- |
| **跳转分流**     | 识别后重定向到不同域名（`m.xxx.com`） | 两套完全独立的项目     |
| **动态组件**     | 同页面内根据 `isMobile` 渲染不同组件  | 一套代码，组件级适配   |
| **CSS 媒体查询** | 组件相同，仅样式变化                  | 布局差异小，仅视觉调整 |

最好是用响应式来做，通过响应式来实现不同端的网页适配，或者通过查询访问媒体的最小宽度（useMediaQuery）来进行端的判断（不太准确）



## QPS 达到峰值

### 背景

当前端的应用的每秒查询次数(QPS)达到峰值会对服务端产生较大的压力，可能导致系统崩溃，我们需要进行请求的降低或者优化管理请求的发出来减轻负担。

### 方案

- 限流

### 请求限流

具体会在后端来做，请求来到服务端后会先有拦截，然后做对比这个请求与上一个请求的时间频率等等就不处理这个请求。

以nodejs为例，用rate-limt:

```js
const limter = ratelimte({
windowMs：60*1000,
max:100,
message:"请求过多，请稍后重试"
})
```

### 请求合并

将短时间内的请求进行合并， 以此降低服务端的压力。其实就是防抖节流：

- debounce（一般就用debounce防抖）
- throttle。

## web应用中对静态资源加载失败的场景做降级处理

### 背景

图片，css文件，js文件，放在CDN的文件，字体文件，服务端渲染失败等等，出现加载失败

### 解决方案

onerror='handleError()'

图片得有占位图，有alt来描述图片，加载失败，通过onerror调用对应处理函数，显示占位图，或可对图片做一个重试机制，然后上报服务器等等

css文件没加载到，关键性样式通过内联样式进行书写，使用备用样式，

js文件可能因为网络错误或者异常导致资源加载错误，同样是使用内联脚本以及备用脚本处理，上报

CDN的错误，就使用本地的备份文件，或者进行动态切换，切换到另一个CDN。

字体出错，font-family中可以写多个字体，前面的无效就自动降级到后面的字体，使用降级字体，使用webfont处理字体问题
