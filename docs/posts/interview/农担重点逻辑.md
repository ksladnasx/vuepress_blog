# 农担智能体开发平台

## 农担项目前端定位（先讲这个）

在农担项目里，我负责的是**企业级 AI 平台前端**，核心不是做单页展示，而是做三类能力：

- **业务配置化**：把复杂 AI 应用通过可视化页面配置出来（应用、流程、知识库）
- **系统治理化**：权限、系统管理、日志审计等管理端能力
- **交互产品化**：让构建端（platform）和会话端（client）都可稳定使用

---

## 前端架构怎么讲（面试高分点）

- **技术栈**：`React + TypeScript + Vite + react-router + axios`，并结合状态管理与组件化封装
- **双前端形态**：
  - `frontend/platform`：管理后台、应用构建、知识库、系统管理
  - `frontend/client`：用户会话与应用使用端
- **分层设计**：
  - 页面层：`pages/*`
  - 接口层：`controllers/API/*`
  - 网络层：`controllers/request.ts`（统一拦截器）
  - 路由层：`routes.tsx`（路由定义 + 权限过滤）

一句话概括：  **“页面负责交互编排，API 层负责业务调用，request 层负责统一治理（鉴权/错误/下载）。”**

---

## 重点知识点 & 记忆点（只讲前端）

1. **权限路由动态过滤**  
   - 在 `routes.tsx` 中给路由打 `permission` 标记，通过 `getPrivateRouter(permissions)` 递归过滤页面  
   - 价值：同一套前端支持不同角色视图，减少分叉开发

2. **统一请求治理**  
   - `request.ts` 对 `axios` 做统一处理：token 注入、业务码处理、401 续登、blob 下载兼容、错误 toast  
   - 价值：接口异常处理一致，减少页面重复代码

3. **构建器页面复杂交互拆分**  
   - `BuildPage` 下按能力拆分 `apps / flow / assistant / tools / skills`  
   - 价值：复杂业务场景仍能保持模块边界清晰、可维护

4. **知识库前端流程化管理**  
   - `KnowledgePage` 做文件库/QA 库、上传、预览、召回测试等完整链路  
   - 价值：把 AI 知识处理变成可操作的产品流程，而不只是接口调用

5. **系统管理后台能力完整**  
   - `SystemPage` 下用户、角色、用户组等治理能力  
   - 价值：前端不仅是“展示层”，而是企业权限管理的关键承载

6. **错误边界与稳定性保障**  
   - 路由页面通过 `ErrorBoundary` 包裹，防止局部崩溃影响全局  
   - 价值：提升后台场景稳定性与可运维性

## 重点问题 -> 前端解决方案（面试可直接讲）

- **问题1：不同角色看到不同菜单和页面怎么做？**  
  方案：路由元信息 + 权限过滤函数（不是后端返回整套路由），前端动态生成私有路由树。

- **问题2：每个页面都写鉴权和错误处理太重复，怎么收敛？**  
  方案：统一 axios 拦截器，页面只关心业务数据，不关心底层异常细节。

- **问题3：后台页很多，如何避免“巨石页面”难维护？**  
  方案：按领域拆页面目录（Build/Knowledge/System/Log），并把接口和 UI 解耦。

- **问题4：文件下载、普通 JSON、业务错误码并存怎么处理？**  
  方案：网络层识别 `responseType/blob/content-type`，分流处理，避免页面层混乱判断。

- **问题5：复杂页面崩溃会影响整个系统怎么兜底？**  
  方案：路由级 `ErrorBoundary + Fallback`，局部故障可恢复，提升线上可用性。

### 2) 你们前端架构是怎么分层的？为什么这样分？

**答法（主答）**  
我采用“路由层 -> 页面层 -> API层 -> request层”的分层：  
- 路由层：页面组织和权限过滤  
- 页面层：业务交互和状态编排  
- API层：领域接口聚合（按模块拆）  
- request层：统一鉴权、错误码、下载处理  

这样做的价值是：**减少重复代码、降低页面复杂度、提升可测试性和可替换性**。

**追问可补**  
当后端接口规范调整时，通常改 API 层或 request 层，不需要大面积改页面。

---

### 3) 多角色权限前端是怎么做的？

**答法（主答）**  
在农担项目里，我在路由配置里加 `permission` 元信息，然后通过 `getPrivateRouter(permissions)` 做递归过滤，只渲染当前角色可访问路由。  
这比“页面内到处写 if 权限判断”更可控，也便于后续扩展。

**追问可补**  
前端权限主要负责“展示和导航控制”，真正的安全边界仍以后端鉴权为准，两边是协同关系。

### 4) 你如何统一处理接口异常和登录态？

**答法（主答）**  
我在 `axios` 拦截器里统一做了：  
- token 注入  
- 业务状态码解包

- 文件下载响应透传

**追问可补**

## 重点技术

你说得对，前一版偏“架构概览”，这次我给你补上**前端技术深讲版**，专门讲你面试会被追问的两块：  
1) 农担项目里的实时通信（WebSocket/SSE）  
2) 前端工作流是怎么配置、保存、测试、发布的

### 1) 实时通信：农担前端怎么做的（重点讲 WebSocket）

先说结论：在工作流调试/运行链路里，前端主通道是 **WebSocket**（不是轮询），SSE 在项目里有上下文封装，但你这条主链路用的是 WS。

#### A. 为什么用 WebSocket
- 场景是“工作流逐节点执行 + LLM 流式输出 + 用户中断输入”
- 前端需要持续收到 `node_run`、`stream_msg`、`input`、`error`、`processing close` 等事件
- WS 比普通 HTTP 更适合双向、持续、低延迟事件流

#### B. 连接与协议设计（你可以这么讲）
在 `ChatInput` 里动态按 `http/https` 切成 `ws/wss`，连接 `workflow/chat` 通道：

```tsx
const ws = new WebSocket(`${webSocketProtocol}://${wsUrl}`)
wsRef.current = ws
// websocket linsen
ws.onopen = () => {
    console.log("WebSocket connection established!");
    res('ok')
};
```

- 建连后前端会先发 `init_data` 或 `check_status`（根据是否新会话）  
- 之后发 `input` 消息把用户输入与节点上下文传给后端  
- 后端持续推流，前端按 `category/type` 分发处理

#### C. 前端消息分发机制（这是技术亮点）
你可以说：我把 WS 事件做成“协议驱动 UI 状态机”，关键分支有：

- `error`：统一转 toast + 锁输入，写通知中心
- `node_run`：更新节点运行状态（loading/success/error），并把节点入参缓存给单节点调试
- `input`：动态决定是文本输入还是表单输入，控制 UI 解锁
- `stream_msg`：逐 token 追加，`end` 时触发完成事件
- `processing close`：一轮会话结束，插入会话分隔符并锁定输入

### 3) 面试时可直接说的“技术问题 -> 方案”

- **问题：如何支撑工作流运行中的实时节点反馈？**  
  方案：WebSocket 事件协议 + category/type 分发，节点状态通过自定义事件广播到画布节点。

- **问题：流式输出怎么避免消息错位？**

  方案：前端先做节点校验，再保存版本，再执行上线 API；在线版本通过版本分叉管理。
