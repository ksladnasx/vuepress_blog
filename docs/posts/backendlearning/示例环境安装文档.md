# 本地开发与部署说明

这份文档记录一种比较舒服的开发模式：数据库等基础服务用 Docker 跑，前端和后端在本机直接运行。这样既能省掉本机安装 PostgreSQL、Mailcatcher 等服务的麻烦，又能保留本地调试、热重载、断点调试的便利。

推荐模式如下：

```text
Docker: PostgreSQL / Adminer / Mailcatcher
本地: FastAPI 后端
本地: React + Vite 前端
```

## 一、项目结构先搞清楚

这个仓库是一个全栈项目：

```text
full-stack-fastapi-template/
├── backend/          # FastAPI 后端
├── frontend/         # React + Vite 前端
├── compose.yml       # 生产/完整 Docker Compose 配置
├── compose.override.yml
├── .env              # 后端、数据库、Docker Compose 共用配置
├── pyproject.toml    # uv workspace 配置
└── uv.lock
```

后端自己的依赖写在：

```text
backend/pyproject.toml
```

根目录的 `pyproject.toml` 把 `backend` 声明成 uv workspace 的一个成员：

```toml
[tool.uv.workspace]
members = ["backend"]
```

所以使用 uv 原生环境时，可以在仓库根目录统一同步依赖。但如果你想使用 conda 环境，最好明确告诉 uv 要把包安装到哪个 Python 里，后面会讲。

## 二、推荐开发模式：Docker 跑数据库，本地跑后端和前端

### 1. 准备 conda 后端环境

这个项目要求 Python 3.14。先创建并进入专门的 conda 环境：

```powershell
conda create -n fastapi-template python=3.14
conda activate fastapi-template
```

确认当前 Python 是这个 conda 环境里的 Python：

```powershell
python --version
where.exe python
```

`python --version` 应该显示 `Python 3.14.x`。`where.exe python` 应该优先指向类似下面的位置：

```text
C:\Users\你的用户名\miniconda3\envs\fastapi-template\python.exe
```

 conda 环境常用管理命令包括：

```bash
conda env list            # 查看所有环境
conda activate 环境名      # 进入环境
conda deactivate          # 退出环境
conda remove -n 环境名 --all   # 删除环境
```

在该环境中安装项目依赖通常使用：

```bash
pip install -r requirements.txt
```

或在支持现代工具的情况下使用 `uv sync ` 自动完成依赖安装。

### 2. 安装 uv

如果当前环境里没有 `uv`，先装：

```powershell
pip install uv
```

确认：

```powershell
uv --version
```

### 3. 把后端依赖装进 conda 环境

进入仓库根目录：

```powershell
cd C:\Users\wh826\Desktop\团支书\实习\Try\full-stack-fastapi-template
```

把 `backend` 安装到当前 conda 环境里：

```powershell
uv pip install --python "$env:CONDA_PREFIX\python.exe" -e .\backend
```

这里的重点是 `--python "$env:CONDA_PREFIX\python.exe"`。它明确告诉 uv：不要猜环境，直接把依赖装进当前 conda 环境。

验证 FastAPI 是否真的装进来了：

```powershell
python -c "import fastapi; print(fastapi.__version__)"
```

如果能打印版本号，后端依赖就没问题。

### 4. 启动数据库和辅助服务

在仓库根目录启动 PostgreSQL、Adminer、Mailcatcher（其实只用db就行）：

```powershell
docker compose up -d db adminer mailcatcher
```

这些服务的作用：

```text
db           PostgreSQL 数据库
adminer      数据库管理页面
mailcatcher  本地邮件捕获服务
```

常用地址：

```text
Adminer:     http://localhost:8080
Mailcatcher: http://localhost:1080
```

如果第一次执行这个命令很慢，通常不是项目启动慢，而是 Docker 正在拉镜像。比如 `adminer` 和 `mailcatcher` 第一次都需要从镜像仓库下载，国内网络可能会卡很久。下载完成后，后面再次启动一般就会很快。

**如果你现在只是想先把后端跑起来，其实可以只启动数据库：**

```powershell
docker compose up -d db
```

后端只依赖 PostgreSQL。**`adminer` 只是数据库管理页面，`mailcatcher` 只是本地收邮件用的工具，不是启动后端的必需项。**

如果你后面需要看数据库，再单独启动：

```powershell
docker compose up -d adminer
```

如果你后面需要测试邮件，再单独启动：

```powershell
docker compose up -d mailcatcher
```

另外，启动时看到下面这个 warning 通常可以忽略：

```text
The "CI" variable is not set. Defaulting to a blank string.
```

这是 `compose.override.yml` 里给 Playwright 测试服务预留的环境变量。我们只跑本地开发服务时，它不会影响数据库、后端或前端启动。

后端默认从根目录 `.env` 读取数据库配置。当前 `.env` 里默认是：

```env
POSTGRES_SERVER=localhost
POSTGRES_PORT=5432
POSTGRES_DB=app
POSTGRES_USER=postgres
POSTGRES_PASSWORD=changethis
```

因为我们本地跑后端，数据库端口又由 Docker 映射到了本机的 `5432`，所以 `POSTGRES_SERVER=localhost` 正好合适。

### 5. 初始化数据库

第一次启动数据库后，需要跑迁移和初始数据。

进入后端目录：

```powershell
cd backend
```

等待数据库可用：

```powershell
python app/backend_pre_start.py
```

执行数据库迁移：

```powershell
alembic upgrade head
```

创建初始数据：

```powershell
python app/initial_data.py
```

默认超级用户是来自根目录`.env`：

```env
FIRST_SUPERUSER=admin@example.com
FIRST_SUPERUSER_PASSWORD=changethis
```

### 6. 本地启动后端

在 `backend` 目录：

```powershell
python -m fastapi dev app/main.py
```

后端地址：

```text
http://localhost:8000
http://localhost:8000/docs
```

如果你看到下面这些 warning，不是依赖问题：

```text
The value of SECRET_KEY is "changethis"
The value of POSTGRES_PASSWORD is "changethis"
The value of FIRST_SUPERUSER_PASSWORD is "changethis"
```

这是项目提醒你：这些默认密钥只能用于本地开发，部署时必须改掉。

### 7. 本地启动前端

另开一个终端，进入仓库根目录或前端目录。

如果还没装前端依赖：

```powershell
cd C:\Users\wh826\Desktop\团支书\实习\Try\full-stack-fastapi-template\frontend
bun install
```

启动前端：

```powershell
bun run dev
```

或者在仓库根目录执行：

```powershell
bun run dev
```

前端地址：

```text
http://localhost:5173
```

前端会读取 `frontend/.env`：

```env
VITE_API_URL=http://localhost:8000
MAILCATCHER_HOST=http://localhost:1080
```

这正好对应本地后端和本地 Mailcatcher。

## 四、前面遇到的问题总结

### 1. `uv` 命令找不到

报错：

```text
uv : 无法将“uv”项识别为 cmdlet、函数、脚本文件或可运行程序的名称
```

原因很简单：当前环境里没有安装 uv，或者 uv 不在 PATH 里。

解决：

```powershell
pip install uv
```

然后：

```powershell
uv --version
```

### 2. `fastapi` 命令找不到

报错：

```text
fastapi : 无法将“fastapi”项识别为 cmdlet、函数、脚本文件或可运行程序的名称
```

这通常说明当前环境没有安装 FastAPI CLI，或者脚本目录没有进 PATH。

先用模块方式启动，更稳：

```powershell
python -m fastapi dev app/main.py
```

如果报：

```text
No module named fastapi
```

那就说明 FastAPI 根本没装进当前 Python 环境。

### 3. `uv sync --active --package app` 看起来成功，但 conda 里没有 FastAPI

我们遇到的情况是：

```powershell
uv sync --active --package app
```

显示同步成功，但：

```powershell
python -m fastapi dev app/main.py
```

仍然报：

```text
No module named fastapi
```

原因是：`uv sync` 是“项目同步命令”，它会根据仓库里的 `pyproject.toml`、`uv.lock`、workspace 配置去同步项目环境。这个项目根目录已经有 `.venv`，所以 uv 很容易继续使用项目自己的 `.venv`，而不是你当前激活的 conda 环境。

`conda activate` 会让命令行里的 `python` 指向 conda 环境，但 uv 的 workspace 同步逻辑不一定会把 conda 环境当成它要同步的目标。

所以 conda 场景下更推荐：

```powershell
uv pip install --python "$env:CONDA_PREFIX\python.exe" -e .\backend
```

这句命令的意思是：明确把后端项目安装到当前 conda 环境的 Python 里，不让 uv 猜。

### 4. Python 版本不对

后端 `backend/pyproject.toml` 里写了：

```toml
requires-python = ">=3.14,<4.0"
```

所以当前环境必须是 Python 3.14。如果你用的是 Python 3.11、3.12、3.13，都可能出现各种依赖或解析问题。

检查：

```powershell
python --version
```

### 5. `.venv` 和 conda 环境不要混用

这个项目可以用两种方式管理 Python 环境：

```text
方式一：uv 自己管理项目根目录 .venv
方式二：conda 管理 fastapi-template 环境
```

两种都可以，但开发时最好选一种坚持用。

如果使用 conda，就用：

```powershell
conda activate fastapi-template
uv pip install --python "$env:CONDA_PREFIX\python.exe" -e .\backend
python -m fastapi dev app/main.py
```

如果使用 uv 的 `.venv`，就用：

```powershell
uv sync --package app
uv run --package app fastapi dev backend/app/main.py
```

不要一会儿在 `.venv` 里装包，一会儿用 conda 的 `python` 启动。这样最容易出现“明明装了，启动时却说没有”的错觉。

### 6. 注册接口 500：`relation "user" does not exist`

如果前端注册时报 500，后端日志里看到类似：

```text
sqlalchemy.exc.ProgrammingError: relation "user" does not exist
```

这说明后端已经连上数据库了，但数据库表还没创建。常见原因是：只启动了 Docker 里的 PostgreSQL，但还没有执行 Alembic 迁移。

在 `backend` 目录执行：

```powershell
conda activate fastapi-template
python app/backend_pre_start.py
python -m alembic upgrade head
python app/initial_data.py
```

确认迁移版本：

```powershell
python -m alembic current
```

如果输出类似：

```text
fe56fa70289e (head)
```

就说明数据库已经升级到最新表结构。

## 五、其他本地启动方式

### 方式一：完整 Docker Compose 开发模式

这是模板官方最省事的方式。前端、后端、数据库、邮件服务都放进 Docker。

在仓库根目录：

```powershell
docker compose watch
```

访问：

```text
前端:       http://localhost:5173
后端:       http://localhost:8000
后端文档:   http://localhost:8000/docs
Adminer:    http://localhost:8080
Traefik UI: http://localhost:8090
Mailcatcher:http://localhost:1080
```

优点：最接近项目默认配置，少折腾本机 Python。

缺点：后端调试不如本地 Python 直接运行方便。

### 方式二：Docker 跑全套，但本地改前端

先启动全套：

```powershell
docker compose watch
```

停掉 Docker 里的前端：

```powershell
docker compose stop frontend
```

本地启动前端：

```powershell
cd frontend
bun run dev
```

这个模式适合主要改前端。

### 方式三：Docker 跑全套，但本地改后端

先启动全套：

```powershell
docker compose watch
```

停掉 Docker 里的后端：

```powershell
docker compose stop backend
```

本地启动后端：

```powershell
conda activate fastapi-template
cd backend
python -m fastapi dev app/main.py
```

注意：本地后端连接 Docker 里的数据库时，`.env` 里的 `POSTGRES_SERVER` 应该是 `localhost`。如果在 Docker 容器内部跑后端，则通常应该是 `db`。

### 方式四：uv 原生 `.venv` 模式

如果你不想用 conda，可以直接使用 uv 创建项目环境。

仓库根目录：

```powershell
uv python install 3.14
uv sync --package app
```

启动后端：

```powershell
uv run --package app fastapi dev backend/app/main.py
```

或者激活项目 `.venv`：

```powershell
.\.venv\Scripts\Activate.ps1
cd backend
fastapi dev app/main.py
```

这种方式和项目本身的 uv workspace 最贴合。

## 六、部署这个项目

部署和本地开发不一样。部署时建议使用 Docker Compose 跑完整服务，而不是用本地 conda 环境启动后端。

生产部署的大致结构是：

```text
公网请求
  ↓
Traefik
  ↓
frontend 容器 / backend 容器 / adminer 容器
  ↓
PostgreSQL 容器
```

### 1. 准备服务器和域名

你需要：

```text
一台 Linux 服务器
服务器上安装 Docker Engine
一个域名
域名 DNS 指向服务器 IP
最好配置通配子域名，例如 *.example.com
```

项目默认会把服务拆成这些域名：

```text
dashboard.example.com  前端
api.example.com        后端 API
adminer.example.com    数据库管理
traefik.example.com    Traefik 控制台
```

### 2. 先部署公共 Traefik

这个项目期望有一个公共 Traefik 负责 HTTPS 证书和反向代理。

在服务器上创建网络：

```bash
docker network create traefik-public
```

复制 `compose.traefik.yml` 到服务器，然后设置环境变量：

```bash
export USERNAME=admin
export PASSWORD=请换成强密码
export HASHED_PASSWORD=$(openssl passwd -apr1 $PASSWORD)
export DOMAIN=example.com
export EMAIL=admin@example.com
```

启动 Traefik：

```bash
docker compose -f compose.traefik.yml up -d
```

### 3. 配置生产环境变量

部署前一定要改根目录 `.env` 或通过服务器环境变量注入这些值：

```env
ENVIRONMENT=production
DOMAIN=example.com
SECRET_KEY=请换成随机强密钥
POSTGRES_PASSWORD=请换成强密码
FIRST_SUPERUSER_PASSWORD=请换成强密码
BACKEND_CORS_ORIGINS=https://dashboard.example.com,https://api.example.com
```

生成随机密钥可以用：

```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

不要在生产环境继续使用 `changethis`。

### 4. 使用生产 compose 文件启动

生产环境不要使用 `compose.override.yml`。这个文件主要是给本地开发用的。

在服务器项目目录：

```bash
docker compose -f compose.yml build
docker compose -f compose.yml up -d
```

查看状态：

```bash
docker compose -f compose.yml ps
docker compose -f compose.yml logs backend
```

`compose.yml` 里有一个 `prestart` 服务，它会在后端正式启动前做这些事：

```text
等待数据库启动
执行 alembic upgrade head
创建初始数据
```

所以生产部署时不需要手动在服务器上进入 Python 环境跑迁移，正常让 Docker Compose 流程完成即可。

### 5. GitHub Actions 自动部署

项目里已经带了 GitHub Actions 部署相关配置，适合后面做持续部署。

大致思路是：

```text
推送代码或发布 release
GitHub Actions 被触发
自托管 runner 在服务器上执行部署命令
Docker Compose 构建并更新服务
```

要使用它，需要在 GitHub 仓库里配置环境、secrets、自托管 runner。详细步骤看根目录的 `deployment.md`。

## 七、推荐怎么选

如果你是日常开发，推荐：

```text
Docker 跑 db/adminer/mailcatcher
conda 跑 backend
bun 跑 frontend
```

也就是本文第二节的方式。

如果你只是想快速把项目跑起来看看：

```powershell
docker compose watch
```

如果你要部署到服务器：

```bash
docker compose -f compose.yml build
docker compose -f compose.yml up -d
```

开发时最重要的一条原则是：先确定你到底在用哪个 Python 环境。只要环境不串，依赖问题基本就会少很多。
