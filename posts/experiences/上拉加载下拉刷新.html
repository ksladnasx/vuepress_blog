上拉加载与下拉刷新实现笔记

这篇笔记整理 tests/上拉加载下拉刷新.html 这个 demo 的核心实现逻辑。它实现的是移动端常见的两种列表交互：

  • 下拉刷新：列表在顶部时，手指继续向下拖动，超过阈值后松手刷新。
  • 上拉加载：列表滚动到底部附近时，自动追加更多数据。

页面结构

页面主要由三块组成：

<div class="scroll-container" id="scrollContainer">
  <div class="pull-refresh" id="pullRefreshBar">...</div>
  <div class="feed-list" id="feedList">...</div>
  <div class="load-more-trigger" id="loadMoreTrigger">...</div>
</div>

其中：

  • scrollContainer 是真正负责滚动的容器。
  • pullRefreshBar 是顶部下拉刷新提示条，默认高度为 0。
  • feedList 是列表内容区域。
  • loadMoreTrigger 是底部加载更多提示区域。

整体结构可以理解为：

固定头部 app-header
  |
  v
滚动容器 scrollContainer
  |
  |-- 顶部刷新条 pullRefreshBar
  |-- 列表内容 feedList
  |-- 底部加载提示 loadMoreTrigger

核心状态变量

demo 中最关键的几个状态变量如下：

let allItems = [];
let isLoadingMore = false;
let isRefreshing = false;
let hasMore = true;

const MAX_ITEMS_LIMIT = 180;
const INITIAL_COUNT = 10;

//滚动容器顶部被卷起的高度，即滚动条距离顶部的垂直偏移量
 const scrollTop = scrollContainer.scrollTop;
//滚动容器的可视高度，即元素内容区的高度（不包括滚动条和边框）
 const clientHeight = scrollContainer.clientHeight;
//滚动容器的内容高度，即元素内容区的总高度（包括被卷起的不可见部分）
 const scrollHeight = scrollContainer.scrollHeight;

它们分别表示：

  • allItems：当前列表中所有数据。
  • isLoadingMore：是否正在执行上拉加载。
  • isRefreshing：是否正在执行下拉刷新。
  • hasMore：是否还有更多数据可以加载。
  • MAX_ITEMS_LIMIT：最多加载多少条数据。
  • INITIAL_COUNT：初始化或刷新时生成多少条数据。

这里最重要的是 isLoadingMore 和 isRefreshing。它们相当于两个互斥锁，用来避免刷新和加载更多同时发生。

performRefresh() 开始
  -> isRefreshing = true
  -> 禁止重复刷新
  -> 禁止同时加载更多

loadMoreData() 开始
  -> isLoadingMore = true
  -> 禁止重复加载
  -> 禁止同时刷新

所以代码里经常能看到类似判断：

if (isRefreshing || isLoadingMore) return;

初始化流程

初始化函数 init() 做了几件事：

页面加载
  |
  v
init()
  |
  |-- generateBatch(INITIAL_COUNT) 生成初始数据
  |-- allItems = 初始数据
  |-- renderFullList() 渲染完整列表
  |-- updateLoadMoreUI(false, ...) 初始化底部加载文案
  |-- initEventListeners() 绑定 touch 和 scroll 事件
  |-- resetPullRefreshBar() 收起顶部刷新条
  v
页面进入可交互状态

初始化后的页面就开始同时监听两类事件：

  • touchstart、touchmove、touchend：负责下拉刷新。
  • scroll：负责上拉加载更多。

下拉刷新流程

下拉刷新依赖触摸事件，因为它需要感知用户手指拖动的过程。

1. 按下时记录起点

onTouchStart() 会记录两个值：

touchStartY = touch.clientY;
startScrollTop = scrollContainer.scrollTop;

含义是：

  • touchStartY：手指刚按下时的 Y 坐标。
  • startScrollTop：刚按下时滚动容器的位置。

2. 移动时判断是否可以下拉

onTouchMove() 中会计算拖动距离：

const deltaY = currentY - touchStartY;
const currentScrollTop = scrollContainer.scrollTop;

只有同时满足下面三个条件，才进入下拉刷新逻辑：

currentScrollTop <= 0 && deltaY > 0 && !isRefreshing

它们的含义是：

  • currentScrollTop <= 0：列表已经滚动到顶部。
  • deltaY > 0：手指正在向下拉。
  • !isRefreshing：当前没有正在刷新。

满足条件后，代码会阻止浏览器默认滚动，并动态改变顶部刷新条高度：

e.preventDefault();

let newHeight = Math.min(deltaY * 0.8, refreshBarMaxHeight);
pullRefreshBar.style.height = `${newHeight}px`;

这里的 deltaY * 0.8 是一个阻尼效果。用户手指向下拖 100px，刷新条不会真的展开 100px，而是展开 80px，这样手感更自然。

3. 超过阈值后切换文案

demo 中设置了一个刷新阈值：

let pullThreshold = 60;

当顶部刷新条高度超过 60px 时，文案会从“下拉刷新”变成“释放刷新”：

if (newHeight >= pullThreshold) {
  refreshTextSpan.innerText = '释放刷新';
  refreshIconSpan.style.transform = 'rotate(180deg)';
  hasReachedThreshold = true;
}

如果用户又往回收，低于阈值，则文案恢复为“下拉刷新”。

4. 松手时决定是否刷新

onTouchEnd() 会根据当前刷新条高度判断是否触发刷新：

const currentHeight = parseInt(pullRefreshBar.style.height, 10);

if (isPulling && currentHeight >= pullThreshold && !isRefreshing && !isLoadingMore) {
  performRefresh();
} else {
  resetPullRefreshBar();
}

也就是说：

  • 超过阈值再松手：执行刷新。
  • 没超过阈值就松手：直接收起刷新条。

5. 真正执行刷新

performRefresh() 是刷新动作的核心：

performRefresh()
  |
  |-- 如果正在刷新或加载，直接 return
  |-- isRefreshing = true
  |-- 顶部刷新条固定为 56px
  |-- 文案改成“刷新中...”
  |-- delay(600) 模拟接口请求
  |-- generateBatch(INITIAL_COUNT) 生成新数据
  |-- allItems = 新数据
  |-- renderFullList() 整体重绘列表
  |-- updateLoadMoreUI(false, ...)
  |-- delay(200)
  |-- resetPullRefreshBar()
  |-- isRefreshing = false

完整流程图如下：

用户手指按下
  |
  v
onTouchStart()
  |
  |-- 记录 touchStartY
  |-- 记录 startScrollTop
  |-- 重置 isPulling / hasReachedThreshold
  v
用户向下拖动
  |
  v
onTouchMove()
  |
  |-- 判断 scrollTop <= 0
  |-- 判断 deltaY > 0
  |-- 判断当前没有刷新
  |
  |-- 不满足条件:
  |     忽略或重置顶部刷新条
  |
  |-- 满足条件:
  |     阻止默认滚动
  |     根据拖动距离设置刷新条高度
  |
  |-- 高度 >= 60:
  |     文案变成“释放刷新”
  |     图标旋转
  |
  |-- 高度 < 60:
  |     文案保持“下拉刷新”
  v
用户松手
  |
  v
onTouchEnd()
  |
  |-- 高度 >= 60:
  |     performRefresh()
  |
  |-- 高度 < 60:
  |     resetPullRefreshBar()
  v
刷新完成后收起顶部刷新条

一句话总结：下拉刷新 = 顶部状态判断 + 手指拖动距离 + 超阈值松手触发。

上拉加载流程

上拉加载不直接依赖手指方向，而是监听滚动容器是否接近底部。这样做在实际项目里更常见，因为鼠标滚轮、触控板、键盘滚动、代码滚动都能统一触发。

1. 监听滚动事件

事件绑定在 initEventListeners() 里：

scrollContainer.addEventListener('scroll', onScrollHandler);

onScrollHandler() 开始时会先判断是否已经在加载或刷新：

if (isLoadingMore || isRefreshing) return;

然后使用一个 50ms 的防抖：

if (scrollTimer) clearTimeout(scrollTimer);

scrollTimer = setTimeout(() => {
  // 判断是否接近底部
}, 50);

这样可以避免滚动过程中频繁触发计算和加载。

2. 判断是否接近底部

代码通过三个值判断滚动位置：

const scrollTop = scrollContainer.scrollTop;
const clientHeight = scrollContainer.clientHeight;
const scrollHeight = scrollContainer.scrollHeight;

它们的含义是：

  • scrollTop：已经向上滚动出去的高度。
  • clientHeight：滚动容器可视区域高度。
  • scrollHeight：滚动内容的总高度。

判断接近底部的公式是：

scrollHeight - scrollTop - clientHeight <= 30

可以这样理解：

内容总高度 - 已滚动高度 - 可视区域高度 = 距离底部还剩多少

当这个结果小于等于 30px 时，就认为已经接近底部，可以加载更多。

3. 执行加载更多

满足条件后会调用：

loadMoreData();

loadMoreData() 的流程如下：

loadMoreData()
  |
  |-- 如果正在加载或刷新，直接 return
  |-- 如果没有更多数据或已经达到上限，更新底部文案并 return
  |-- isLoadingMore = true
  |-- updateLoadMoreUI(true, true) 显示“加载中...”
  |-- delay(500) 模拟请求
  |-- getLoadMoreCount() 随机生成 5 到 8 条
  |-- 如果即将超过上限，则裁剪本次新增数量
  |-- generateBatch(actualAddCount) 生成新数据
  |-- allItems.push(...newItems)
  |-- appendItemsToEnd(newItems) 只追加新增 DOM
  |-- 根据是否达到上限更新底部 UI
  |-- isLoadingMore = false

完整流程图如下：

用户向上滑动列表
  |
  v
scroll 事件触发
  |
  v
onScrollHandler()
  |
  |-- 如果正在刷新或加载:
  |     直接 return
  |
  |-- 50ms 防抖
  |
  |-- 读取 scrollTop
  |-- 读取 clientHeight
  |-- 读取 scrollHeight
  |
  |-- 判断:
  |     scrollHeight - scrollTop - clientHeight <= 30
  |
  |-- 如果接近底部且 hasMore 为 true:
  |     loadMoreData()
  |
  |-- 如果已经达到最大数量:
  |     hasMore = false
  |     updateLoadMoreUI(false, false)
  v
loadMoreData()
  |
  |-- 显示加载中
  |-- 模拟接口等待
  |-- 生成 5 到 8 条新数据
  |-- 追加到 allItems
  |-- 追加到 DOM 尾部
  |-- 更新底部加载状态
  v
加载完成

一句话总结：上拉加载 = 监听滚动位置 + 接近底部 + 异步追加数据。

刷新和加载的区别

这份 demo 中，下拉刷新和上拉加载的数据更新方式不同：

// 下拉刷新：替换整份数据
allItems = [...newData];
renderFullList();

// 上拉加载：追加新增数据
allItems.push(...newItems);
appendItemsToEnd(newItems);

区别可以总结为：

  • 下拉刷新通常代表“重新拉取最新数据”，所以会替换当前列表。
  • 上拉加载通常代表“继续请求下一页数据”，所以会追加到列表尾部。
  • 下拉刷新使用 renderFullList()，因为它需要重建完整列表。
  • 上拉加载使用 appendItemsToEnd()，因为它只需要把新增项插入尾部。

UI 状态更新

顶部刷新条的复位由 resetPullRefreshBar() 统一处理：

function resetPullRefreshBar() {
  pullRefreshBar.style.height = '0px';
  if (refreshIconSpan) refreshIconSpan.className = 'refresh-icon';
  if (refreshTextSpan) refreshTextSpan.innerText = '下拉刷新';
}

底部加载更多文案由 updateLoadMoreUI() 统一处理：

function updateLoadMoreUI(isLoading, hasData = true) {
  if (isLoading) {
    loadMoreText.innerHTML = '<span class="spinner"></span> 加载中...';
  } else {
    if (!hasData || allItems.length >= MAX_ITEMS_LIMIT) {
      loadMoreText.innerHTML = '已经到底啦';
      hasMore = false;
    } else {
      loadMoreText.innerHTML = '上拉加载更多';
      hasMore = true;
    }
  }
}

把 UI 状态集中在函数里处理，好处是主流程更清晰：业务函数只负责决定“现在是什么状态”，具体文案和样式交给 UI 函数。

最后总结

这个 demo 的核心可以压缩成四句话：

  • 下拉刷新靠 touchstart、touchmove、touchend，重点是判断是否在顶部和是否超过下拉阈值。
  • 上拉加载靠 scroll，重点是判断距离底部是否小于指定阈值。
  • isRefreshing 和 isLoadingMore 用来避免两个异步流程互相打架。
  • 刷新是替换列表，加载更多是追加列表。

真正写项目时，可以把这里的 generateBatch() 和 delay() 替换成接口请求，把 allItems 换成框架里的响应式状态，例如 Vue 的 ref([]) 或 React 的 useState([])。
