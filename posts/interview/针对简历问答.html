一、项目深挖（这是重点）

1. Localstorage + BroadcastChannel（非常容易被问）

面试官： 你简历里提到用 BroadcastChannel 解决 SSE 连接数限制，详细说一下。

面试者： 当时我们的数据上云平台用了 SSE 做实时消息推送。用户如果打开多个标签页连接会迅速增加，而在http1.1的情况下，浏览器只支持最大六个sse长连接，如果多开页面，会导致新的浏览器页面echart的数据展示阻塞，从而影响用户体验。

由于从业务逻辑上讲，用户客观上存在打开多个页面监控多个不同服务器数据的需求，因此不能简单从逻辑上限制用户多开页面，并且升级http协议或者sharedworkker的方式都会导致代码大幅度的修改，浪费大量时间且会影响后续的代码维护，因此我在搜寻多方解法后选择了BroadcastChannel 的方式来解决这个问题。这个解决方式的核心就是，通过只允许一个主标签页建立 SSE，其他标签页通过 BroadcastChannel 与主页面通信来获取各自的数据更新消息，在主标签页被关闭后重新选举来实现多页面只占一个sse连接从而大幅减少连接资源消耗，提高页面稳定性并优化用户体验。并且只需修改一个sse的总的配置文件，而不需要对所有建立sse连接的地方进行代码修改。

当然为了解决不同页面订阅不同数据的问题，我们在 BroadcastChannel 消息中增加了 serverId、pageId 等标识。主标签页收到 SSE 推送后会进行统一广播，而各子标签页根据自身订阅的资源标识进行过滤，只处理与当前页面相关的数据，从而避免无效渲染和消息串扰。

BroadcastChannel的运行逻辑，容易被问的

简单来说，这套 BroadcastChannel + localStorage 的方案核心目标就是保证任意时刻只有一个标签页持有 SSE 连接。新标签页打开时会先给自己分配一个 tabId，然后注册到 localStorage 的 activeTabs 里，再立即参与主标签页选举。选举规则很简单，就是谁的 tabId 最小谁当 Master，所以第一次打开的页面通常会成为主标签页。

新标签页本身并不是通过 BroadcastChannel 被发现的，而是通过 activeTabs 这份共享数据被其他页面感知的。BroadcastChannel 更多负责实时通信，比如通知谁是 Master、转发 SSE 消息以及发送心跳。

主标签页关闭时会触发 beforeunload 或 unload，在卸载逻辑里发送 master-disconnected 消息，同时清除 localStorage 中记录的 masterKey。其他标签页收到这个消息后会立即发起新一轮选举，因此主标签页正常关闭时切换过程非常快。

如果主标签页不是正常关闭，比如浏览器崩溃、进程被杀或者电脑突然断电，那么卸载事件不会执行。这时候依靠的是心跳机制。从标签页每 30 秒发送一次 ping，主标签页收到后回复 pong，如果 5 秒内没有收到 pong，就认为主标签页已经失联，然后主动发起重新选举，所以即使 Master 异常消失，整个系统最终也能恢复。

从标签页关闭就简单很多，因为它本来就没有 SSE 连接，它只需要把自己从 activeTabs 中移除即可，对其他页面几乎没有影响，Master 甚至不需要专门感知某个 Slave 是否离开。

刷新标签页其实等价于“关闭旧页面再打开新页面”。如果当前刷新的是 Master，那么它会先执行卸载逻辑，通知其他页面重新选举，原来的 Slave 会很快接管成为新的 Master。等刷新完成后，新页面会获得一个全新的 tabId 重新加入系统，但由于已经有新的 Master 存在，所以它通常只能作为 Slave 存在。这意味着当前实现里 Master 一旦刷新，大概率就再也拿不回 Master 身份了。

对于长时间没有切换到的标签页，代码依赖每分钟更新一次 timestamp 来证明自己还活着。但问题在于现代浏览器会对后台标签页进行节流（Timer Throttling），浏览器对后台页面会进行定时器降频甚至暂停，所以某些标签页虽然实际上还开着，却可能几分钟都没有更新心跳，最终被 cleanExpiredTabs 当成失效页面清理掉。

不过被清理不代表彻底出局。当用户重新切回这个页面时，下次定时器执行会发现自己已经不在 activeTabs 中，于是调用 registerActiveTab 重新注册，因此它可以重新加入集群。

重新加入之后当然还能参与选举，但当前实现采用的是“非抢占式选举”。严格来说，代码并没有实现真正的 localStorage 分布式锁，它采用的是“基于 activeTabs 的确定性选举 + masterKey 公示”的方案。选举时所有标签页根据 activeTabs 按 tabId 排序，理论上都会得到同一个最小 tabId 作为 Master，然后由获胜者将自己的 instanceId 写入 masterKey，其余标签页通过 BroadcastChannel 和 storage 事件感知结果。因此 masterKey 更像是领导者公告，而不是具备原子性的抢占锁。由于 localStorage 不支持 Compare-And-Set 等原子操作，所以这种方案仍然存在极端时序下的脑裂风险。

也就是说，如果现在已经有一个正常工作的 Master，即使后来出现了一个 tabId 更小的页面，也不会主动触发重新选举，所以这个恢复的标签页通常只能继续做 Slave，除非未来某一天现任 Master 挂掉，它才有机会重新竞争。

最后还有一个容易忽略的点：这套代码完全没有监听 visibilitychange，也就是说页面切到后台、再切回来，对主从身份没有任何直接影响。它只会影响定时器是否正常执行，从而间接影响活跃状态判断。因此当前方案实际上是“基于存活状态选举”，而不是“基于当前激活页面选举”。

如果从架构设计角度评价的话，这套方案已经能解决 90% 的 SSE 多标签页共享问题，但最大的风险就是后台标签页被浏览器限速后误判失活，这也是最值得指出的优化点。

严格来说仍然存在理论上的脑裂（两个标签页同时认为自己是主标签页）风险，因为 activeTabs 的维护依赖 localStorage 的读改写而非事务，无法保证所有标签页始终基于完全一致的成员列表进行选举。但由于选举规则是确定性的最小 tabId 选举，只要 activeTabs 一致，即使多个标签页同时进入 electionMaster，也会得出相同结果。因此在实际运行中，这套方案的脑裂概率较低，真正更值得关注的是后台标签页被浏览器限速后导致的误判失活问题。

面试官可能追问

为什么不用 localStorage 通信？

面试者： localStorage 的 storage 事件，不能通知当前页面且通信实时性一般，并且数据同步的性能较差。

而 BroadcastChannel原生就支持多标签页实时，且通信API，只用修改sse的总的配置文件， 更简单不需要轮询，有更适合实时场景。

2. React Error Boundary

面试官： Error Boundary 你是怎么封装的？

面试者： 因为智能体平台模块很多，有构建，模型，知识库，系统等模块页面，其中部分模块渲染异常后会直接导致白屏。

我在未封装全局报错边界的之前，用户有反馈点击某些页面会白屏系统崩溃的情况，但是在有些pc上却无法复现，我在排查了之后发现是用户的谷歌浏览器自带的翻译插件存在修改源代码的dom的情况导致dom报错

因此我封装了全局 Error Boundary，核心目的是给用户更完备的系统体验，给用户重新加载或者返回上一次运行正常模块的页面，不会导致用户认为整个系统的崩溃而手足无措。

实现方式是：

class ErrorBoundary extends React.Component {
  state = { hasError: false }

  static getDerivedStateFromError() {
    return { hasError: true }
  }

  componentDidCatch(error, info) {
    console.log(error, info)
  }

  render() {
    if (this.state.hasError) {
      return <Fallback />
    }

    return this.props.children
  }
}

然后在路由文件的关键业务模块外层包裹。这样即使某个节点异常：

  • 页面不会整体崩溃
  • 用户还能继续操作其他功能

3. 动态渲染工作流

面试官： 你说“配置驱动动态渲染”，什么意思？

面试者：

在农担智能体项目里，我们的工作流编辑器不是把每一种节点都写死在前端，而是做成了配置驱动。

简单来说，后端会返回一份工作流配置，里面主要包括三类信息：第一是节点列表，第二是节点之间的连线关系，第三是画布的位置和缩放信息。每个节点里面又会带自己的类型、名称、描述，以及这个节点需要展示哪些参数表单。

比如一个大模型节点，它的配置里会声明它需要模型选择、提示词输入、温度参数、输出变量这些字段；知识库节点会声明知识库选择、召回数量、相似度阈值等字段；条件节点会声明条件分支规则。前端不需要提前知道每个节点长什么样，而是根据节点类型和参数类型去动态匹配对应的 UI 组件。

前端这边大致分两层渲染。第一层是工作流画布，我们用 React Flow 的方式，把后端返回的节点和连线渲染成可拖拽、可连接的流程图。第二层是节点内部表单渲染，每个节点会读取自己的参数配置，然后根据参数类型动态渲染不同控件，比如输入框、文本域、下拉选择、开关、滑块、变量选择器、知识库选择器等等。

这样设计的好处是扩展成本比较低。比如后面要新增一个“报告生成节点”或者“工具调用节点”，如果它用到的参数类型前端已经支持，那只需要后端新增一份节点配置，前端就能自动渲染出来；如果出现新的表单类型，前端只需要补一个新的控件映射，不需要重写整个节点页面。

我觉得这个优化的核心就是把“节点是什么、有哪些参数”交给配置描述，把“怎么画节点、怎么渲染表单”沉淀成前端通用能力。这样工作流节点越来越多的时候，前端代码不会随着业务节点数量线性膨胀，维护性会好很多。

ReactFlow是什么

ReactFlow 是一个基于 React 的流程图引擎，通过 Node 和 Edge 描述图结构，内部维护节点坐标、连线关系和视图状态，常用于低代码平台、AI 工作流编排、DAG 可视化编辑器等场景。实际项目中通常结合自定义节点的注册和使用、状态管理（Redux/Zustand）以及后端 DAG 执行引擎实现完整工作流系统。

ReactFlow 本身更像是一个“流程图画布引擎”。在我们项目里，前端不会手动画每个节点和每条线，而是把后端返回的工作流数据整理成 ReactFlow 认识的结构，然后交给 ReactFlow 渲染。

它主要依赖两个核心数据：nodes 和 edges。

nodes 表示画布上有哪些节点。每个节点里会有节点 id、节点类型、坐标位置，以及业务数据。比如这个节点是大模型节点、知识库节点还是条件节点，节点名称是什么，需要展示哪些参数表单，这些都放在节点的 data 里。

edges 表示节点之间的连线关系。每条边会记录从哪个节点出来，连到哪个节点，以及对应的连接点。

前端拿到这些数据后，会把它们传给 ReactFlow。ReactFlow 负责根据节点坐标把节点摆到画布上，根据 edges 把线连起来，同时提供拖拽、缩放、连线、删除、选中这些交互能力。

但是具体“节点长什么样”，不是 ReactFlow 决定的，而是我们通过自定义节点组件来决定。项目里会配置一个节点类型映射，比如工作流节点统一映射到自定义的 FlowNode 组件。ReactFlow 渲染某个节点时，会根据节点的 type 找到对应组件，然后把这个节点的 data 传进去。

FlowNode 拿到 data 后，再去渲染节点标题、描述、图标、输入输出连接点，以及节点内部的动态表单。动态表单又是根据节点配置里的参数类型渲染出来的，比如输入框、下拉框、变量选择器、知识库选择器等。

所以整个过程可以概括成：

后端返回工作流配置，前端整理成 nodes 和 edges，ReactFlow 负责画布和连线，自定义节点组件负责节点内容，参数组件负责节点内部表单。

这样 ReactFlow 解决的是“流程图怎么画、怎么交互”的问题，我们自己的配置驱动逻辑解决的是“每个业务节点里面展示什么”的问题。

二、React 深挖（你大概率会被重点问）

4.React 已经有 Context 和 useReducer 了，为什么项目里还要使用 Redux？

​ 在慢 SQL 分析系统里，当时之所以选择 Redux，是因为系统里存在比较多跨页面共享且关联复杂的全局状态，比如查询条件、分页信息、当前选中的 SQL、分析结果缓存等，如果只用 Context 和 useReducer，随着业务变复杂，会面临 Context 高频更新导致的大范围重渲染，以及状态修改路径不清晰的问题。

​ Redux 通过 Store 集中管理状态，所有修改都必须经过 dispatch → reducer → 更新 state 这条单向数据流，使得状态变化可预测、可追踪，再加上 Redux DevTools 能够查看每一次 action 和状态变化，对于排查复杂后台系统的问题非常有帮助。

​ 不过如果让我现在重新选型，我不会默认继续使用 Redux，而是会根据状态类型拆分：服务端数据，例如 SQL 列表、统计结果、详情信息等，我会交给 React Query 管理，因为它天然支持缓存、重试、失效更新和请求状态管理；少量客户端全局状态，比如筛选条件、当前选中项、弹窗状态等，我会更倾向于使用 Zustand，它比 Redux 更轻量、样板代码更少，同时具备按需订阅能力。至于 Context，我通常只会用于主题、国际化、用户信息这类更新频率较低的状态。因此，我认为 Redux 并不是过时了，它在大型、多人协作、状态关系复杂且强调规范性的项目中依然很有价值，但对于现在大多数中后台项目而言，React Query 加 Zustand 往往是更加灵活、高效的组合。

React Hooks 为什么比 class 更好？

面试者：

class组件存在this指向辅助，生命周期逻辑分散且逻辑复用困难的问题，而Hooks的这种函数式开发的方式更加简洁且逻辑的复用更方便，更适合TS语言以及项目的维护。因此我个人封装组件的时候都优先用函数式组件的方式。

5. useEffect 为什么会死循环？

面试者： 因为 useEffect 依赖变化会重新执行。如果 effect 内部又修改了依赖就会无限更新。本质是上是依赖更新导致effect执行又导致重新 更新 而又触发 effect执行而无限循环。

useEffect(() => {
  setCount(count + 1)
}, [count])

6. React 为什么需要 key？

面试者： React diff 时需要判断节点是否复用。key 本质是节点唯一标识。如果没有 key：React 可能错误复用 DOM。导致输入框错乱，状态异常以及不必要重渲染

面试者：

我做过几类：

减少重复渲染。

2. SSE 连接优化

3. 大量组件重绘优化

智能体开发平台中，知识库卡片频繁重绘。我当时：

  • 拆分组件-->减少重绘，组件内部state变化只会影响特定组件，SearchBar 内部的输入状态变化，只会引起 SearchBar 自身重渲染，而 ProductList 和 Sidebar 不会受影响。如果原来这三组件在一个card组件里面那么都得被重绘
  • 降低父组件 state 粒度--->一个 state 包含了多个不相关的数据，如一个user对象既包含用户名又包含地址，改用户名的时候改整个对象会导致引用改变而依赖地址的组件也得重绘，且因为是引用变化，memo 也会失效，因此得将user进行细化粒度，尽量少用一个对象做一个 state，尽量拆分为多个属性的state，对于有关联的状态，可以使用 useReducer
// 对于有关联的状态，可以使用 useReducer
const initialState = { count: 0, step: 1 };

function reducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { ...state, count: state.count + state.step };
    case 'setStep':
      return { ...state, step: action.payload };
    default:
      return state;
  }
}

function Component() {
  const [state, dispatch] = useReducer(reducer, initialState);
  
  // dispatch 是稳定的，不会导致不必要的重渲染，但 state 对象仍然每次都是新的
  return (
    <div>
      <Counter value={state.count} />
      <StepControl value={state.step} onChange={(step) => dispatch({ type: 'setStep', payload: step })} />
    </div>
  );
}
  • 避免匿名函数重复创建------>匿名函数每次渲染都创建新的函数引用，会破坏 React.memo 的浅比较优化，父组件状态变化时，每次都会创建新函数 传给子组件，子组件用了memo本应跳过渲染，但是由于父组件传给子组件的匿名函数每次都是新函数，导致每次都重新渲染。应该使用 useCallback 缓存函数：
  // 只有当 count 变化调用handleIncrement才重新创建函数
  const handleIncrement = useCallback(() => {
    setCount(c => c + 1);
  }, []); // 空依赖 → 永远同一个函数

明显减少页面卡顿。

四、前后端联调（非常符合你简历）

8. 联调时最常见的问题是什么？

面试者：

最常见的是：

字段不一致
数据结构变化
类型错误
空值问题

比如后端返回：

{
  list: null
}

前端直接 map：

list.map(...)

页面直接崩。

所以我现在会：

  • 做 TS 类型约束
  • 做默认值处理
  • 做接口数据校验
