人脸识别接口中加密算法

注：

  • SHA1 / SHA256 ： 都是哈希算法，通过把任意长度的数据转换成一个固定长度的字符串（摘要），转换后不可逆且结果完全由输入决定，结果固定长度（SHA1 输出 160 位，SHA256 输出 256 位）。SHA1已经是老算法，存在安全性不足的问题，现在已经不推荐了。
  • HMAC ：即Hash-based Message Authentication Code，为基于哈希的消息认证码。他本质上是 哈希算法 + 密钥(key)。他加入了secret key，使得只有知道密钥的人才能通过哈希算法算出结果，而不是像简单的哈希算法那样只能验证内容的完整性。HMAC_SHA256(data, secret)

嗯，这个加密对接的逻辑是这样的。接口要求我们每次请求的时候都要带一个签名，签名的作用是验证请求的合法性和完整性，防止被篡改或者重放攻击。

签名的生成规则其实不复杂。公司那边给了 accessKeyId 和 accessKeySecret 两个凭证，我这边需要先把四个东西拼接起来：accessKeyId、accessKeySecret、当前的时间戳 timestamp、还有请求参数的前 512 个字符。拼接成一个字符串作为待签名的原始数据。

然后根据文档要求的签名类型，比如 SHA256 或者 SHA1，用 HMAC 算法对这个拼接好的字符串进行签名。这里注意签名用的密钥是 accessKeySecret，不是凭证明文。签名出来是一个二进制数据，再转成 Base64 字符串，这就是最终的 sign 值。

我代码里用的是浏览器的 Web Crypto API 来做 HMAC 签名，因为原生支持 SHA 系列算法，性能也还可以。但 MD5 不行，Web Crypto API 不支持，如果要 MD5 就得换 crypto-js 这种库。

拿到 sign 之后，每次调用人脸识别的接口，我都会把 accessKeyId、timestamp、sign 这三个东西带上，后端收到后会拿同样的逻辑重新算一遍签名，比对是否一致。只要签名对得上，请求才会被认为是合法的。

然后插件这边我用的是浏览器打包的方式，把写好的代码打成插件文件，放到 126 浏览器的扩展程序里加载测试。整个流程跑通之后，活体识别接口就能正常调用了。

BroadcastChannel的运行逻辑，容易被问的

简单来说，这套 BroadcastChannel + localStorage 的方案核心目标就是保证任意时刻只有一个标签页持有 SSE 连接。新标签页打开时会先给自己分配一个 tabId，然后注册到 localStorage 的 activeTabs 里，再立即参与主标签页选举。选举规则很简单，就是谁的 tabId 最小谁当 Master，所以第一次打开的页面通常会成为主标签页。

新标签页本身并不是通过 BroadcastChannel 被发现的，而是通过 activeTabs 这份共享数据被其他页面感知的。BroadcastChannel 更多负责实时通信，比如通知谁是 Master、转发 SSE 消息以及发送心跳。

主标签页关闭时会触发 beforeunload 或 unload，在卸载逻辑里发送 master-disconnected 消息，同时清除 localStorage 中记录的 masterKey。其他标签页收到这个消息后会立即发起新一轮选举，因此主标签页正常关闭时切换过程非常快。

如果主标签页不是正常关闭，比如浏览器崩溃、进程被杀或者电脑突然断电，那么卸载事件不会执行。这时候依靠的是心跳机制。从标签页每 30 秒发送一次 ping，主标签页收到后回复 pong，如果 5 秒内没有收到 pong，就认为主标签页已经失联，然后主动发起重新选举，所以即使 Master 异常消失，整个系统最终也能恢复。

从标签页关闭就简单很多，因为它本来就没有 SSE 连接，它只需要把自己从 activeTabs 中移除即可，对其他页面几乎没有影响，Master 甚至不需要专门感知某个 Slave 是否离开。

刷新标签页其实等价于“关闭旧页面再打开新页面”。如果当前刷新的是 Master，那么它会先执行卸载逻辑，通知其他页面重新选举，原来的 Slave 会很快接管成为新的 Master。等刷新完成后，新页面会获得一个全新的 tabId 重新加入系统，但由于已经有新的 Master 存在，所以它通常只能作为 Slave 存在。这意味着当前实现里 Master 一旦刷新，大概率就再也拿不回 Master 身份了。

对于长时间没有切换到的标签页，代码依赖每分钟更新一次 timestamp 来证明自己还活着。但问题在于现代浏览器会对后台标签页进行节流（Timer Throttling），浏览器对后台页面会进行定时器降频甚至暂停，所以某些标签页虽然实际上还开着，却可能几分钟都没有更新心跳，最终被 cleanExpiredTabs 当成失效页面清理掉。

不过被清理不代表彻底出局。当用户重新切回这个页面时，下次定时器执行会发现自己已经不在 activeTabs 中，于是调用 registerActiveTab 重新注册，因此它可以重新加入集群。

重新加入之后当然还能参与选举，但当前实现采用的是“非抢占式选举”。严格来说，代码并没有实现真正的 localStorage 分布式锁，它采用的是“基于 activeTabs 的确定性选举 + masterKey 公示”的方案。选举时所有标签页根据 activeTabs 按 tabId 排序，理论上都会得到同一个最小 tabId 作为 Master，然后由获胜者将自己的 instanceId 写入 masterKey，其余标签页通过 BroadcastChannel 和 storage 事件感知结果。因此 masterKey 更像是领导者公告，而不是具备原子性的抢占锁。由于 localStorage 不支持 Compare-And-Set 等原子操作，所以这种方案仍然存在极端时序下的脑裂风险。

也就是说，如果现在已经有一个正常工作的 Master，即使后来出现了一个 tabId 更小的页面，也不会主动触发重新选举，所以这个恢复的标签页通常只能继续做 Slave，除非未来某一天现任 Master 挂掉，它才有机会重新竞争。

最后还有一个容易忽略的点：这套代码完全没有监听 visibilitychange，也就是说页面切到后台、再切回来，对主从身份没有任何直接影响。它只会影响定时器是否正常执行，从而间接影响活跃状态判断。因此当前方案实际上是“基于存活状态选举”，而不是“基于当前激活页面选举”。

如果从架构设计角度评价的话，这套方案已经能解决 90% 的 SSE 多标签页共享问题，但最大的风险就是后台标签页被浏览器限速后误判失活，这也是最值得指出的优化点。

严格来说仍然存在理论上的脑裂（两个标签页同时认为自己是主标签页）风险，因为 activeTabs 的维护依赖 localStorage 的读改写而非事务，无法保证所有标签页始终基于完全一致的成员列表进行选举。但由于选举规则是确定性的最小 tabId 选举，只要 activeTabs 一致，即使多个标签页同时进入 electionMaster，也会得出相同结果。因此在实际运行中，这套方案的脑裂概率较低，真正更值得关注的是后台标签页被浏览器限速后导致的误判失活问题。

农担封装了什么组件

在这个项目最核心的智能体对话界面，我进行了许多的组件封装，

首先最核心的是 ChatInput.tsx，这个是输入框组件。它不光是个输入框，还处理了多行文本的自动撑高、Enter 发送和 Shift+Enter 换行的区分，还有上传图片或者文件的触发入口。它和 FileBs.tsx 配合，文件那块逻辑单独抽出来，不至于让输入框组件太臃肿。

然后是消息展示这一块，我没有只写一个大组件把所有消息类型都塞进去，而是拆成了 MessageUser.tsx 和 MessageSystem.tsx，分别渲染用户发的消息和系统返回的消息。它们两个都依赖底层的 MessageBs.tsx 来承载公共样式或者气泡结构。另外还单独抽了 MessageButtons.tsx，用来渲染消息下面的快捷回复按钮，这个按钮组在不同场景会复用。还有一个 MessagePanne.tsx，是用做消息面板的容器，管理滚动到底部、加载历史消息这些逻辑。

我还封装了 Separator.tsx 配合 SeparatorFactory.tsx 来做分隔线。比如显示“昨天”、“今天”这种时间分段，或者用来插一些提示文案。用工厂模式是为了根据不同的分隔类型动态生成不同的样式和行为，方便扩展。

RunLog.tsx 是用于展示 AI 执行步骤或者调用链信息的，比如“思考中”、“正在检索”这种过程性的日志，方便用户理解 AI 的进度。GuideQuestions.tsx 是进入对话前的引导问题列表，用户点一下就能直接发送，减少输入成本。

再说另外两个通用组件。多选下拉栏组件，我封装的时候支持了搜索过滤、批量删除、全选、最大选择数限制这些功能。对外暴露的接口比较简洁，传 options 数组和一个 v-model 绑定的选中值就行。主页的卡片组件，我做了比较通用的设计，支持传入 ICON 的组件或者图片，卡片标题、描述内容、底部的操作按钮都可以通过插槽或者 props 来定制。这样在不同的列表页或者仪表盘都能复用，不需要重复写卡片的布局和样式。

总的来说，我的习惯是尽量让组件保持单一职责，组合起来用。像 AI 对话这种复杂模块，拆得细一点，后期维护和替换具体实现都会轻松很多。

你在农担智能体项目中提到“基于配置驱动优化工作流渲染机制”，请具体说明这个配置驱动的数据结构是怎么设计的，以及前端如何根据配置动态渲染不同节点和表单？

回答

在农担智能体项目里，我们的工作流编辑器不是把每一种节点都写死在前端，而是做成了配置驱动。

简单来说，后端会返回一份工作流配置，里面主要包括三类信息：第一是节点列表，第二是节点之间的连线关系，第三是画布的位置和缩放信息。每个节点里面又会带自己的类型、名称、描述，以及这个节点需要展示哪些参数表单。

比如一个大模型节点，它的配置里会声明它需要模型选择、提示词输入、温度参数、输出变量这些字段；知识库节点会声明知识库选择、召回数量、相似度阈值等字段；条件节点会声明条件分支规则。前端不需要提前知道每个节点长什么样，而是根据节点类型和参数类型去动态匹配对应的 UI 组件。

前端这边大致分两层渲染。第一层是工作流画布，我们用类似 React Flow 的方式，把后端返回的节点和连线渲染成可拖拽、可连接的流程图。第二层是节点内部表单渲染，每个节点会读取自己的参数配置，然后根据参数类型动态渲染不同控件，比如输入框、文本域、下拉选择、开关、滑块、变量选择器、知识库选择器等等。

这样设计的好处是扩展成本比较低。比如后面要新增一个“报告生成节点”或者“工具调用节点”，如果它用到的参数类型前端已经支持，那只需要后端新增一份节点配置，前端就能自动渲染出来；如果出现新的表单类型，前端只需要补一个新的控件映射，不需要重写整个节点页面。

我觉得这个优化的核心就是把“节点是什么、有哪些参数”交给配置描述，把“怎么画节点、怎么渲染表单”沉淀成前端通用能力。这样工作流节点越来越多的时候，前端代码不会随着业务节点数量线性膨胀，维护性会好很多。

ReactFlow是什么

ReactFlow 本身更像是一个“流程图画布引擎”。在我们项目里，前端不会手动画每个节点和每条线，而是把后端返回的工作流数据整理成 ReactFlow 认识的结构，然后交给 ReactFlow 渲染。

它主要依赖两个核心数据：nodes 和 edges。

nodes 表示画布上有哪些节点。每个节点里会有节点 id、节点类型、坐标位置，以及业务数据。比如这个节点是大模型节点、知识库节点还是条件节点，节点名称是什么，需要展示哪些参数表单，这些都放在节点的 data 里。

edges 表示节点之间的连线关系。每条边会记录从哪个节点出来，连到哪个节点，以及对应的连接点。

前端拿到这些数据后，会把它们传给 ReactFlow。ReactFlow 负责根据节点坐标把节点摆到画布上，根据 edges 把线连起来，同时提供拖拽、缩放、连线、删除、选中这些交互能力。

但是具体“节点长什么样”，不是 ReactFlow 决定的，而是我们通过自定义节点组件来决定。项目里会配置一个节点类型映射，比如工作流节点统一映射到自定义的 FlowNode 组件。ReactFlow 渲染某个节点时，会根据节点的 type 找到对应组件，然后把这个节点的 data 传进去。

FlowNode 拿到 data 后，再去渲染节点标题、描述、图标、输入输出连接点，以及节点内部的动态表单。动态表单又是根据节点配置里的参数类型渲染出来的，比如输入框、下拉框、变量选择器、知识库选择器等。

所以整个过程可以概括成：

后端返回工作流配置，前端整理成 nodes 和 edges，ReactFlow 负责画布和连线，自定义节点组件负责节点内容，参数组件负责节点内部表单。

这样 ReactFlow 解决的是“流程图怎么画、怎么交互”的问题，我们自己的配置驱动逻辑解决的是“每个业务节点里面展示什么”的问题。

详细逻辑

在农担智能体项目里，工作流渲染不是前端写死每一种节点页面，而是把工作流抽象成一份配置数据。核心结构分三层：工作流、节点、参数。

工作流层大概是：

{
  id,
  name,
  description,
  nodes,
  edges,
  viewport
}

其中 nodes / edges / viewport 对应 React Flow 的画布数据。nodes 描述有哪些节点、节点坐标和节点业务配置；edges 描述节点之间的连线；viewport 记录画布缩放和位置。

节点层的核心结构是：

{
  id,
  type,
  name,
  description,
  group_params,
  tab
}

type 表示节点类型，比如开始节点、LLM 节点、知识库检索节点、条件节点、输出节点、工具节点等。真正决定节点内部表单怎么渲染的是 group_params。

group_params 是参数分组，例如“模型配置”“输入参数”“输出变量”等。每个分组下面有一组 params：

{
  key,
  label,
  type,
  value,
  placeholder,
  required,
  options,
  hidden,
  tab
}

这里最关键的是 type 字段。前端并不关心这是哪个业务节点，而是根据参数的 type 去选择对应表单组件。比如：

  • input 渲染普通输入框
  • textarea 渲染文本域
  • var_textarea 渲染支持变量选择的文本域
  • postwise_model 渲染模型选择器
  • knowledge_select_multi 渲染知识库多选
  • condition 渲染条件配置
  • output_form 渲染输出变量配置
  • switch / slide / number 渲染开关、滑块、数字输入等

前端的渲染链路大概是这样的：

  1. 进入工作流编辑页时，通过接口拿到工作流配置。
  2. 把后端返回的 data.nodes / data.edges / data.viewport 转成 React Flow 需要的数据。
  3. 在画布组件里用 <ReactFlow nodes={nodes} edges={edges} nodeTypes={nodeTypes} /> 渲染整个工作流。
  4. nodeTypes 里把业务节点统一映射成一个自定义节点组件，例如 flowNode -> FlowNode。
  5. FlowNode 内部读取当前节点的 data，展示节点标题、描述、图标、连接点，然后遍历 group_params。
  6. 每个 ParameterGroup 再遍历 params。
  7. 最后在 Parameter 组件中根据 item.type 做组件映射，动态渲染具体表单项。

可以理解为：

node.group_params.map(group =>
  group.params.map(param =>
    renderComponentByType(param.type)
  )
)

这样做的好处是，新加一种节点时，前端大多数情况下不需要重新写一个完整页面，只要后端或模板里新增节点配置。如果参数类型已经存在，前端可以直接渲染；如果是新的参数类型，只需要在 Parameter 的映射里新增一种组件即可。

还有一个细节是，左侧节点面板也是配置驱动的。前端会读取节点模板配置，生成可拖拽节点。用户把节点拖到画布上时，前端会给节点生成唯一 id，初始化参数中的变量引用，然后加入 React Flow 的 nodes。之后用户编辑节点、拖动节点、连接节点，都会更新本地的 nodes / edges，保存时再把这份配置提交给后端。

所以这个机制本质上是：后端或模板提供“节点描述 + 参数 schema”，前端提供“渲染引擎 + 组件映射表”。配置决定渲染内容，React Flow 负责画布和连线，自定义节点组件负责把配置翻译成具体 UI。这样可以降低工作流节点扩展成本，也让不同节点的表单渲染逻辑保持统一。

你在电信慢SQL分析系统中“清理了遗留的 MockJS 逻辑”，请说明当时为什么会有 MockJS 遗留问题，你清理时具体做了哪些事情来保证接口切换稳定？

项目早期因前后端并行开发，当时部分后端接口还没有完全稳定，前端为了不阻塞页面开发，就先用 MockJS 模拟接口数据，把慢 SQL 列表、多条件筛选、详情展示这些页面先搭起来。

后面切真实接口的时候，问题就暴露出来了：Mock 数据和真实接口返回不一定完全一致。比如字段名可能不同，分页结构可能不同，有些字段 Mock 里一直有值，但真实接口可能返回空值；还有数组层级、列表字段、总数字段这些也可能不一致。如果直接把 Mock 删掉，很容易导致页面渲染异常，或者筛选、分页参数传错。

我在对其进行清理的时候首先是梳理 Mock 依赖，然后对照 Swagger 和真实接口做字段校验，然后如果真实接口和 Mock 数据结构不一致，我尽量把差异收敛在接口封装或者数据处理层，而不是让页面组件里到处写兼容逻辑，这个项目筛选条件比较多，切真实接口后最容易出问题的是：查询时参数没带全、重置后旧参数残留、分页切换时筛选条件丢失。所以我会重点验证查询、重置、分页变化、表格刷新这些流程，并通过真实环境联调保证数据链路稳定。

所以说这件事的重点不是简单“删掉 MockJS”，而是要保证从 Mock 数据切到真实接口之后，整个页面的数据链路、查询逻辑和异常场景都是稳定的。

我印象比较深、复用价值比较高的是在 农担智能体开发平台 里封装的 Error Boundary 异常边界组件。 它解决的重复问题是：平台里有很多模块，比如构建、模型、知识库、系统、用户管理、召回测试、平台概览等页面。之前如果某个模块内部组件渲染异常，可能会直接导致整个页面白屏。 而且我们确实遇到过一次用户反馈某些页面点击后白屏，但本地不太好复现，后来排查发现和用户浏览器自带的翻译插件修改 DOM 有关，导致 React 渲染异常。

所以我封装这个组件的目的不是单纯为了展示一个异常 UI，而是为了统一处理 局部模块异常导致全局白屏 的问题，避免每个页面都单独写一套 try/catch 或异常兜底逻辑。 Props 设计上我没有做得特别复杂，核心就是用 children 接收被保护的业务模块： tsx <ErrorBoundary> <BusinessModule /> </ErrorBoundary> 因为它主要包在路由文件或者关键业务模块外层，所以外部只需要关心“哪些模块需要被保护”，不需要关心内部怎么捕获异常。 内部逻辑主要用 class 组件实现，因为 React 的 Error Boundary 需要依赖生命周期： tsx class ErrorBoundary extends React.Component< { children: React.ReactNode }, { hasError: boolean } > { state = { hasError: false } static getDerivedStateFromError() { return { hasError: true } } componentDidCatch(error: Error, info: React.ErrorInfo) { console.log(error, info) } render() { if (this.state.hasError) { return ( <div> <div>当前模块出现异常</div> <button onClick={() => window.location.reload()}> 重新加载 </button> </div> ) } return this.props.children } } 核心逻辑是： 1. 正常情况下直接渲染 children 也就是业务模块本身不需要改造。 2. 子组件渲染异常时触发 getDerivedStateFromError 把 hasError 改成 true，避免继续渲染出错的业务模块。 3. 在 componentDidCatch 里记录错误信息 方便后续定位是哪个组件、哪个渲染链路出现了问题。 4. 展示兜底页面 给用户提供重新加载或者返回上一个正常模块的入口，避免用户觉得整个系统完全崩溃了。 这个组件的复用价值在于，它可以统一包裹在不同业务模块外层，比如用户管理、召回测试、平台概览等页面。这样某一个模块出问题，不会影响整个系统继续使用，用户也还能操作其他功能。 所以它解决的是一个平台级的重复问题：每个业务模块都可能异常，但不应该每个模块都自己写一套异常兜底逻辑。
