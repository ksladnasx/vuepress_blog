面试高频考点

BFC触发条件

BFC（Block Formatting Context）即块级格式化上下文，它是一块独立的渲染区域，内部元素的布局不会影响外部元素。常见触发方式包括设置 overflow: hidden/auto、display: flex、display: inline-block、position: absolute/fixed、float 等。

BFC 常用于解决一些布局问题，比如清除浮动、防止 margin 塌陷以及实现多栏布局。例如父元素没有高度时，子元素浮动会导致高度塌陷，可以通过给父元素设置 overflow: hidden 创建 BFC 解决。

前端怎么做的移动端适配

移动端适配主要是根据不同设备屏幕尺寸调整页面布局，常见方案包括响应式布局、rem 适配和 vw 适配。

实际开发中通常会结合媒体查询和弹性布局，例如使用 flex、grid 实现自适应布局，同时通过 vw 根据屏幕宽度动态计算尺寸。对于需要精确还原设计稿的项目，可以使用 postcss-pxtorem 将 px 转换成 rem，实现不同屏幕下等比例缩放。

Promise、async/await

Promise 是 JavaScript 处理异步操作的一种机制，通过状态管理异步结果，包含 pending、fulfilled 和 rejected 三种状态，一旦状态改变就不会再次变化。

async/await 是基于 Promise 的语法糖，可以让异步代码看起来像同步代码，提高代码可读性。async 函数返回一个 Promise，await 会暂停当前 async 函数执行，等待 Promise 返回结果后继续执行。

防抖和节流做法

防抖是指连续触发事件时，只执行最后一次操作，常用于搜索框输入、窗口 resize 等场景。

实现方式一般是通过定时器，每次触发事件先清除之前的定时器，再重新设置新的定时器。

节流是指规定时间内只执行一次操作，常用于滚动监听、拖拽等高频事件。

实现方式可以通过时间戳或者定时器控制函数执行频率。

异步与事件循环

JavaScript 是单线程语言，但是通过事件循环机制实现异步操作。

JS 执行时会把同步任务立即执行，遇到异步任务就交给浏览器或其他环境去处理，等异步任务有了结果，就把它的回调函数扔进一个“任务队列”里。主线程执行完所有同步代码后，再去任务队列里拿回调来执行。这个过程不断重复，就像个循环，所以叫事件循环。

事件循环中任务分为宏任务和微任务，执行顺序是先执行同步代码，然后执行所有微任务，再执行下一个宏任务。

宏任务包括整体 script 代码、setTimeout、setInterval、I/O 操作这些；微任务主要是 Promise.then、MutationObserver、queueMicrotask 这些。

闭包的概念及作用

闭包是指函数能够访问并保存其定义时所在作用域中的变量，即使外部函数已经执行结束。

闭包常用于数据封装、模块化开发以及实现私有变量。例如通过闭包保存计数状态，避免变量被外部直接修改。

需要注意的是，闭包会长期保存变量引用，如果使用不当可能导致内存无法释放。

Cookie与localStorage与indexDB

Cookie 主要用于浏览器和服务器之间的数据传递，通常用于身份认证，例如保存 token，但是存储容量较小，并且每次请求都会自动携带。

localStorage 是浏览器本地存储，容量较大，数据会长期保存，适合保存一些非敏感的用户配置。

IndexedDB 是浏览器提供的本地数据库，支持存储大量结构化数据，适合离线应用、大量数据缓存等场景。

HTTP 不同状态码含义

状态码的类别：

类别原因描述
1xxInformational(信息性状态码)接受的请求正在处理
2xxSuccess(成功状态码)请求正常处理完毕
3xxRedirection(重定向状态码)需要进行附加操作一完成请求
4xxClient Error (客户端错误状态码)服务器无法处理请求
5xxServer Error(服务器错误状态码)服务器处理请求出错

具体含义：

状态码含义说明
100Continue（继续）请求头已收到，客户端应继续发送请求体（常用于大文件上传）。
101Switching Protocols（切换协议）服务器同意客户端升级协议的请求（如从 HTTP 升级到 WebSocket）。
200OK（成功）标准成功响应，GET/PUT/POST 请求正常返回数据。
201Created（已创建）资源创建成功（通常用于 POST），响应头应含 Location 地址。
202Accepted（已接受）请求已进入队列，但尚未处理完成（适用于异步任务）。
204No Content（无内容）操作成功，但响应体不返回任何数据（常用于 DELETE）。
206Partial Content（部分内容）服务器成功处理了部分 GET 请求（用于断点续传或分片下载）。
301Moved Permanently（永久重定向）资源永久变更，后续请求应使用新 URL（搜索引擎会更新索引）。
302Found（临时重定向）资源临时位于其他 URL（历史原因，实际常替代为 303/307）。
303See Other（查看其他）重定向到新 URL，且必须使用 GET 方法（常用于 POST 后的结果页跳转）。
304Not Modified（未修改）资源未变更，可使用客户端缓存（需配合 If-Modified-Since 头）。
307Temporary Redirect（临时重定向）类似 302，但不允许改变请求方法（如 POST 必须仍用 POST）。
308Permanent Redirect（永久重定向）类似 301，但不允许改变请求方法。
400Bad Request（错误请求）语法错误、参数缺失或格式不正确（如 JSON 格式错误）。
401Unauthorized（未授权）缺乏身份验证凭据或凭据无效（需重新登录）。
403Forbidden（禁止访问）认证通过，但无权限访问该资源（如角色权限不足）。
404Not Found（未找到）服务器无此资源（或已隐藏该资源的存在）。
405Method Not Allowed（方法不允许）请求方法不被该 URL 支持（如用 PUT 访问只读接口）。
409Conflict（冲突）资源状态冲突（如文件版本不一致、用户名已被占用）。
413Payload Too Large（请求体过大）上传的文件超出服务器限制。
415Unsupported Media Type（不支持的媒体类型）请求体的格式不受支持（如只接受 JSON 却发送了 XML）。
422Unprocessable Entity（无法处理）请求格式正确，但语义错误（如必填字段校验失败）。
429Too Many Requests（请求过多）触发了频率限制（通常用于限流降级）。
500Internal Server Error（内部服务器错误）通用服务器错误（通常是代码异常、数据库连接失败等）。
501Not Implemented（未实现）服务器不支持该请求方法（如老旧服务器不支持 PUT）。
502Bad Gateway（网关错误）网关或代理收到后端服务的无效响应（如 Nginx 连接 Tomcat 失败）。
503Service Unavailable（服务不可用）服务器过载或正在停机维护（通常伴随 Retry-After 头）。
504Gateway Timeout（网关超时）网关或代理等待后端服务响应超时。

Vue2和3以及React的响应式原理

Vue2 使用 Object.defineProperty 劫持对象属性，通过 getter 收集依赖，setter 通知更新，但是无法监听对象新增属性和数组变化，需要通过 Vue.set 等方式解决。

Vue3 使用 Proxy 代理整个对象，可以监听对象新增、删除以及数组变化，同时性能和扩展性更好。

React 本身不是响应式框架，而是通过状态驱动视图更新，当 state 或 props 发生变化时，React 会重新执行组件函数，通过虚拟 DOM diff 比较前后变化，再更新真实 DOM。

Vue2和Vue3的区别

Vue3 相比 Vue2 最大变化是响应式系统从 Object.defineProperty 升级为 Proxy，同时新增 Composition API，提高代码复用能力。

Vue3 支持更好的 TypeScript 类型支持，组件逻辑可以通过 setup 进行组合，减少 mixin 带来的命名冲突问题。

另外 Vue3 对虚拟 DOM 和编译器进行了优化，提升了运行性能。

模块化组件该怎么封装，有无相关经验

在项目中封装组件时，我一般会遵循高内聚、低耦合和可配置化的原则。首先会分析组件的通用逻辑，把和业务强相关的部分抽离出去，只保留通用能力，通过 props 控制组件展示，通过 emit 或回调函数向外暴露事件，同时利用插槽或者 render props 提供扩展能力，避免组件内部写死业务逻辑。

比如之前做智能体平台时，工作流节点有很多不同类型，每种节点都有自己的参数配置，如果每个节点单独写一个表单组件，后期维护成本比较高。所以我抽象了一套配置驱动的动态渲染机制，把节点类型、字段类型、校验规则等信息统一放到配置对象中，组件根据配置动态生成对应的表单项，这样新增节点时只需要增加配置，不需要重复开发组件。

另外在组件设计时，我会考虑组件的边界，比如基础组件只负责 UI 和交互，不处理接口请求；业务组件负责数据组合和状态管理。对于复杂组件还会提供一些公共 hooks，比如数据请求、状态逻辑复用，保证组件既能独立使用，也方便后续扩展。整体目标就是让组件具备复用性，同时避免为了复用而过度抽象。

内存泄漏和栈溢出

内存泄漏指程序中已经不需要的数据仍然被引用，导致垃圾回收机制无法释放，例如未清除的定时器、未解绑的事件监听、大量闭包引用等。

栈溢出通常是由于函数递归调用过深或者调用栈层级过多导致，例如无限递归最终超过 JavaScript 调用栈限制。

性能指标及做过的性能优化

前端性能指标主要关注页面加载速度和交互体验，包括 FCP（首次内容绘制）、LCP（最大内容绘制）、FID（首次输入延迟）和 CLS（布局稳定性）。

常见优化方式包括资源压缩、代码分割、懒加载、图片优化、减少 HTTP 请求、缓存优化以及减少不必要的组件渲染。

项目中可以通过 React.memo、Vue computed、虚拟列表等方式减少重复渲染，提高页面响应速度。

高访问量页面的前端优化手段

高访问量页面优化主要从资源加载、请求控制和渲染性能三个方面考虑。

资源方面可以使用 CDN、缓存策略、代码压缩和按需加载减少首屏资源。

请求方面可以进行接口缓存、防止重复请求、请求合并以及使用防抖节流降低压力。

渲染方面可以使用虚拟列表、分页加载、组件懒加载减少 DOM 数量。

处理大规模请求并发

处理大量请求时需要控制请求数量，避免同时发送大量请求导致浏览器或服务器压力过高。

常见方案包括请求队列、并发控制、取消无效请求以及接口缓存。

例如通过 Promise 控制同时执行的请求数量，完成一个请求后再继续执行队列中的下一个请求。

大量数据一次性渲染

大量数据直接渲染会导致 DOM 节点过多，引起页面卡顿。

常见优化方式是虚拟列表，只渲染当前可视区域的数据，根据滚动位置动态替换内容。

同时可以结合分页加载、懒加载以及数据分片处理，减少一次性计算和渲染压力。

大文件上传

大文件上传通常采用分片上传，将文件切割成多个小文件块分别上传。

上传过程中可以支持断点续传，通过记录已经上传成功的分片，在失败后只重新上传未完成部分。

同时可以结合 Web Worker 进行文件计算，避免大文件处理阻塞主线程。
